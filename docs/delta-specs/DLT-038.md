# Delta Spec: DLT-038 - Hot-reload skills at runtime

## Story

As the assistant, I want the skill registry to reflect the current state of the skills directory so that skills I create or modify during execution become available for detection without requiring an application restart.

## What It Does

Skills added, modified, or deleted at runtime become available for detection at the next session boundary without requiring a restart. A filesystem watcher monitors the skills directory and signals when changes occur, triggering a registry re-scan before the next detection cycle. Change events are published on the event bus for consumption by other subsystems. The re-scan is transparent to the provider and coordinator — no API changes.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Skills added or modified at runtime become available for detection without application restart | Core goal |
| R1 | Registry re-scans the skills directory to pick up additions, modifications, and deletions | Must-have |
| R2 | The registry knows when skills have changed on disk so it only re-scans when necessary | Must-have |
| R3 | Re-scan preserves existing error handling (invalid skills skipped, valid ones loaded) | Must-have |
| R4 | Re-scan is transparent to the context provider — no API change to `provide()` | Must-have |
| R5 | Deleted skill directories are removed from the registry after re-scan | Must-have |
| R6 | Mid-session skill changes do not affect the current session's detected skills (session stability invariant preserved) | Must-have |
| R7 | A `SkillsChanged` event is emitted via the event bus when skill changes are detected, available for future consumers | Must-have |
| R8 | Watcher lifecycle managed through bootstrap (start) and graceful shutdown | Must-have |
| R9 | Burst changes during skill authoring result in a single registry refresh and event, not one per file change | Must-have |

## Acceptance Criteria

### New skill available after creation (R0, R1)
- [ ] Given the registry has loaded, when a new skill directory with valid SKILL.md is created in `workspace/skills/`, then the next new session's `provide()` call discovers the new skill and includes it in classification
- [ ] Given a new skill with agents is created, when the registry re-scans, then the new skill's agents are available for detection and delegation

### Modified skill reflected after update (R0, R1)
- [ ] Given an existing skill's SKILL.md description is modified, when the registry re-scans, then the updated description is used for classification
- [ ] Given an existing skill's agent definitions are modified (added, removed, or changed), when the registry re-scans, then the updated agents are reflected in the registry
- [ ] Given an agent file is added to an existing skill without modifying SKILL.md, when the registry re-scans, then the new agent is discovered under that skill

### Deleted skill removed (R5)
- [ ] Given a skill directory is deleted from `workspace/skills/`, when the registry re-scans, then the skill and its agents are no longer in the registry
- [ ] Given a deleted skill was previously detected in an active session, then the current session's detected skills are not affected (R6)
- [ ] Given the entire `workspace/skills/` directory is deleted, when the registry re-scans, then it gracefully empties itself (no skills, no agents)

### Change detection triggers re-scan only when needed (R2)
- [ ] Given the watcher is running, when a file is added to the skills directory, then the registry is marked for re-scan
- [ ] Given the watcher is running, when a file is modified in the skills directory, then the registry is marked for re-scan
- [ ] Given the watcher is running, when a file is deleted from the skills directory, then the registry is marked for re-scan
- [ ] Given no changes have occurred since the last re-scan, when the provider triggers a refresh, then the registry skips the re-scan

### Burst changes coalesced (R9)
- [ ] Given a burst of file changes occurs within a short window (e.g., skill authoring creating directory + SKILL.md + agent files), then a single re-scan mark and event are produced after the burst settles
- [ ] Given changes keep occurring within the debounce window, then the window resets and waits for quiet before firing

### Event emission (R7)
- [ ] Given a skill change is detected, when the debounce window expires, then a `SkillsChanged` event is dispatched on the event bus
- [ ] Given external consumers are subscribed to `SkillsChanged`, then they receive the event for their own processing

### Error handling preserved (R3)
- [ ] Given a newly added skill has invalid SKILL.md (bad YAML, missing description), when the registry re-scans, then the invalid skill is logged as a warning and skipped, and other valid skills load normally
- [ ] Given the re-scan itself fails (e.g., permission error on the directory), then the registry retains its previous valid state and logs the error

### Transparent to provider (R4)
- [ ] Given the provider calls registry methods (`skills` property, `get_agents_for_skill()`), then the interface is unchanged — the dirty check and conditional re-scan happen internally
- [ ] Given the provider is initialized, then no changes to `SkillsContextProvider.provide()` signature are needed

### Session stability preserved (R6)
- [ ] Given skills are detected at session start, when skills change on disk during the session, then subsequent messages in the same session continue using the originally detected skills — guaranteed by the existing coordinator behavior where `provide()` only runs on the first message of a new session (via `is_new_session` guard)

### Watcher lifecycle (R8)
- [ ] Given the application starts, when the skills bootstrap hook runs, then the filesystem watcher is started as a background task monitoring `workspace/skills/`
- [ ] Given the application shuts down, then the watcher task is cancelled gracefully without errors
- [ ] Given the skills directory does not exist at startup, then the watcher handles this gracefully (either waits for creation or starts watching once the directory appears)
