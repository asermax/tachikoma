# Delta Spec: DLT-037 - Ensure Telegram push notifications for streamed responses

## Story

As a user, I want to receive a Telegram push notification for every agent response so that I know the agent replied even when I've closed the app before the response started streaming.

## What It Does

After the agent finishes streaming a response via progressive message editing, the Telegram channel replaces the last streamed message with a fresh copy — creating a new message that triggers a push notification — and then deletes the original. All messages created during streaming (including the initial message, splits, and status messages) are sent silently so the user receives exactly one push notification: for the final, complete response. As a side benefit, the copied message appears after any steering messages the user sent during processing, preserving correct chronological order.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | User receives a Telegram push notification for every agent response, even when the response is delivered via progressive edits | Core goal |
| R1 | All messages created during streaming (initial, splits, status) are sent silently — no push notifications until the response is complete | Must-have |
| R2 | After streaming completes, the last response message is replaced with a new copy that triggers push; the original is deleted. The copied message appears after any user messages sent during processing | Must-have |
| R3 | The copy+delete is performed in safe order: copy first, then delete — if copy fails, the original is preserved | Must-have |
| R4 | For split responses, only the last message is copy+deleted; earlier split messages are left in place | Must-have |
| R5 | The notification mechanism is configurable (enable/disable) | Nice-to-have |

## Acceptance Criteria

**R0/R2 — Push notification via copy+delete**
- [ ] Given the agent streams a response via progressive edits, when the Result event arrives and the response completes, then the last message is copied to create a new message that triggers a push notification, and the original is deleted
- [ ] Given the copy+delete is performed, then it happens after the final edit completes and before the renderer state is cleared for the next turn
- [ ] Given the user sent steering messages while the agent was processing, when the copy completes, then the copied response message appears after the user's messages in the chat

**R1 — Silent streaming messages**
- [ ] Given the agent starts streaming a response, when the initial message is created, then it is sent silently (no push notification for the incomplete message)
- [ ] Given a response is split into multiple messages during streaming, when each split message is created, then it is also sent silently
- [ ] Given a Status event (e.g., "Thinking...") is sent during processing, then it is sent silently

**R3 — Safe ordering**
- [ ] Given the response is complete and copy is attempted, when copy succeeds, then delete is called on the original
- [ ] Given the response is complete and copy is attempted, when copy fails, then delete is NOT called and the original message is preserved as-is
- [ ] Given copy succeeds but delete fails, then the user sees a duplicate message (acceptable degradation) and the failure is logged

**R4 — Split responses**
- [ ] Given a response was split into multiple messages during streaming, when the Result event arrives, then only the last message (the one being actively edited) is copy+deleted
- [ ] Given a response was split, then earlier split messages remain in place unchanged

**R5 — Configurability**
- [ ] Given the notification mechanism is disabled in config, when a response completes, then no copy+delete is performed and messages are sent with normal notification behavior (current behavior)
- [ ] Given the notification mechanism is enabled (default), when a response completes, then the copy+delete flow is used

**Error/edge cases**
- [ ] Given the copy or delete fails (network error, API error), then the failure is logged but does not crash the bot or affect the delivered response
- [ ] Given the response consisted only of an error event (no text streamed), then no copy+delete is performed (the error message itself is a new message that triggers push)
- [ ] Given a Result event arrives with no preceding text, tools, or errors (empty response), then no copy+delete is performed
