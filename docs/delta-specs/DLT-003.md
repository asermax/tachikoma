# DLT-003: Skill System Foundation and Sub-Agent Delegation

## User Story

As the coordinator, I want to load and manage focused sub-agents organized into reusable skill packages so that the agent system can delegate specialized work to agents with appropriate scoped context and tools.

## What It Does

The skill system provides directory-based skill packages that bundle sub-agent definitions. A skill registry discovers all skills at startup and loads agent definitions from each skill's `agents/` subdirectory. The coordinator passes all discovered agents to the Claude Code SDK via `ClaudeAgentOptions.agents`, enabling the SDK's internal orchestrator to delegate work to appropriate sub-agents. Agents are namespaced by skill (e.g., `memory/episodic-extractor`) to prevent collisions. Errors in skill loading are caught gracefully — invalid skills/agents are skipped with warnings, allowing the system to continue.

## Acceptance Criteria

### Skill Definition (R0, R3)

- [ ] Given a skill directory `skills/skill-name/`, when the registry loads it, then it contains a `SKILL.md` file with YAML frontmatter (name, description, version, etc.)
- [ ] Given a SKILL.md file, when parsed, then the YAML frontmatter contains at least: name (string matching folder name), description (string)
- [ ] Given a SKILL.md with empty description, when loaded, then a warning is logged and the skill is skipped
- [ ] Given a SKILL.md with non-matching name (folder name ≠ frontmatter name), when loaded, then a warning is logged and the skill is skipped
- [ ] Given a skills/ directory, when scanning for skills, then only directory entries are considered (regular files ignored)
- [ ] Given a skill directory with no SKILL.md, when the registry loads it, then a warning is logged and the skill is skipped

### Agent Definitions (R2, R3, R4)

- [ ] Given a skill with an `agents/` subdirectory, when the registry loads it, then agent definitions are discovered as `.md` files in that directory
- [ ] Given an agent definition `agents/agent-name.md`, when parsed, then it contains YAML frontmatter with: description (string), model (string, optional), tools (list of strings, optional)
- [ ] Given an agent markdown body, when the agent is loaded, then the markdown body is captured as the agent's system prompt (context)
- [ ] Given an agent with an empty description, when loaded, then a warning is logged and the agent is skipped
- [ ] Given an agent with a model value, when loaded, then the model string is passed through to the AgentDefinition without validation (SDK validates at delegation time)
- [ ] Given an agent definition with a tools list, when loaded, then the tools are included in the AgentDefinition
- [ ] Given an agents/ directory with non-.md files (e.g., `.txt`, `.json`), when scanned, then non-.md files are ignored
- [ ] Given an agent definition with an empty markdown body, when loaded, then the agent is created with an empty system prompt (valid state)
- [ ] Given a skill agents/ directory that doesn't exist, when the skill is loaded, then it is processed without an agents/ subdirectory (valid — skill can be defined without agents)

### Skill Registry (R1, R2)

- [ ] Given skills are placed in `skills/` directory at workspace root, when the registry initializes at startup, then all subdirectories are scanned as potential skills
- [ ] Given valid skills in `skills/`, when the registry loads them, then all valid agents from all skills are discovered and indexed
- [ ] Given multiple skills each with agents, when the registry loads them, then all agents are indexed without collision (agent namespace = "skill-name/agent-name")
- [ ] Given a skill in `skills/` that is invalid (no SKILL.md or bad format), when the registry loads, then a warning is logged and that skill is skipped; other skills load normally
- [ ] Given an empty `skills/` directory, when the registry loads, then it completes with no agents indexed (valid state)
- [ ] Given the registry is loaded with no `skills/` directory present, then the registry is initialized with an empty agents dictionary (valid state)

### Coordinator Integration (R5, R6, R7)

- [ ] Given the coordinator initializes, when it requests agents from the registry, then all discovered agents are passed to `ClaudeAgentOptions.agents`
- [ ] Given agents are passed to the SDK at initialization, when a conversation is active, then the same agents remain available throughout the session (no re-discovery mid-session)
- [ ] Given an agent definition with a tools list, when the coordinator creates the AgentDefinition, then the tools list is included in the AgentDefinition object
- [ ] Given an agent definition without a tools list, when the coordinator creates the AgentDefinition, then tools is omitted or empty (SDK handles default)
- [ ] Given an agent with model specified, when the coordinator creates the AgentDefinition, then the model is included in the AgentDefinition
- [ ] Given an agent with model omitted, when the coordinator creates the AgentDefinition, then model is omitted (SDK applies default)

### Startup Loading (R1, R5, R8)

- [ ] Given the application starts, when the bootstrap runs, then a skills bootstrap hook executes and creates the `skills/` directory if it doesn't exist
- [ ] Given the bootstrap hook completes, when the coordinator initializes, then the skill registry is populated with all discovered agents
- [ ] Given the coordinator is created, when it constructs ClaudeAgentOptions, then the agents dictionary from the registry is passed as the `agents` parameter
- [ ] Given agents are loaded at startup, when the coordinator is ready for messages, then all agents are available for the session

### Context Isolation (R7)

- [ ] Given an agent definition specifies a tools list, when the SDK receives the AgentDefinition, then the agent's tool access is constrained to those tools (SDK enforces scoping at invocation)
- [ ] Given an agent definition omits tools, when loaded, then the agent can use any tool available in the system context (SDK default behavior)

### Error Handling (R9)

- [ ] Given a skill is invalid (missing SKILL.md, bad YAML, missing name), when the registry loads, then a warning is logged with the skill name/path and loading continues
- [ ] Given an agent definition is invalid (missing required fields, bad markdown parsing), when loaded, then a warning is logged with the agent name/skill and the agent is skipped
- [ ] Given a skill directory is present but empty (no SKILL.md, no agents/), when loaded, then a warning is logged and loading continues
- [ ] Given the `skills/` directory doesn't exist when the registry initializes, then it's treated as an empty registry (no error; bootstrap hook creates it on first run)
- [ ] Given registry loading fails for any reason, then errors are caught and logged, and the coordinator continues with whatever agents were successfully loaded

### Bootstrap (R8)

- [ ] Given the bootstrap runs, when the skills hook is registered, then the hook runs after the workspace hook (explicitly ordered)
- [ ] Given the skills hook runs for the first time, when the `skills/` directory doesn't exist, then it is created
- [ ] Given the skills hook runs on subsequent launches, when the `skills/` directory already exists, then no action is taken (idempotent)
- [ ] Given the skills hook fails to create the directory, when it logs an error, then the bootstrap aborts with the error message

## Notes

- Agent definitions in each skill's `agents/` directory are the minimal unit of sub-agent definition for this delta. Automatic detection and loading of which agents to use (DLT-021) is explicitly deferred.
- Custom MCP tools scoped to agents (beyond the standard tools available to the coordinator) is deferred to a later delta.
- Skill-level markdown instructions bundled with the skill (in addition to individual agent instructions) is deferred — agents carry their own context via their markdown bodies.
- The SDK's orchestrator decides when and how to delegate to sub-agents based on agent descriptions, context, and its internal reasoning. The application does not explicitly control delegation decisions.
- Tool scoping via `AgentDefinition.tools` is enforced by the SDK at agent invocation time. The application's responsibility is populating the field correctly from the agent definition.
- Agents are set at coordinator initialization (session creation) and persist for the session lifetime due to the SDK client's internal design.
