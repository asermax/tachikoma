# Delta Spec: DLT-092 - Timezone-aware scheduling for one-shot tasks

## Story

As a user, I want one-shot task schedules specified without an explicit timezone to be interpreted in my configured timezone so that "remind me at 3pm" creates a task that fires at 3pm local time, not 3pm UTC.

## What It Does

Fixes `_parse_schedule` to interpret bare ISO datetimes (e.g. `2026-04-01T15:00:00`) using the configured `tasks.timezone` instead of defaulting to UTC. Centralizes timezone resolution in `TaskSettings` so the config is always a concrete timezone value, eliminating fallback logic in the scheduler. Updates tool descriptions and the system preamble to document datetime format behavior.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Bare datetimes in one-shot schedules are interpreted using the configured timezone, not UTC | Core goal |
| R1 | Tool descriptions for `create_task` and `update_task` document timezone behavior and accepted datetime formats with examples | Must-have |
| R2 | The future-check validation works correctly with timezone-aware datetimes | Must-have |
| R3 | Datetimes with explicit tz offset (e.g. `+05:00`, `Z`) are respected as-is, not overridden | Must-have |
| R4 | Schedule display (`_format_schedule`) shows times in the configured timezone | Must-have |
| R5 | `TaskSettings.timezone` resolves to system timezone by default, eliminating fallback logic in scheduler | Must-have |

## Acceptance Criteria

### Bare datetime interpretation (R0, R3)

- [ ] Given `tasks.timezone` is configured to `"America/Argentina/Buenos_Aires"` (UTC-3), when the agent calls `create_task` with schedule `"2026-04-01T15:00:00"` (no tz info), then the parsed datetime is `2026-04-01T15:00:00-03:00` (interpreted as local time)
- [ ] Given the agent calls `create_task` with schedule `"2026-04-01T15:00:00Z"`, then the parsed datetime is `2026-04-01T15:00:00+00:00` (explicit UTC preserved)
- [ ] Given the agent calls `create_task` with schedule `"2026-04-01T15:00:00+05:30"`, then the parsed datetime is `2026-04-01T15:00:00+05:30` (explicit offset preserved)
- [ ] Given `tasks.timezone` is `"America/Argentina/Buenos_Aires"` (UTC-3), when the agent calls `update_task` with schedule `"2026-04-01T18:00:00"`, then the parsed datetime is `2026-04-01T18:00:00-03:00`

### Future-check validation (R2)

- [ ] Given `tasks.timezone` is `"America/Argentina/Buenos_Aires"` (UTC-3) and current local time is 14:00 ART, when the agent creates a task for `"2026-04-01T15:00:00"`, then the task is accepted (15:00 ART is in the future)
- [ ] Given `tasks.timezone` is `"America/Argentina/Buenos_Aires"` (UTC-3) and current local time is 16:00 ART, when the agent creates a task for `"2026-04-01T15:00:00"`, then the task is rejected as in the past
- [ ] Given the agent creates a task with an explicit UTC datetime that is in the future in UTC, then the task is accepted regardless of local time

> **Note**: The existing future-check comparison (`schedule_config.at <= datetime.now(UTC)`) remains correct after this delta because Python compares timezone-aware datetimes by absolute instant. The fix is entirely in `_parse_schedule` producing the correct tz-aware datetime — the comparison logic itself does not need changing.
>
> **Note**: `update_task` does not currently perform future-check validation on one-shot schedules. This delta does not change that behavior — adding a future-check to `update_task` is a separate concern.

### Schedule display (R4)

- [ ] Given a one-shot task was created with a bare datetime and `tasks.timezone` is `"America/Argentina/Buenos_Aires"`, when `list_tasks` displays the schedule, then it shows the time converted to the configured timezone with a timezone indicator
- [ ] Given a one-shot task was created with an explicit UTC datetime, when `list_tasks` displays the schedule, then it converts and shows the time in the configured timezone

### Tool descriptions (R1)

- [ ] Given the `create_task` tool schema, then the description documents accepted datetime formats: bare ISO datetime (interpreted as configured timezone), ISO with `Z` suffix (UTC), ISO with offset (preserved as-is), and cron expressions
- [ ] Given the `update_task` tool schema, then the description documents the same datetime format behavior for the `schedule` field

### Config timezone resolution (R5)

- [ ] Given `tasks.timezone` is not set in the config file, then `TaskSettings.timezone` resolves to the system's local timezone (not empty string)
- [ ] Given `tasks.timezone` is explicitly set to `"US/Eastern"`, then that value is used
- [ ] Given `tasks.timezone` is set to an invalid value (e.g. `"Fake/Timezone"`), then the application exits with a clear validation error at startup
- [ ] Given the scheduler's `_get_timezone()` function, then it uses `settings.timezone` directly without its own fallback logic

### Preamble documentation (R1)

- [ ] Given the system preamble Tasks/Scheduling section, then it documents that bare ISO datetimes are interpreted in the user's configured timezone
- [ ] Given the system preamble, then it shows example datetime formats including bare, with Z, and with offset

> **Note**: Cron expression behavior is unaffected by this delta — cron evaluation already uses the configured timezone via the scheduler's `_get_timezone()`. This delta only changes one-shot datetime interpretation in the MCP tools layer.
>
> **Note**: The instance generator's comparison `schedule.at <= now_utc` in `scheduler.py` remains correct because Python compares timezone-aware datetimes by absolute instant. No changes needed in the scheduler's one-shot firing logic.
