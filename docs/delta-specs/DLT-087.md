# Delta Spec: DLT-087 - Disable Claude Code built-in skills in default config

## Story

As a developer running Tachikoma, I want the Claude Code `Skill` tool to be blocked by default so that the agent doesn't conflate Claude Code's skill system with Tachikoma's own skill subsystem.

## What It Does

Ensures `"Skill"` is always present in the disallowed tools list regardless of user configuration, by merging a system-level block set with user-configured entries. This prevents the agent from accessing Claude Code's built-in skill/slash-command system, which shadows Tachikoma's own skill subsystem and causes incorrect tool routing. The merge preserves all existing user-configured deny entries alongside the system block.

**Scope**: This delta blocks only `"Skill"`. DLT-073 will later extend the same system-level set to include cron tools (`CronCreate`, `CronDelete`, `CronList`).

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Prevent the agent from accessing Claude Code's built-in `Skill` tool by default | Core goal |
| R1 | Block `"Skill"` via the disallowed tools mechanism | Must-have |
| R2 | The system block is non-overridable — `"Skill"` is present even when user has customized `disallowed_tools` | Must-have |
| R3 | The merge is non-destructive — existing user-configured disallowed tools are preserved alongside the system block | Must-have |
| R4 | User can opt out of the system block if desired | Out |

## Acceptance Criteria

### Core blocking (R0, R1)

- [ ] Given a fresh install with default config (no explicit `disallowed_tools`), when the agent starts, then `"Skill"` is present in the disallowed tools passed to the SDK
- [ ] Given the agent is running with `"Skill"` blocked, when a message is processed, then the agent cannot invoke the Claude Code `Skill` tool

### Merge behavior (R2)

- [ ] Given a user config that explicitly sets `disallowed_tools = ["AskUserQuestion"]`, when loaded, then the final list contains both `"AskUserQuestion"` and `"Skill"`
- [ ] Given a user config that sets `disallowed_tools = []` (empty list), when loaded, then `"Skill"` is still present in the final list
- [ ] Given a user config that already includes `"Skill"` in their `disallowed_tools`, when loaded, then no duplicate entry exists in the resulting `list[str]`

### Existing tools preserved (R3)

- [ ] Given a user config with custom disallowed tools (e.g., `["AskUserQuestion", "WebSearch"]`), when loaded, then all user-specified tools remain blocked alongside `"Skill"`

### Auto-generated config (R1)

- [ ] Given no config file exists, when the default config is auto-generated, then the commented `disallowed_tools` example reflects the effective default including `"Skill"`
