# Delta Spec: DLT-041 - Persist Session Context to Database

## Story

As a developer inspecting or debugging agent behavior, I want a persistent record of what context the agent was given for each session so that I can understand and audit what informed the agent's responses.

## What It Does

Introduces a `SessionContext` model associated with `Session` that persists every context entry injected into a session — both foundational context (SOUL.md, USER.md, AGENTS.md) assembled at startup and first-message-dependent context (memories, projects, skills) gathered by the pre-processing pipeline. Each entry carries an owner identifier for traceability and an injection order for deterministic reassembly. The coordinator saves context entries when a new session starts and they can be queried back for any session.

**Scope exclusions**: The system preamble (`SYSTEM_PREAMBLE` in `context/loading.py`) is static, version-controlled code — not persisted. Topic-shift summaries ("Previous Conversation") and bridging context injected on session resumption are transient mechanisms — not persisted. Only user-authored file contents and pre-processing provider results are in scope.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Persist all context injected into a session as queryable database records tied to the session | Core goal |
| R1 | Each context entry carries an owner identifier tracing it back to its source (e.g., "soul", "memories", "skills") | Must-have |
| R2 | Each context entry carries an injection order — a single monotonic sequence across both kinds (foundational first in file order, then pre-processing in pipeline result order) for deterministic reassembly | Must-have |
| R3 | Distinguish foundational context (assembled at startup from context files) from pre-processing context (gathered by the pipeline on first message) via a context kind field | Must-have |
| R4 | Coordinator saves context entries when a new session is created — both foundational entries (immediately) and pre-processing entries (after pipeline completes) | Must-have |
| R5 | Context entries are queryable by session ID for inspection, debugging, and future tooling (e.g., session resumption refresh in DLT-042/043) | Must-have |
| R6 | Context persistence failures do not interrupt active conversations (graceful degradation, consistent with session tracking pattern) | Must-have |
| R7 | Context entries store the text content that was injected, enabling reconstruction of what the agent saw (excluding the static system preamble, which is code) | Must-have |

## Acceptance Criteria

**Context Entry Model (R0, R1, R2, R3)**

- [ ] Given a session context entry, then it has: session ID (FK), owner identifier (string), content (text), context kind (foundational or pre-processing), and injection order (integer)
- [ ] Given two entries with different owners, when queried, then the owner identifier distinguishes their source
- [ ] Given entries for a session, when loaded in injection order, then they reconstruct the original context assembly sequence — foundational entries first (SOUL=1, USER=2, AGENTS=3), then pre-processing entries continuing the sequence (e.g., memories=4, projects=5, skills=6)

**Saving Foundational Context (R4, R3)**

- [ ] Given the coordinator starts a new session, when foundational context is available, then each source file (SOUL.md, USER.md, AGENTS.md) is persisted as a separate context entry with kind "foundational" and its file name as owner
- [ ] Given a foundational context file does not exist or is empty, when context is saved, then no entry is created for that source
- [ ] Given a topic shift creates a new session, when foundational context is saved, then the same foundational entries are persisted (reflecting the context loaded at startup, held in memory by the coordinator)

**Saving Pre-Processing Context (R4, R3)**

- [ ] Given the pre-processing pipeline completes with results, when the coordinator processes a new session's first message, then each `ContextResult` is persisted as a context entry with kind "pre-processing" and the result's tag as owner
- [ ] Given a provider returns None or fails, when context is saved, then no entry is created for that provider
- [ ] Given pre-processing is skipped (resumed session), when context is saved, then no pre-processing entries are created for that session

**Loading Context (R5, R7)**

- [ ] Given a session ID, when context entries are queried, then all entries for that session are returned ordered by injection order
- [ ] Given a session ID with no context entries, when queried, then an empty list is returned

**Graceful Degradation (R6)**

- [ ] Given a context entry fails to save (database error), when the coordinator is processing a message, then the error is logged and the conversation continues uninterrupted
- [ ] Given context loading fails, when entries are queried, then the error is logged and an empty result is returned

**Content Integrity (R7)**

- [ ] Given foundational context is saved, then the stored content matches the file contents at the time of session creation (the text before XML wrapping)
- [ ] Given a pre-processing result is saved, then the stored content matches the `ContextResult.content` field (the text before XML wrapping)
