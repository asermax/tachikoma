# Delta Spec: DLT-041 - Persist Session Context to Database

## Story

As a developer maintaining the agent system, I need all session context persisted to the database and used as the canonical source for system prompt assembly, so that context survives across per-message client recreations, is available for inspection and debugging, and provides the foundation for context updates and invalidation.

## What It Does

Introduces a `SessionContextEntry` model associated with `Session` that persists every context entry injected into a session. Each entry has an owner identifier (one per context source) and the text content that was injected. Context sources include: foundational files (SOUL.md, USER.md, AGENTS.md as a single entry), pre-processing pipeline results (memories, projects, skills — one entry each), and session transition context (previous summary on topic shifts, bridging context on session resumption).

The persisted entries become the canonical source for system prompt assembly. The coordinator rebuilds the system prompt from database entries on every SDK client creation (a fresh client is created per message). Pre-processing results move from message text injection to system prompt assembly, unifying all context into a single injection path via `SystemPromptPreset.append`. The existing `assemble_context()` function (which prepends provider results to the message text) is superseded by this new assembly path.

A generalized context assembly function loads entries from the database and builds the system prompt append string. This function is usable by both normal SDK client creation and forked sessions (opt-in), enabling forks to restore the parent session's context when needed.

**Scope exclusions**: The system preamble (`SYSTEM_PREAMBLE` in `context/loading.py`) is static, version-controlled code — always prepended by the assembly function but not persisted as an entry. Non-text artifacts from pre-processing (MCP servers, agent definitions) remain in-memory — the database tracks prompt content only.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Persist all context injected into a session as queryable database records tied to the session | Core goal |
| R1 | Each context entry has an owner identifier — one entry per context source / provider (e.g., foundational files as one combined entry, each pre-processing provider as one entry, each session transition artifact as one entry) | Must-have |
| R2 | All context (foundational, pre-processing results, session transition) is assembled from persisted entries into the system prompt append, replacing both the current in-memory prompt construction and message-text context injection | Must-have |
| R3 | The coordinator rebuilds the system prompt from persisted entries on every SDK client creation, making the database the source of truth | Must-have |
| R4 | A generalized context assembly function loads entries for a session and builds the system prompt, usable by both normal client creation and forked sessions | Must-have |
| R5 | Forked sessions can opt into context restoration — the assembly function builds the system prompt append string which is set on the fork's options | Must-have |
| R6 | Entries are queryable by session ID for inspection, debugging, and future tooling (context invalidation in DLT-042/043) | Must-have |
| R7 | Context persistence failures do not interrupt active conversations (graceful degradation, consistent with session tracking pattern) | Must-have |
| R8 | Stored content matches the text at the time of injection, enabling reconstruction of what the agent saw | Must-have |

## Acceptance Criteria

**Context Entry Model (R0, R1)**

- [ ] Given a session context entry, then it has: session ID (FK), owner identifier (string), and content (text)
- [ ] Given a full new session with topic shift, then entries include owners: foundational, previous-summary, memories, projects, skills

**Assembly Ordering Convention (R2)**

- [ ] Given entries are assembled into the system prompt, then they follow a fixed owner priority: foundational → previous-summary → bridging-context → pre-processing providers (memories, projects, skills)
- [ ] Given the same set of entries, when assembled at different times, then the resulting system prompt is identical (deterministic)

**Saving Foundational Context (R0, R1)**

- [ ] Given the coordinator creates a new session, when foundational context files exist, then a single entry is saved with owner "foundational" containing the combined content of SOUL.md, USER.md, and AGENTS.md
- [ ] Given all foundational context files are missing or empty, when a session starts, then no foundational entry is created
- [ ] Given a topic shift creates a new session, then the foundational entry reflects the context files loaded at startup (held in memory by the coordinator)

**Saving Pre-Processing Context (R0, R1)**

- [ ] Given the pre-processing pipeline completes, when processing a new session's first message, then each `ContextResult` is saved as an entry with the result's tag as owner
- [ ] Given a provider returns None or fails, then no entry is created for that provider
- [ ] Given pre-processing is skipped (resumed session), then no new pre-processing entries are created — the existing entries from the original session are loaded instead

**Saving Session Transition Context (R0, R1)**

- [ ] Given a topic shift creates a new session with a previous summary, then a context entry is saved with owner "previous-summary"
- [ ] Given session resumption produces bridging context, then a context entry is saved with owner "bridging-context"
- [ ] Given no topic shift or resumption occurs, then no transition entries are created

**System Prompt Assembly (R2, R3)**

- [ ] Given a session with persisted entries, when an SDK client is created, then the system prompt append is assembled from: SYSTEM_PREAMBLE (hardcoded) + persisted entries (each wrapped in XML tags by owner, following the ordering convention)
- [ ] Given pre-processing results are persisted, then they appear in the system prompt append — not prepended to the message text (superseding `assemble_context()`)
- [ ] Given the coordinator creates a client for subsequent messages in a session, then context is loaded from the database on each client creation — not from in-memory state

**Fork Context Restoration (R4, R5)**

- [ ] Given a fork helper is called with context restoration enabled, then the assembly function builds the system prompt append from the parent session's entries, set on the fork's `ClaudeAgentOptions`
- [ ] Given a fork helper is called without context restoration (default), then behavior is unchanged — no session context injected

**Querying (R6)**

- [ ] Given a session ID, when context entries are queried, then all entries for that session are returned
- [ ] Given a session ID with no context entries, then an empty list is returned

**Graceful Degradation (R7)**

- [ ] Given a context entry fails to save, then the error is logged and the conversation continues
- [ ] Given context loading fails during prompt assembly, then the error is logged, the system prompt falls back to SYSTEM_PREAMBLE only (no dynamic context entries), and the conversation continues with reduced context

**Content Integrity (R8)**

- [ ] Given foundational context is saved, then the stored content matches the combined file contents at the time of session creation
- [ ] Given a pre-processing result is saved, then the stored content matches the `ContextResult.content` field
- [ ] Given transition context is saved, then the stored content matches the summary/bridging text produced at session creation time
