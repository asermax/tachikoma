# Delta Spec: DLT-006 - Pre-process messages with memory context injection

## Story

As a user, I want my assistant to automatically recall relevant past conversations, known facts, and preferences when I send a message so that responses are contextually aware without me having to repeat myself.

## What It Does

Before the coordinator processes a user message, a pre-processing pipeline runs registered context providers in parallel to gather relevant context. Each provider returns a named, XML-tagged context block (consistent with the existing `<soul>`, `<user>`, `<agents>` convention). The pipeline assembles all results and prepends them to the user message before the agent sees it. Pre-processing runs once per session (on the first message only) — subsequent messages benefit from the conversation history that already includes the enriched context. The first provider — a memory context provider — uses an Opus agent with low effort and search tools (Read, Glob, Grep) to search stored memories (`memories/episodic/`, `memories/facts/`, `memories/preferences/`) and return a ranked list of the most relevant memory files with short summaries, giving the main agent pointers to read further if necessary. This is an agent-based exploration approach; DLT-009 (embedding-based semantic search) is a potential future upgrade to the retrieval mechanism. The architecture is designed so future providers (skills, calendar, email, notes) can be added by registering a new provider — no changes to the pipeline or coordinator.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Before the coordinator processes a user message, automatically gather and inject relevant context so responses are informed by past interactions | Core goal |
| R1 | Reusable pre-processing pipeline that runs registered context providers in parallel and collects their results | Must-have |
| R2 | Context provider interface (ABC) that is domain-agnostic — the pipeline knows nothing about what providers do | Must-have |

| R6 | Context injection uses XML tags consistent with the existing `<soul>`, `<user>`, `<agents>` convention, generalized for easy addition of new context sources | Must-have |
| R7 | Pipeline extensible — adding a new context provider requires only implementing the ABC and registering it; no changes to pipeline or coordinator code | Must-have |

## Acceptance Criteria

### Pre-Processing Pipeline (R0, R1, R2, R7)

- [ ] Given one or more context providers are registered, when a user message arrives, then the pipeline runs all providers in parallel before the coordinator passes the message to the agent
- [ ] Given multiple providers are registered, when the pipeline runs, then all providers execute concurrently (not sequentially)
- [ ] Given multiple providers are registered, when they run in parallel, then each provider operates independently with no access to other providers' results
- [ ] Given the pipeline completes with results, when the coordinator processes the message, then the assembled context XML blocks are prepended to the user message text passed to `client.query()`
- [ ] Given no providers are registered, when a user message arrives, then the message is sent to the agent unmodified
- [ ] Given a new provider is implemented and registered, when the pipeline runs, then it is included in the parallel execution without changes to the pipeline or coordinator
- [ ] Given the pipeline is running, when the user is waiting for a response, then no separate status notification is needed — pre-processing latency is part of the normal response time

### Error Isolation (R3)

- [ ] Given a provider raises an exception, when other providers are running, then they complete normally
- [ ] Given a provider raises an exception, when the pipeline collects results, then the failure is logged and the message proceeds with results from successful providers only
- [ ] Given all providers fail, when the pipeline completes, then the message is sent to the agent unmodified (no context injected)

### Context Injection (R5, R6)

- [ ] Given providers return context results, when the pipeline assembles them, then each result is wrapped in XML tags using the provider's declared tag name (e.g., `<memories>...</memories>`)
- [ ] Given assembled context blocks exist, when the coordinator sends the message to the agent, then the blocks are prepended to the original user message text in the `client.query()` call
- [ ] Given the XML tag convention, when context is injected, then it is consistent with the existing foundational context tags (`<soul>`, `<user>`, `<agents>`)
- [ ] Given no providers return results (all returned None or all failed), when the coordinator sends the message, then the original message is sent unmodified

### Memory Context Provider (R4)

- [ ] Given a user message related to previously stored memories (e.g., "What was that restaurant I liked?"), when the memory provider runs, then it returns a list of up to 10 relevant memory file paths with a short summary for each
- [ ] Given the memory provider returns results, when the context is injected, then it includes instructions telling the main agent to read the referenced files if it needs more detail
- [ ] Given a user message with no related memories, when the memory provider runs, then it returns no context (None)
- [ ] Given the memories directory is empty, when the memory provider runs, then it returns no context without error
- [ ] Given the memory provider runs, when it makes its agent call, then it uses an Opus model with effort="low", max_turns=8, and tools (Read, Glob, Grep), operating from the workspace directory
- [ ] Given the memory provider's agent explores memory files, when it finds relevant content, then it returns a concise ranked list of relevant files with summaries (not raw file dumps)
