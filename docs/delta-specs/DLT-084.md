# Delta Spec: DLT-084 - Resume matching conversation on return after restart

## Story

As a user, I want the system to match my first message after a restart against recently discussed topics so that I seamlessly resume the relevant conversation instead of always starting fresh.

## What It Does

When the system has no active session (typically after a restart), it queries recent closed session summaries and runs boundary detection in candidates-only mode — comparing the incoming message solely against previous session summaries, without a current-topic baseline. If a match is found, that session is reopened with full SDK continuity and bridging context. If no match is found, a fresh session is created normally. Additionally, candidate filtering is hardened universally to exclude sessions that cannot be meaningfully resumed.

Candidates-only mode means the detector compares the message against candidate session summaries with no current-topic reference; the `continues_conversation` field is always `false` (there is nothing to continue), and the match outcome is conveyed solely via `resume_session_id`.

## Acceptance Criteria

### Startup session matching (R0, R1, R2, R6)

- [ ] Given no active session exists and recent closed sessions are available within the resume window, when a message arrives, then the system queries candidates and runs boundary detection in candidates-only mode (no current-topic summary)
- [ ] Given no active session exists and no recent closed sessions exist, when a message arrives, then a fresh session is created without invoking boundary detection
- [ ] Given no active session exists, when boundary detection runs in candidates-only mode, then it completes within the same ~1-2 second latency budget as normal boundary detection
- [ ] Given the system has a configured `session_resume_window`, when querying candidates at startup, then only sessions closed within that window are considered
- [ ] Given boundary detection runs in candidates-only mode, when it returns a result, then `continues_conversation` is always `false` (there is no current conversation to continue) and the match outcome is conveyed solely via `resume_session_id` (non-null for match, null for no match)
- [ ] Given no active session exists and candidates are available, when startup matching begins, then the user sees a "Thinking..." status indicator

### Startup session resumption on match (R3, R7)

- [ ] Given no active session exists and candidates-only detection returns a matching `resume_session_id`, when the coordinator processes the result, then the matched session is reopened directly — no previous session is closed and no post-processing fires (since none exists)
- [ ] Given a session is reopened after startup matching, when bridging context is assembled, then summaries of sessions that occurred between the matched session's close and now are injected into the system prompt
- [ ] Given a session is reopened after startup matching, when the SDK client connects, then it uses the resumed session's `sdk_session_id` for conversation continuity

### Fresh session fallback (R4, R5)

- [ ] Given boundary detection returns no match (all candidates rejected), when the coordinator processes the result, then a fresh session is created normally with full pre-processing
- [ ] Given boundary detection fails with an error or timeout, when the coordinator handles the failure, then a fresh session is created normally (fail-open)
- [ ] Given `reopen_session()` fails validation (missing transcript, stale session), when the coordinator handles the failure, then it falls back to creating a fresh session

### Universal candidate filtering (R8)

- [ ] Given sessions exist with no `sdk_session_id` (interrupted), when candidates are queried for any boundary detection (startup or normal), then those sessions are excluded
- [ ] Given sessions exist with missing transcript files, when candidates are queried, then those sessions are excluded
- [ ] Given sessions exist outside the max age window, when candidates are queried, then those sessions are excluded

Note: The repository query already filters `sdk_session_id IS NOT NULL` and `summary IS NOT NULL`; the registry already checks transcript existence and max age. R8 formalizes and verifies this coverage rather than adding fundamentally new filters.

## Out of Scope

- **Active session with no summary**: When an active session exists but hasn't received a response yet (no summary), boundary detection is skipped per existing behavior. This delta does not change that path — it only addresses the "no active session" case.
