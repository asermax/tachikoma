# Delta Spec: DLT-008 - Extract and store memories from conversations

## Story

As the system, I need to automatically extract and persist learnings from completed conversations so that the assistant can reference past interactions, known preferences, and prior decisions in future sessions — making conversations contextually aware over time.

## What It Does

After a conversation ends, a post-processing pipeline triggers memory extraction by forking the original SDK session — once per memory type, in parallel. Each forked agent receives a tailored prompt directing it to analyze the conversation, read the relevant memory subdirectory, and autonomously create, update, or delete memory files as needed. The processor code does no file I/O — it forks the session, sends the prompt, and consumes the result. Three memory types for v1: episodic (date-stamped conversation summaries), facts (named files about the user), and preferences (named files about how the user likes things). All memories are plain markdown files in the workspace, keeping them human-readable and directly editable. The pipeline is reusable — DLT-018 and future post-processors plug in without modifying the core mechanism. Session forking uses the SDK's standalone `query()` function, which operates independently of the coordinator's `ClaudeSDKClient`.

## Acceptance Criteria

### Post-processing pipeline (R1, R1.1, R1.2, R1.3, R7)

- [ ] Given a session closes with a valid SDK session ID, when the coordinator triggers post-processing, then the pipeline runs all registered processors in parallel
- [ ] Given the pipeline is running, when all processors complete (success or failure), then the pipeline returns and shutdown/next-session proceeds
- [ ] Given one processor fails (e.g., LLM error, network error, file system error), when other processors are running, then the failing processor's error is logged and the others complete normally
- [ ] Given a session closes without an SDK session ID (interrupted session from crash recovery), when post-processing would be triggered, then the pipeline skips processing and logs a warning
- [ ] Given a session with minimal content (e.g., user connected but said little), when the pipeline runs, then all processors execute normally — each processor's forked agent independently determines whether the conversation warrants any memory entries
- [ ] Given a new post-processor is registered with the pipeline, then it runs alongside existing processors without changes to the pipeline or existing processors
- [ ] Given a pipeline is already running from a previous session close, when another session closes, then the new pipeline run is serialized (awaits the previous one before starting)

### Session forking and memory extraction (R2, R2.1, R2.2)

- [ ] Given a closed session with a valid SDK session ID, when a memory processor runs, then it calls the standalone `query()` function with `resume=sdk_session_id` and `fork_session=True` — this is independent of the coordinator's `ClaudeSDKClient` and does not require the client to be connected
- [ ] Given a forked agent session, when the extraction prompt is sent, then the agent has full workspace access and operates from the workspace directory
- [ ] Given a forked agent for a specific memory type, when it executes, then the agent (LLM) autonomously reads the corresponding subdirectory and creates, updates, or deletes files — the processor code performs no file I/O
- [ ] Given a forked session, when the agent completes its extraction, then the async iterator is fully consumed and the forked session ends cleanly
- [ ] Given a memory processor attempts to fork a session and the `query()` call fails (e.g., network error, invalid session ID), then the processor raises an exception that is caught by the pipeline per R1.3

### Extraction prompt requirements (R2.1, R2.2, R3)

- [ ] Given any memory extraction prompt, then it instructs the agent to read the specific memory subdirectory before making changes (to see existing files)
- [ ] Given the episodic extraction prompt, then it instructs the agent to use date-stamped filenames (e.g., `YYYY-MM-DD.md`)
- [ ] Given the facts or preferences extraction prompt, then it instructs the agent to use descriptive, topic-based filenames (e.g., `work-info.md`, `code-style.md`)
- [ ] Given any extraction prompt, then it instructs the agent to only modify files within its assigned subdirectory
- [ ] Given any extraction prompt, then it instructs the agent that creating no files is acceptable when the conversation contains nothing relevant for that memory type

### Episodic memories (R3.1)

- [ ] Given a completed conversation with meaningful content, when the episodic processor runs, then the forked agent creates or updates a date-stamped memory file in `memories/episodic/` summarizing what happened
- [ ] Given multiple conversations on the same day, when the episodic processor runs, then the extraction prompt instructs the agent to consolidate entries for that day rather than creating duplicates
- [ ] Given a trivial conversation, when the episodic processor runs, then the system allows the agent to determine there's nothing meaningful to record — no file creation is forced

### Facts memories (R3.2)

- [ ] Given a conversation where the user reveals new factual information (e.g., "I started a new job at X"), when the facts processor runs, then the forked agent creates a new file in `memories/facts/` or updates an existing file if the topic already exists
- [ ] Given a conversation where previously stored factual information is contradicted, when the facts processor runs, then the agent updates the existing memory file with the corrected information
- [ ] Given a conversation where a previously stored fact becomes invalid, when the facts processor runs, then the extraction prompt instructs the agent that deleting obsolete files is acceptable

### Preferences memories (R3.3)

- [ ] Given a conversation where the user expresses a preference (e.g., "I prefer TypeScript over JavaScript"), when the preferences processor runs, then the forked agent creates a new file or updates an existing file in `memories/preferences/`
- [ ] Given a conversation where the user changes a previously expressed preference, when the preferences processor runs, then the agent updates or deletes the existing memory file
- [ ] Given a conversation with no preference-related content, when the preferences processor runs, then no changes are made to `memories/preferences/`

### Directory structure and bootstrap (R4, R6)

- [ ] Given no `memories/` directory in the workspace, when the memory bootstrap hook runs, then `memories/`, `memories/episodic/`, `memories/facts/`, and `memories/preferences/` are created
- [ ] Given the `memories/` directory structure already exists, when the hook runs, then nothing changes (idempotent)
- [ ] Given the bootstrap system, when hooks are registered, then the memory hook is registered after the workspace hook

### Human readability (R5)

- [ ] Given memory files exist, when the user navigates to the workspace `memories/` directory, then they can read and understand the contents in any text editor
- [ ] Given a user manually edits a memory file, when the next extraction runs, then the agent sees the user's edits and respects them

## Notes

- Forked sessions have no `max_turns` or `max_budget_usd` limits for v1. This is a conscious trade-off — the extraction prompts are designed to be focused (read existing files, analyze conversation, make targeted changes), so sessions should be naturally short. If cost becomes an issue, limits can be added later without changing the architecture.
- Memory extraction quality (precision, recall, accuracy of categorization) is inherently an LLM behavioral concern. Testing extraction quality is DLT-017's domain, not this spec's.
