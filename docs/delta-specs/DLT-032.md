# Delta Spec: DLT-032 - Guide the assistant through skill authoring

## Story

As a developer extending the system, I want the assistant to guide me through creating a new skill so that I can produce well-structured skills without reading external documentation.

## What It Does

Provides a built-in skill that activates when a user asks to create, define, or set up a new skill. The skill injects comprehensive authoring guidance into the assistant's context — directory conventions, SKILL.md format, agent definition format, detection tuning tips, and prompt-writing best practices — enabling the assistant to scaffold well-structured skills through an interactive workflow.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | A built-in skill provides the assistant with comprehensive skill authoring knowledge, enabling it to scaffold well-structured skills on demand | Core goal |
| R1 | Developers need to understand where skill files go and how directories are structured (folder layout, SKILL.md format, agents/ subdirectory) | Must-have |
| R2 | Developers need to understand the agent definition format (frontmatter fields, model options, tool scoping, prompt body) | Must-have |
| R3 | Developers need detection tuning guidance — how description quality affects LLM classification, tips for writing effective descriptions | Must-have |
| R4 | Developers need prompt-writing best practices for agent system prompts | Must-have |
| R5 | Developers benefit from knowing available capabilities (what skills can provide today: agents, context body; what's planned: MCP tools) | Nice-to-have |
| R6 | Skill's own description triggers correctly on authoring-related messages and avoids false activation on non-authoring skill mentions | Must-have |
| R7 | Skill provides a step-by-step workflow the assistant can follow when scaffolding a new skill | Nice-to-have |
| R8 | Deployment mechanism handles the case where the guide already exists in the workspace (e.g., from a previous version) | Must-have |

## Acceptance Criteria

### Core Guidance (R0, R1, R2)

- [ ] Given the skill authoring guide skill exists in the workspace, when a user asks to create a new skill, then the skill system detects and injects the authoring guidance into the assistant's context
- [ ] Given the authoring guidance is injected, when the assistant reads it, then it has enough information to create a valid SKILL.md file with correct YAML frontmatter (description, optional version) and markdown body
- [ ] Given the authoring guidance is injected, then it describes the expected directory layout: `skills/<skill-name>/SKILL.md` with optional `agents/` subdirectory
- [ ] Given the authoring guidance is injected, then it documents agent definition format: markdown files in `agents/` with YAML frontmatter (description required, model and tools optional) and markdown body as the agent's system prompt
- [ ] Given the guidance covers agent definitions, then it lists valid model options (sonnet, opus, haiku, inherit) and explains tool scoping
- [ ] Given the authoring guidance is injected, then it includes at least one complete example of a SKILL.md file and one complete example of an agent definition file

### Detection Tuning (R3)

- [ ] Given the authoring guidance is injected, then it explains how the skill detection system works (LLM classification using skill descriptions)
- [ ] Given the guidance covers detection, then it provides concrete tips for writing descriptions that trigger on relevant messages and avoid false matches
- [ ] Given the guidance covers detection, then it explains the tradeoff between precision (avoiding irrelevant activation) and recall (not missing relevant messages)

### Prompt-Writing Best Practices (R4)

- [ ] Given the authoring guidance is injected, then it includes best practices for writing effective agent system prompts (clarity, scoping, tool usage guidance)

### Available Capabilities (R5)

- [ ] Given the authoring guidance is injected, then it describes what skills can provide today (agents with custom prompts, skill body as context, optional subdirectories for references/data)
- [ ] Given the guidance covers capabilities, then it notes planned future capabilities (MCP tool servers via DLT-054) without documenting them as available

### Skill Self-Detection (R6)

- [ ] Given the skill authoring guide has a SKILL.md with a description, when a user sends a message about creating, defining, or setting up a skill, then the skills context provider classifies the guide as relevant
- [ ] Given a user sends a message unrelated to skill authoring (e.g., "what's the weather"), then the skill authoring guide is not activated
- [ ] Given a user sends a message that mentions "skill" in a non-authoring context (e.g., "what skills do you have?", "use the calendar skill"), then the skill authoring guide is not activated

### Authoring Workflow (R7)

- [ ] Given the skill provides a step-by-step workflow, when the assistant follows it, then it guides the user through: naming the skill, writing the description, creating the SKILL.md, optionally defining agents, and verifying the structure
- [ ] Given the step-by-step workflow section, then it ends with a verification step that instructs the assistant to confirm the created skill directory matches the expected structure (SKILL.md with valid frontmatter, optional agents/ subdirectory)

### Deployment Lifecycle (R8)

- [ ] Given the skill authoring guide does not exist in the workspace, when the deployment mechanism runs, then the guide is created with the latest bundled content
- [ ] Given the skill authoring guide already exists in the workspace, when the deployment mechanism runs, then it handles the existing guide without data loss (specific strategy — overwrite, version check, or skip — is a design decision)
