# Delta Spec: DLT-032 - Guide the assistant through skill authoring

## Story

As a user extending the system, I want the assistant to guide me through creating a new skill so that I can produce well-structured skills without reading external documentation.

## What It Does

Provides a built-in skill that activates when a user asks to create a skill, encode a workflow, or define reusable automation. The skill injects comprehensive authoring guidance into the assistant's context — directory conventions, SKILL.md format, available capabilities (agents, reference files), detection tuning tips, and writing best practices — as a reference document with concise examples. A dedicated reference file covers agent definitions in detail.

To ship this skill without manual deployment, a new built-in skill source within the codebase is introduced, discovered by the registry alongside workspace skills. The static Skills section in the system prompt is trimmed to avoid duplicating structural details now covered by the guide.

## Acceptance Criteria

### Core Guidance (R0, R1)

- [ ] Given the skill authoring guide exists as a built-in skill, when a user asks to create a new skill or encode a workflow, then the skill system detects and injects the authoring guidance into the assistant's context
- [ ] Given the authoring guidance is injected, when the assistant reads it, then it has enough information to create a valid SKILL.md file with correct YAML frontmatter (description required, version optional) and markdown body
- [ ] Given the authoring guidance is injected, then it describes the expected directory layout: `skills/<skill-name>/SKILL.md` with optional subdirectories
- [ ] Given the skill authoring guide is detected, when it is injected, then no sub-agents are loaded for this skill (it is content-only)

### Available Capabilities (R2)

- [ ] Given the authoring guidance is injected, then it explains available capability types: agents for sub-task delegation, reference files for detailed documentation loaded on demand, and other organizational subdirectories
- [ ] Given the guidance covers capabilities, then it explains when to use each — agents for specialized delegated tasks, references for large content that shouldn't live in the main body, etc.
- [ ] Given the guidance covers reference files, then it clarifies that references are regular files in the skill directory that the agent reads via file-system tools when directed by the SKILL.md body (no automatic injection)

### Agent Reference (R3)

- [ ] Given the skill includes a `references/agents.md` file, then it documents agent definition format: markdown files in `agents/` with YAML frontmatter (description required, model and tools optional) and markdown body as the agent's system prompt
- [ ] Given the agent reference, then it lists valid model options (sonnet, opus, haiku, inherit) and explains tool scoping
- [ ] Given the main SKILL.md, then it explains when agents are useful and points to the reference file for detailed format

### Detection Tuning (R4)

- [ ] Given the authoring guidance is injected, then it explains how the skill detection system works (LLM classification using skill descriptions)
- [ ] Given the guidance covers detection, then it provides concrete tips for writing descriptions that trigger on relevant messages and avoid false matches

### Examples (R5)

- [ ] Given the authoring guidance is injected, then it includes at least one concise example of a complete SKILL.md file
- [ ] Given the agent reference file, then it includes at least one concise example of an agent definition file

### Skill Self-Detection (R6)

- [ ] Given a user sends a message about creating, defining, or setting up a skill, then the skill authoring guide is classified as relevant
- [ ] Given a user sends a message about encoding a workflow, saving a process for reuse, or defining reusable automation for the assistant, then the skill authoring guide is classified as relevant
- [ ] Given a user sends a message unrelated to skill authoring (e.g., "what's the weather"), then the skill authoring guide is not activated
- [ ] Given a user sends a message that mentions "skill" in a non-authoring context (e.g., "what skills do you have?", "use the calendar skill"), then the skill authoring guide is not activated

> Note: R6 criteria depend on LLM classification quality, which is non-deterministic. They are primarily validated through the skill description's design and can be more rigorously verified via the evaluation framework (DLT-022).

### Built-in Skill Source (R7)

- [ ] Given built-in skills exist in the codebase source directory, when the skill registry initializes, then it discovers both built-in and workspace skills
- [ ] Given a workspace skill has the same name as a built-in skill, when the registry loads both, then the workspace skill entirely replaces the built-in (all metadata, body, and agents are from the workspace copy); built-in sources are scanned first, workspace sources second
- [ ] Given no workspace skills exist, when the registry loads, then built-in skills are still discoverable
- [ ] Given the built-in skill source directory cannot be resolved at runtime (e.g., corrupted installation), when the registry initializes, then it logs a warning and continues with workspace skills only

### System Prompt Trimming (R8)

- [ ] Given the system prompt is assembled, then the Skills section retains: awareness that skills exist, that they're auto-detected per session, that the agent can create and manage skills, and the distinction from Claude Code's native skills
- [ ] Given the system prompt is assembled, then the Skills section no longer includes: SKILL.md format description, agents/ subdirectory description, YAML frontmatter field listing, or optional subdirectory listing (these are now covered by the authoring guide skill)

### Writing Tips (R9)

- [ ] Given the authoring guidance is injected, then it includes tips for writing effective skill content (explain the reasoning, keep it lean, use progressive disclosure)
- [ ] Given the agent reference file, then it includes tips for writing effective agent prompts

### Future Capabilities (R10)

- [ ] Given the authoring guidance is injected, then it notes planned future capabilities (MCP tool servers via DLT-054) without documenting them as available
