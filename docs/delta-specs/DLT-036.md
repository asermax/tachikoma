# Delta Spec: DLT-036 - Auto-close idle sessions

## Story

As a user, I want idle sessions to close automatically so that post-processing (memory extraction, context updates, git commit) runs without me needing to change topics or restart the app.

## What It Does

Adds a periodic check that closes the active session after a configurable period of inactivity, measured from the last message exchange completion. When the idle timeout fires, it follows the same session close path used by boundary detection — closing the session in the registry, triggering async post-processing, and clearing SDK state. If the coordinator is busy (processing a message or a session task is in-flight), the close is snoozed for 5 minutes and retried. The next user message creates a new session through the normal first-message path. This complements boundary-detection-based closing by handling conversations that simply trail off.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Auto-close the active session after a configurable inactivity period so post-processing triggers without user action | Core goal |
| R1 | Configurable idle timeout in seconds via `[agent]` config section, default 900 (15 min) | Must-have |
| R2 | Setting timeout to 0 disables idle-close entirely | Must-have |
| R3 | Timer measures inactivity from the completion of the most recent message exchange | Must-have |
| R4 | When timeout fires: close session in registry, trigger async post-processing, clear SDK session state, store previous summary for next session | Must-have |
| R5 | After idle close, next user message follows normal first-message path (new session, pre-processing, possible resumption via later boundary detection) | Must-have |
| R6 | If idle timeout is reached but a message is being processed or a session task is in-flight (dispatched but not yet completed), snooze for 5 minutes and retry | Must-have |
| R7 | Idle close does not fire if there is no active session or no message exchange has occurred | Must-have |
| R8 | Idle close failures are logged but never crash the application; loop is cleanly cancellable on shutdown | Must-have |
| R9 | Idle timeout is independent of the task scheduler's idle window | Must-have |

## Acceptance Criteria

### Idle Timeout Configuration (R1, R2)

- [ ] Given `[agent]` section has `session_idle_timeout = 600`, when loaded, then 600 seconds is used as the idle timeout
- [ ] Given no `session_idle_timeout` specified, when loaded, then default 900 seconds (15 min) is used
- [ ] Given `session_idle_timeout = 0`, when loaded, then idle-close is disabled entirely — no periodic check runs
- [ ] Given auto-generated default config, when created, then `session_idle_timeout` is included (commented out) with annotation

### Periodic Check (R3)

- [ ] The idle-close check runs at a fixed 60-second interval (acceptable delay between actual timeout expiry and close execution is up to 60 seconds)
- [ ] Given an active session and idle timer running, when a message exchange completes (agent response finished), then the idle elapsed time resets
- [ ] Given the idle timer was close to firing, when a new message arrives and processing begins, then idle-close does not fire during the exchange (snooze applies per R6)

### Idle Close Trigger (R0, R4)

- [ ] Given an active session and no message exchange for longer than `session_idle_timeout`, when the periodic check runs and the coordinator is not busy, then the session is closed in the registry and async post-processing is triggered
- [ ] Given a session closed due to idle timeout, then SDK session state (`_sdk_session_id`) is cleared so the next message creates a fresh SDK session
- [ ] Given a session closed due to idle timeout that had a summary, then the summary is stored as `_previous_summary` for injection into the next session's system prompt
- [ ] Given the active session has no `sdk_session_id` (session created but no agent response yet), when idle timeout fires, then the session is closed but post-processing is skipped (same behavior as boundary detection)

### Snooze on Busy (R6)

The busy check has two layers:
1. **Coordinator-level**: a message exchange is in progress (`send_message` active — covers both user messages and session task delivery once it reaches the coordinator)
2. **Task dispatch-level**: a `SessionTaskReady` event has been dispatched but the channel hasn't completed delivery yet (the gap between event dispatch and `send_message()` completion that the coordinator alone cannot observe)

- [ ] Given idle timeout has elapsed but a message exchange is in progress at the coordinator, when the idle check runs, then close is deferred and retried after 5 minutes
- [ ] Given idle timeout has elapsed but a session task has been dispatched and is awaiting or in-flight delivery (between `SessionTaskReady` dispatch and delivery completion), when the idle check runs, then close is deferred and retried after 5 minutes
- [ ] Given idle close was snoozed, when 5 minutes pass and the coordinator is no longer busy, then idle close proceeds
- [ ] Given idle close was snoozed, when 5 minutes pass but the coordinator is still busy, then another snooze is applied (repeating until conditions clear)
- [ ] Given idle close was snoozed but a new message exchange completed during the snooze period, then the idle elapsed time resets and the idle timeout starts fresh (snooze is effectively cancelled)
- [ ] Given `session_idle_timeout` is less than 300 seconds, when a snooze occurs, then the snooze duration is capped to `session_idle_timeout` (snooze never exceeds the timeout itself)

### Post-Idle Behavior (R5)

- [ ] Given a session was closed due to idle timeout, when the user sends a new message, then a new session is created following the normal first-message path (including pre-processing context assembly)
- [ ] Given a session was closed due to idle timeout, when a session task fires after the close, then the task follows the normal `send_message` path (creates a new session if none exists)

### No Active Session (R7)

- [ ] Given no active session exists, when the idle check runs, then idle-close is skipped
- [ ] Given an active session exists but no message exchange has occurred yet (`_last_message_time` is None), when the idle check runs, then idle-close is skipped (cannot determine inactivity without a reference point)

### Graceful Degradation (R8)

- [ ] Given the idle-close check encounters an error (e.g., registry failure), then the error is logged and the check loop continues on the next cycle
- [ ] Given idle-close fails mid-execution, then the application continues running and the next cycle retries
- [ ] Given idle-close is pending or snoozed, when the application shuts down, then the idle-close loop is cancelled cleanly and the existing shutdown session-close logic handles the session (no race between idle-close and `__aexit__`)

### Independence from Task Scheduler (R9)

- [ ] Given `session_idle_timeout = 900` and `tasks.idle_window = 300`, when the user has been idle for 6 minutes, then session tasks fire but the session remains open (task idle window < session idle timeout)
