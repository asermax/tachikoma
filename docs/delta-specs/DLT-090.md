# Delta Spec: DLT-090 - Prevent duplicate task instances from scheduler

## Story

As a user, I want each scheduled task to produce exactly one notification per cron period so that I don't receive duplicate messages for the same scheduled execution.

## What It Does

Ensures each cron period produces exactly one task instance per definition by tightening the firing condition and broadening the duplicate check. The instance generator only fires when a cron match time has definitively passed, and a safety net verifies no instance already exists for that period across all non-failed states.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Each cron match produces at most one task instance per definition | Core goal |
| R1 | Cron evaluation only fires when the scheduled time has definitively passed (no tolerance/lookahead); firing up to `GENERATION_INTERVAL_SECONDS` late is acceptable | Must-have |
| R2 | Duplicate instance safety net considers pending, running, and completed instances within the current cron period; failed instances are excluded to allow retry | Must-have |
| R3 | Log when a task was about to fire but was found to have already run for the current period | Must-have |
| R4 | Existing catch-up-on-restart behavior preserved (one catch-up instance per missed definition) | Must-have |
| R5 | One-shot task behavior unchanged | Must-have |
| R6 | No regression in cron evaluation accuracy (tasks still fire on correct schedule) | Must-have |

## Acceptance Criteria

### Cron period deduplication (R0, R1)

- [ ] Given an enabled cron task definition, when the instance generator evaluates it and the next cron match has not yet passed, then no instance is created (no early firing)
- [ ] Given an enabled cron task definition, when the next cron match time has passed and no instance exists for that period, then exactly one pending instance is created with `scheduled_for` set to the cron match time
- [ ] Given a cron task that fired on the previous generator iteration, when the same cron period is still current, then no additional instance is created
- [ ] Given the generator loop wakes up slightly before a cron minute boundary, then it does not fire early; it fires on the next iteration (up to `GENERATION_INTERVAL_SECONDS` late), which is acceptable

### Safety net (R2, R3)

- [ ] Given a cron task definition where an instance (pending, running, or completed) already exists with `scheduled_for` matching the current cron match time, when the generator evaluates it, then no new instance is created and a log message is emitted indicating the duplicate was suppressed
- [ ] Given a cron task definition where the most recent instance completed before the current cron period, when the generator evaluates it and the current period has passed, then a new instance is created normally
- [ ] Given a cron task definition where a `failed` instance exists for the current cron period, when the generator evaluates it, then a new instance is created (retry within the same period is allowed)

### Catch-up on restart (R4)

- [ ] Given the system restarts after being down for multiple cron periods, when the generator runs, then at most one catch-up instance is created per definition (using `last_fired_at` to determine the next missed period)
- [ ] Given the system restarts within the same cron period that last fired, when the generator runs, then no duplicate instance is created for that period (safety net catches it via `scheduled_for` match)

### One-shot preservation (R5)

- [ ] Given an enabled one-shot task definition with `last_fired_at is None` and the scheduled time has passed, then a single instance is created and the definition is disabled
- [ ] Given a one-shot definition with `last_fired_at` already set, then no instance is created regardless of other conditions

### Schedule accuracy (R6)

- [ ] Given a cron task with expression `*/5 * * * *` (every 5 minutes), when the generator runs over a 30-minute period, then at most 6 instances are created (one per 5-minute period, exact count depends on generator timing alignment)
- [ ] Given a cron task with expression `0 9 * * *` (daily at 9am), when the generator runs throughout the day, then exactly one instance is created at or shortly after 9:00
