# Delta Spec: DLT-023 - Bootstrap agent workspace on first run

## Story

As the system, I need a formal workspace initialization process that runs on every launch, so that modules can register idempotent setup hooks and the agent always operates from a predictable workspace directory.

## What It Does

On every launch, the bootstrap runs a sequence of explicitly registered initialization hooks in registration order. Each hook self-determines whether it needs to act (e.g., create directories, initialize git, prompt for config), making the process idempotent and resilient to partial failures. The workspace root and an internal `.tachikoma/` data folder are created if missing (via a standard hook, not special-cased logic). The agent's working directory is set to the workspace path regardless of where the process was launched. Hooks can prompt the user for terminal input during first-time setup.

## Acceptance Criteria

### Workspace creation (R0, R1)
- [ ] Given no workspace directory exists, when the app starts, then the workspace root and `.tachikoma/` data folder are created
- [ ] Given the workspace already exists, when the app starts, then no directories are re-created and no errors occur
- [ ] Given the workspace path's parent directory is not writable, when the app starts, then it exits with a clear error indicating the workspace could not be created
- [ ] Given the workspace path resolves to an existing regular file (not a directory), when the app starts, then it exits with a clear error

### Hook execution (R2, R5, R7)
- [ ] Given hooks are registered with explicit ordering, when the bootstrap runs, then hooks execute in registration order
- [ ] Given the bootstrap runs on every launch, when all hooks detect they're already initialized, then startup completes quickly with no side effects
- [ ] Given a hook is invoked, when it inspects its context, then it has access to the full settings object and the `.tachikoma/` data folder path
- [ ] Given zero hooks are registered, when the bootstrap runs, then it completes successfully (no-op)

### Hook idempotency (R3)
- [ ] Given a previous launch was interrupted mid-initialization, when the app is restarted, then hooks that already completed skip themselves and remaining hooks run normally

### User input and settings persistence (R4, R4.1)
- [ ] Given a hook needs user input (e.g., workspace path not configured), when the hook runs, then it prompts the user via terminal and uses their response
- [ ] Given a hook updates a setting value (e.g., user provides a custom workspace path), when the value is saved, then the configuration file is updated and the change persists across restarts
- [ ] Given a hook updates a setting during bootstrap, when subsequent hooks run, then they see the updated setting value

### Failure handling (R6)
- [ ] Given a hook raises an error, when the bootstrap is running, then startup aborts immediately with a clear error message naming the failing hook
- [ ] Given a hook failed on previous launch, when the app is restarted, then the failed hook re-runs (due to R3 idempotency) and previously successful hooks skip themselves

### Working directory (R8)
- [ ] Given the user launches the app from any directory, when the agent starts, then its working directory is set to the workspace path
- [ ] Given the workspace path is configured to a custom location, when the agent starts, then the working directory matches that custom location

## Notes

This spec evolves the DELTAS.md description: the original says "subsequent launches detect the existing workspace and skip initialization," but through spec collaboration we adopted the "always run hooks, each hook self-skips" model (R7). This is more resilient to partial failures and avoids global initialization state tracking. DELTAS.md should be updated during reconciliation.
