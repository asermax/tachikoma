# Delta Spec: DLT-034 - Summarize agent actions instead of generic tool markers

## Story

As a user, I want to see a concise summary of what the agent did instead of a generic "Ran tools" marker so that I can understand what happened during tool-heavy responses without reading tool-by-tool output.

## What It Does

Replaces the static "_🔧 Ran tools_" text that appears at each tool→text transition in Telegram messages with a dynamically generated summary of the tools that ran in that segment (e.g., "🔧 Read 3 files and searched for 'logging config'"). The summary is produced by a deterministic, rule-based generator that takes the collected `ToolActivity` events and formats them into a compact, human-readable description.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Replace generic "Ran tools" marker with a concise summary of actions performed | Core goal |
| R1 | Summarize from the sequence of ToolActivity events between two text boundaries | Must-have |
| R2 | Summary is human-readable and compact — single line, first word capitalized, verb-phrase items joined by commas with "and" before the last (2 items: "A and B"; 3+: "A, B, and C"), rendered as `_🔧 <summary>_` (italic with emoji prefix) matching the current visual style | Must-have |
| R3 | Handle single-tool and multi-tool sequences | Must-have |
| R4 | Support all current tool types (Read, Grep, Glob, Bash, ToolSearch, and others like Edit, Write) and unknown tools via a fallback to the tool name | Must-have |
| R5 | Aggregate repeated tool types adaptively (>2 of same type → count, otherwise list individually); cap total summary at 5 tool groups with a trailing "and more" for very long sequences | Must-have |
| R6 | Summary generation is channel-agnostic (reusable beyond Telegram) | Nice-to-have |
| R7 | Apply the same summarization to the REPL channel | Out |

## Acceptance Criteria

### Core summary generation (R0, R2)

- [ ] Given tool activity occurs during a response, when text resumes after tools, then the tool→text marker shows a human-readable summary instead of "🔧 Ran tools"
- [ ] Given any generated summary, when rendered in Telegram, then it is formatted as `_🔧 <summary text>_` (italic with emoji prefix), fits on a single line, first word is capitalized, and uses verb-phrase items joined by commas with "and" before the last (no terminal period)

### Event collection and boundary handling (R1, R3)

- [ ] Given a single tool runs before text resumes, when the summary is generated, then it describes that one tool action (e.g., "🔧 Read src/main.py")
- [ ] Given multiple tools run before text resumes, when the summary is generated, then it describes all tool actions in a combined summary (e.g., "🔧 Read 2 files and ran a command")
- [ ] Given a response with multiple tool→text transitions, when each transition occurs, then each generates its own independent summary from only the tools in that segment
- [ ] Given tools run but no text follows (response ends with tools), when the response is finalized, then the last tool activity line is cleared without a summary marker (current behavior preserved)
- [ ] Given the summary generator receives an empty tool activity list, when invoked, then it returns an empty string and no marker is inserted

### Tool type support (R4)

- [ ] Given Read tools run, when the summary includes them, then it shows file path details (e.g., "Read src/main.py" or "Read 3 files")
- [ ] Given Grep tools run, when the summary includes them, then it shows the search pattern (e.g., "searched for 'pattern'")
- [ ] Given Glob tools run, when the summary includes them, then it shows the glob pattern (e.g., "globbed '\*\*/\*.py'")
- [ ] Given Bash tools run, when the summary includes them, then it prefers the `description` field from `tool_input` if present (e.g., "ran tests"), falling back to a prefix-truncated `command` (first 40 chars + "..." if longer)
- [ ] Given an unknown tool runs (not in the known tool set), when the summary includes it, then it falls back to the tool name (e.g., "used UnknownTool")
- [ ] Given a tool activity has missing or malformed input (e.g., Read without `file_path`), when the summary includes it, then it falls back gracefully using default placeholders (e.g., "Read a file")

### Adaptive aggregation (R5)

- [ ] Given 3 or more tools of the same type run, when the summary is generated, then they are aggregated by count (e.g., "Read 3 files")
- [ ] Given 2 or fewer tools of the same type run, when the summary is generated, then each is listed with its details (e.g., "Read a.py and read b.py")
- [ ] Given a mix of tool types (e.g., 3 Reads + 1 Grep), when the summary is generated, then aggregated and individual items are joined by commas with "and" before the last (e.g., "Read 3 files and searched for 'pattern'" for 2 groups; "Read 3 files, searched for 'pattern', and ran a command" for 3+)
- [ ] Given more than 5 distinct tool groups in a single segment, when the summary is generated, then it shows the first 5 groups and appends "and more" to remain concise

### Channel-agnostic placement (R6)

- [ ] Given the summary generation function exists, when imported from the display module, then it is usable by any channel without Telegram-specific dependencies
