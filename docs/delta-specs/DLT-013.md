# Delta Spec: DLT-013 - Add structured logging for agent actions

## Story

As a developer, I need structured, machine-parseable logging for key agent actions so that I can debug issues and understand production behavior.

## What It Does

Instruments the assistant with structured logging via loguru so that key agent actions — startup, message processing, coordinator lifecycle, and errors — are recorded in a consistent format. Log entries include timestamps, log level, component context, and relevant metadata via keyword arguments. Logs are written to a file in the workspace data directory, configured through a bootstrap hook that plugs into DLT-023's initialization system. Log level is configurable via the existing TOML config file.

## Acceptance Criteria

### Logging infrastructure (R0, R2)

- [ ] Given the application starts, when a component logs an event, then the log entry contains a timestamp, log level, component name, and message with keyword metadata
- [ ] Given structured logging is configured, when events are logged, then log calls use keyword arguments (e.g., `_log.info("Session created: session_id={sid}", sid=session_id)`) rather than f-strings (e.g., `_log.info(f"Session created: {session_id}")`)
- [ ] Given log entries are written to the file, when the file is read, then each line can be parsed as structured data with timestamp, level, component, message, and keyword metadata

### Current component events (R1)

- [ ] Given the application starts, when startup completes successfully, then an INFO log records the startup event with workspace path and configuration summary
- [ ] Given the application starts, when startup fails (e.g., auth error), then an ERROR log records the failure with error details before exit
- [ ] Given a user sends a message, when the coordinator receives it, then a DEBUG log records the message received event
- [ ] Given the coordinator processes a message, when the agent responds, then a DEBUG log records the response completion
- [ ] Given the coordinator connects to the agent service, when the session is created, then an INFO log records the session lifecycle event
- [ ] Given the coordinator disconnects from the agent service, when the session ends, then an INFO log records the disconnection
- [ ] Given any component encounters an error, when the error is caught, then an ERROR log records the error with `err=str(e)` and appropriate context; unexpected errors use `.exception()` (with traceback), expected errors use `.error()` (no traceback)

### Configuration (R3)

- [ ] Given a config file with `[logging] level = "DEBUG"`, when the application starts, then log output includes DEBUG-level messages
- [ ] Given a config file with no `[logging]` section, when the application starts, then log level defaults to INFO
- [ ] Given a config file with an invalid log level value (e.g., `level = "VERBOSE"` — valid values are DEBUG, INFO, WARNING, ERROR, CRITICAL), when the application starts, then it exits with a clear validation error

### File output (R4)

- [ ] Given the workspace is initialized, when logging is configured, then log entries are written to `.tachikoma/logs/tachikoma.log`
- [ ] Given the logs directory does not exist, when the logging hook runs, then the directory is created
- [ ] Given the logs directory cannot be created (permissions), when the logging hook runs, then a clear error is raised naming the logging hook

### Bootstrap hook (R5)

- [ ] Given the workspace has been initialized, when the logging hook executes during bootstrap, then loguru is configured with the file handler pointing to the workspace data path
- [ ] Given the logging hook has run, when subsequent components log events, then entries appear in the log file
- [ ] Given the logging hook runs on subsequent launches, when logging is already configured, then the hook reconfigures loguru idempotently (no duplicate handlers)

### Log rotation (R7)

- [ ] Given the log file reaches the rotation threshold (default: 100 MB), when a new log entry is written, then the file is rotated
- [ ] Given rotated log files exist beyond the retention period (default: 7 days), when rotation occurs, then old files are cleaned up

### Third-party log interception (R8)

- [ ] Given third-party libraries use stdlib logging, when they emit log messages, then those messages are redirected through loguru and appear in the log file

### Development stderr output (R9)

- [ ] Given a development configuration (e.g., `[logging] console = true`), when events are logged, then human-readable colored output appears on stderr in addition to the file

## Requires

Dependencies:
- DLT-012: Logging settings are part of the TOML configuration system
- DLT-023: Bootstrap hook system provides workspace initialization lifecycle; workspace data path provides log file location

## Notes

- **Pre-bootstrap logging**: `config.py`'s `print()` calls happen during config loading — before the logging bootstrap hook runs. These are error paths (config parse failures, validation errors) where the application exits immediately. They should remain as stderr prints since logging isn't available yet.
- **Loguru default handler**: Before the logging hook runs, loguru's default stderr handler is active. Early bootstrap logs (workspace hook, etc.) will appear on stderr. The logging hook replaces this with the file handler.
- **Performance**: Loguru is inherently lightweight (2-4x faster than stdlib per ADR-006). The implementation should use lazy evaluation via keyword args (not f-strings) and avoid expensive serialization in log calls to maintain this.
- **UTF-8 encoding**: Log files must handle Unicode content (user messages may contain non-ASCII characters). Loguru handles UTF-8 by default.
- **Log file unwritable mid-operation**: If the log file becomes unwritable after startup (disk full, permissions changed), loguru's default behavior applies. This edge case does not need special handling in v1.
- **Workspace path changes**: If a user changes `workspace.path` in config between launches, old logs remain at the old path. This is expected — no migration needed.
- **Concurrent access**: Loguru is thread-safe. When future deltas add async concurrent flows (DLT-010 background tasks), simultaneous log writes will be handled correctly.
- **Future extensibility**: When new components are added (DLT-002 Telegram, DLT-003 delegation, DLT-008 memory), each delta adds its own log events following DES-002 conventions. No changes to the logging infrastructure delta are needed.
