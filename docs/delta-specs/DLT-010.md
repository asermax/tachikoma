# Delta Spec: DLT-010 - Queue and process background tasks during idle time

## Story

As a user, I want Tachikoma to create, schedule, and execute tasks on my behalf so that it can proactively remind me, process information, and take initiative without me having to manually trigger every action.

## What It Does

Adds a full task lifecycle to Tachikoma: persistent cron-like task definitions, automatic instance generation, and two execution modes. **Session tasks** inject messages into the main conversation during idle periods (e.g., "ask the user how their day is going") — these are proactive messages that the agent initiates when the user hasn't interacted recently. **Background tasks** run in isolated parallel sessions without interrupting the user, with an evaluator loop that monitors completion and can notify the user of results via transient session tasks. The agent manages task definitions through MCP tools exposed during conversations.

## Acceptance Criteria

### Task Definition Management (R1, R8, R10, R16)

- [ ] Given the agent is in a conversation, when it calls the `create_task` tool with a valid cron expression (e.g., `"0 9 * * *"`), type, prompt, and name, then a task definition is persisted in the database with `enabled=true` and `last_fired_at=null`
- [ ] Given the agent calls `create_task` with a one-shot schedule (e.g., `{"type": "once", "at": "2026-03-22T10:00:00Z"}`), then a task definition is created that will fire exactly once
- [ ] Given a one-shot task definition has fired and its instance has been generated, then the definition is automatically set to `enabled=false`
- [ ] Given the agent calls `list_tasks`, then it receives all task definitions with their current enabled status, schedule, and last_fired_at information
- [ ] Given the agent calls `update_task` with a modified schedule or prompt, then the definition is updated and future instances use the new configuration
- [ ] Given the agent calls `delete_task`, then the definition is removed and no further instances are generated
- [ ] Given the agent calls `create_task` with an invalid cron expression, then the tool returns a clear error message without creating a definition
- [ ] Given the agent calls `create_task` without a required field (name, schedule, type, or prompt), then the tool returns a clear error identifying the missing field
- [ ] Given the agent calls `create_task` with a type value other than "session" or "background", then the tool returns a clear error

### Task Instance Generation (R2, R11, R12, R14)

- [ ] Given a cron-based task definition with `enabled=true`, when the cron expression matches the current time, then a new task instance with status `pending` is created and the definition's `last_fired_at` is updated
- [ ] Given a one-shot task definition with `enabled=true`, when the scheduled datetime has passed, then a single task instance is created and the definition is disabled
- [ ] Given a task definition with `enabled=false`, then no instances are generated regardless of schedule
- [ ] Given the instance generator is running, then it continuously evaluates all enabled definitions and creates instances at the appropriate times
- [ ] Given a definition already has a pending or running instance, then the instance generator does not create a duplicate instance for the same definition
- [ ] Given the system restarts, then the instance generator resumes and creates at most one catch-up instance per definition that was missed during downtime (using `last_fired_at` to determine what was missed)
- [ ] Given cron expressions are evaluated, then they use the user's configured timezone (via `cronsim` + stdlib `zoneinfo`)

### Session Task Execution (R3, R4, R9)

- [ ] Given a pending session task instance exists and the time since the last coordinator message exchange exceeds the idle window (default 5 min), then the task message is injected into the coordinator
- [ ] Given a pending session task instance exists but the last message exchange was less than the idle window ago, then the instance is skipped for this check cycle and retried at the next tick
- [ ] Given a session task message is injected, then it goes through the full pre-processing pipeline (memory context injection, etc.)
- [ ] Given a session task message is injected and the boundary detector classifies it as a topic change, then a new session is created following normal boundary detection behavior
- [ ] Given the coordinator processes a session task message, then the agent responds via the active channel (Telegram/REPL) as if it were a normal conversation turn
- [ ] Given a session task message is being processed and the user sends a message simultaneously, then the user's message is handled safely without corrupting state (exact behavior is a design-phase decision — options include queuing the user message or treating it as a steer)
- [ ] Given no pending session task instances exist, then the periodic check completes without side effects
- [ ] Given `last_message_time` tracking, then it is updated on both user messages and agent responses at the coordinator level

### Background Task Execution (R5, R6, R7, R15)

- [ ] Given a pending background task instance, then a fresh SDK session is created (not forked from the main session) with an adapted base prompt explaining the background task context
- [ ] Given a background task session starts, then the pre-processing pipeline runs (memory/context injection active)
- [ ] Given a background task session completes post-processing, then the adapted pipeline runs: episodic memory extraction (yes), facts extraction (no), preferences extraction (no), core context updates (no), git commit (yes)
- [ ] Given a background task agent produces a response, then the evaluator prompt assesses whether the task is complete based on the task definition
- [ ] Given the evaluator determines the task is not complete, then the agent receives feedback and continues working (next iteration)
- [ ] Given the evaluator determines the task is complete and the definition has a non-null `notify` field, then a transient session task instance is created with the notification content
- [ ] Given the evaluator determines the task is complete and `notify` is null, then no notification is generated
- [ ] Given the evaluator detects the agent is stuck or looping, then the task instance is marked as `failed` and a transient session task is created to inform the user of the failure
- [ ] Given the background task reaches the maximum iteration limit (default 10), then the evaluator forces completion assessment and marks the task as failed if not done
- [ ] Given multiple background task instances are pending, then they execute concurrently up to a configurable limit (default 3); excess instances remain pending until a slot opens
- [ ] Given a transient session task instance (no parent definition), then it is automatically cleaned up from the database after successful delivery

### Infrastructure (R13, R14)

- [ ] Given the application starts for the first time, then the bootstrap step creates the task definitions and task instances tables in the database
- [ ] Given the application restarts, then all previously created task definitions and pending instances are available
- [ ] Given the task scheduler configuration specifies custom values for idle window, check interval, max iterations, or max concurrent background tasks, then those values are used instead of defaults
- [ ] Given the application shuts down gracefully, then running background task instances are marked as `failed` with a shutdown reason

### Flow Description

**Entry point**: User asks the agent to do something on a schedule during a normal conversation (e.g., "remind me to check emails every morning at 9", "summarize my notes every evening"). The agent interprets the request and calls the `create_task` MCP tool. The agent may also proactively create tasks based on conversation context (e.g., "I'll follow up on this topic in a few hours").

**Happy path — Session task**: The instance generator creates a task instance when the cron schedule fires. The periodic session task check (every ~5 minutes) finds the pending instance, verifies the session is idle (last message exchange > 5 min ago), and injects the task prompt into the coordinator. The agent responds naturally via Telegram (e.g., "Good morning! Time to check your emails"). The message goes through pre-processing (gets relevant memories) and may trigger a session boundary if the topic changed.

**Happy path — Background task**: The instance generator creates a task instance. The background task runner picks it up (if under the concurrency limit), creates a fresh isolated session with an adapted prompt and pipeline, and the agent works on the task. The evaluator checks each response — if done and `notify` is set, a transient session task is queued. This transient task is delivered at the next session task check during an idle period.

**Decision points**: Session task idle gating (busy → skip, idle → inject). Background task evaluator (done → complete, not done → continue, stuck → fail). Notification gating (notify field null → silent, non-null → queue transient task). Concurrency gating (under limit → run, at limit → wait).

**Exit points**: Session tasks end when the agent's response is delivered. Background tasks end when the evaluator marks them complete or failed. Failed tasks always generate a notification to the user.

## Detected Impacts

### Affected Features
- **agent domain** — Modifies: Coordinator needs message injection mechanism for session tasks; needs `last_message_time` tracking at coordinator level
- **configuration domain** — Adds: New `[tasks]` configuration section for scheduler parameters (idle_window, check_interval, max_iterations, max_concurrent_background)
- **memory domain** — Modifies: Post-processing pipeline needs to support adapted processor selection for background task sessions (skip facts, preferences, core context processors)

### Notes for Reconciliation
- New feature domain `tasks/` needs to be created in feature-specs covering: task definitions, task instances, session task execution, background task execution, evaluator loop
- Agent domain feature spec (when created) needs to document the message injection interface and last_message_time tracking
- Configuration domain feature spec (when created) needs to include task scheduler settings
