# Delta Spec: DLT-027 - Track conversation sessions

<!-- This spec describes WHAT the delta does, not HOW it's implemented. -->

## Story

As a system, I need a persistent registry of conversation sessions so that downstream pipelines (memory extraction, post-processing) can identify which conversation to analyze and when conversations start and end.

## What It Does

Maintains a SQLite-backed registry that tracks conversation sessions the agent has with the user. Each session records when it started, when it ended, the SDK session ID, and the path to the SDK-managed transcript file. The registry provides the lifecycle foundation that boundary detectors (DLT-026, DLT-004) use to signal session transitions, and that post-processing pipelines (DLT-008) use to find completed conversations for analysis. On startup, a bootstrap hook detects sessions left open from ungraceful shutdowns and closes them with best-effort timestamps.

> **Design note**: VISION.md states "File-based — no database for v1." SQLite is a single embedded file with no server process, satisfying the spirit of that constraint while providing the query capabilities (time range lookups, ID filtering) that flat-file formats cannot offer efficiently. VISION.md should be updated to reflect this clarification during reconciliation.

## Acceptance Criteria

### Session Creation (R1, R2)

- [ ] Given the agent receives the first message in a new conversation, when no active session exists, then a new session is created with a unique ID and the current timestamp as `started_at`
- [ ] Given a boundary detector (DLT-026) signals a new conversation, when the current session is closed, then a new session is created for the incoming message
- [ ] Given a session is created, then its `ended_at` is null, `sdk_session_id` is null, and `transcript_path` is null until the conversation produces a Result event
- [ ] Given a session creation is already in progress, when another creation signal arrives, then only one session is created (the operation is serialized)

### Session Metadata Update (R1)

- [ ] Given an active session, when the coordinator produces a Result event with `session_id`, then the session's `sdk_session_id` and `transcript_path` are populated from the SDK data
- [ ] The `transcript_path` is derived from the SDK session ID using the known SDK transcript directory structure (`~/.claude/projects/<sanitized-cwd>/<session-id>.jsonl`)

### Session Closing (R3)

- [ ] Given an active session, when a conversation end signal is received (from DLT-026 or DLT-004), then the session's `ended_at` is set to the current timestamp
- [ ] Given a session is already closed (`ended_at` is set), when a close signal is received again, then the operation is idempotent (no error, no change)
- [ ] Given no active session exists, when a close signal is received, then the operation is a no-op (no error)

### Querying (R4, R5)

- [ ] Given sessions exist in the registry, when querying by time range, then all sessions whose time span (`started_at` to `ended_at`) overlaps with the query range are returned ordered by `started_at` descending (open sessions with null `ended_at` are treated as ongoing and included if their `started_at` is before the range end)
- [ ] Given a session ID, when querying by ID, then the matching session is returned or None if not found
- [ ] Given no sessions match a query, then an empty result is returned (not an error)

### Persistence (R6, R7)

- [ ] Given the registry is stored as a SQLite database at `settings.workspace.data_path / "sessions.db"`, when the application restarts, then all previously recorded sessions are still queryable
- [ ] Given the database does not exist when the registry is first accessed, then it is created with the correct schema
- [ ] Given schema creation fails due to a filesystem error (permissions, disk full), then a clear error is raised

### Crash Recovery (R8)

- [ ] Given the application starts and the registry contains sessions with `ended_at` null, when the bootstrap recovery hook runs, then those sessions have their `ended_at` set to a best-effort timestamp (last modified time of transcript file if available, otherwise current time)
- [ ] Given a session has `ended_at` set and `sdk_session_id` is null, when its status is queried, then it is identified as "interrupted" — indicating the conversation ended without producing a complete SDK result (either due to crash recovery or a close signal before the first response)
- [ ] Given the recovery hook runs and no sessions have null `ended_at`, then the hook completes with no side effects (idempotent per DLT-023 R3)
