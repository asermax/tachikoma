# Delta Spec: DLT-091 — Replace task notify field with agent-driven notification tool

## Story

As a user, I want background tasks to decide at runtime whether and what to notify me about so that I receive meaningful, context-aware notifications instead of static all-or-nothing messages.

## What It Does

Replaces the static `notify` field on task definitions with a `send_notification` MCP tool that background task agents call during execution. Instead of deciding notification behavior at task creation time, the agent evaluates its results at runtime and sends notifications when meaningful — conditionally, multiple times for progress, or not at all. Failure notifications remain automatic since the agent can no longer act after a failure.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Background task agents can send user notifications on demand during execution via an MCP tool | Core goal |
| R1 | Remove the `notify` field from task definitions, MCP tool parameters (create/update/list), model, and database schema | Must-have |
| R2 | Remove the executor's automatic success notification dispatch (session fork + text generation + event dispatch) | Must-have |
| R3 | Failure notifications remain automatic — executor dispatches them when the agent can no longer act | Must-have |
| R4 | `send_notification` tool accepts a raw message string and delivers it to the user through the existing `TaskNotification` event pipeline; available only during background task execution | Must-have |
| R5 | Notification prompt wrapping — both agent-driven and error notifications use a consistent template that identifies the source task, dispatch timestamp, and frames the content as a notification (not a user message) | Must-have |
| R6 | The preamble Tasks section and background task system prompt are updated to describe the new notification model and the `send_notification` tool | Must-have |
| R7 | Support multiple notifications during a single task execution (e.g., progress updates) | Nice-to-have |

## Acceptance Criteria

### Notify field removal (R1)

- [ ] Given an existing database with `notify` column on `task_definitions`, when the schema migration runs, then the column is dropped (existing `notify` data is intentionally discarded — the agent now controls notifications at runtime)
- [ ] Given the agent calls `create_task`, then the `notify` parameter is not accepted and does not appear in tool documentation
- [ ] Given the agent calls `update_task`, then the `notify` parameter is not accepted and does not appear in tool documentation
- [ ] Given the agent calls `list_tasks`, then no `notify` field appears in the output

### Success notification removal (R2)

- [ ] Given a background task completes successfully, then the executor does not fork the session or dispatch any notification event — the task completes silently unless the agent called `send_notification` during execution
- [ ] Given the `_generate_notification()` and success path of `_dispatch_notification()` are removed, then no code references `definition.notify` in the executor

### Automatic failure notifications (R3)

- [ ] Given a background task fails (stuck, error, or max iterations), then the executor dispatches a `TaskNotification` event with severity "error", regardless of whether the agent called `send_notification` during execution
- [ ] Given a failure notification is dispatched, then it uses the same prompt wrapping template as agent-driven notifications (task name, dispatch timestamp, notification framing)
- [ ] Given the agent called `send_notification` earlier in execution and then the task fails, then both the agent's notification(s) and the automatic failure notification are delivered independently

### Agent-driven notification tool (R0, R4)

- [ ] Given a background task agent is executing, when it calls `send_notification` with a message string, then a `TaskNotification` event with the wrapped prompt and severity "info" is dispatched on the event bus
- [ ] Given the `TaskNotification` event is dispatched, then channels receive it and route the prompt through the coordinator for delivery to the user (existing pipeline, unchanged)
- [ ] Given the `send_notification` tool is called with an empty message, then the tool returns an error without dispatching a notification
- [ ] Given the agent is in a normal conversation (not a background task execution), then the `send_notification` tool is not available in the MCP tool listing

### Notification prompt wrapping (R5)

- [ ] Given any notification (agent-driven or failure), then the prompt delivered to the coordinator includes: identification that this is a background task notification (not a user message), the task name, the dispatch timestamp, and the notification content
- [ ] Given the main session receives a notification prompt, then it can distinguish this from a regular user message and deliver it appropriately

### Preamble and system prompt update (R6)

- [ ] Given the system prompt preamble, then the Tasks section describes `send_notification` as a tool available during background task execution for delivering messages to the user
- [ ] Given the preamble, then it no longer mentions the `notify` parameter on `create_task` or `update_task`
- [ ] Given the preamble, then it explains that failures are automatically notified and success notifications are the agent's responsibility via `send_notification`
- [ ] Given the background task system prompt (`BACKGROUND_TASK_SYSTEM_PROMPT`), then it guides the agent to use `send_notification` for communicating results to the user

### Multiple notifications (R7)

- [ ] Given a long-running background task, when the agent calls `send_notification` multiple times during execution, then each notification is dispatched and delivered independently
- [ ] Given a notification dispatch fails (e.g., bus error), then the task execution continues unaffected
