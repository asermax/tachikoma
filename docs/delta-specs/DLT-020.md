# Delta Spec: DLT-020 - Git module for workspace version tracking

## Story

As the system, I need automatic git version tracking for all workspace file changes so that every modification to memories, context files, and configuration is recorded with descriptive commits after each session, providing built-in history and rollback capability.

## What It Does

A git post-processor that plugs into DLT-008's post-processing pipeline and runs in a finalization phase — after all other processors (memory extraction, etc.) have completed. It spawns a Haiku agent via a fresh `query()` call (no session fork needed) that inspects uncommitted workspace changes using bash git commands, groups them into cohesive sets (e.g., episodic memories in one commit, facts in another, project config in a third), and creates one descriptive commit per group. A separate bootstrap hook initializes the workspace as a git repo on first run and configures a fixed committer identity via repo-local git config.

## Acceptance Criteria

### Post-processor pipeline integration (R1, R1.1)

- [ ] Given the post-processing pipeline runs after session close, when all main-phase processors complete (success or failure), then the finalize-phase processors run
- [ ] Given the pipeline supports phased execution, when processors are registered, then each declares its phase (`main` or `finalize`) and the pipeline runs phases sequentially
- [ ] Given a processor declares an unknown phase at registration time, then registration fails with a clear error
- [ ] Given multiple processors register for the finalize phase, when the finalize phase runs, then they execute in parallel (same behavior as main-phase parallelism)
- [ ] Given no other post-processors are registered, when the pipeline runs, then the git post-processor still runs in its finalization phase (empty main phase is a no-op)
- [ ] Given a main-phase processor fails, when the finalize phase begins, then the git post-processor still runs (error isolation per DLT-008 R1.3 applies across phases)

### Haiku agent spawning and guardrails (R2, R2.1)

- [ ] Given uncommitted changes exist in the workspace, when the git post-processor runs, then it spawns a Haiku agent via a fresh `query()` call (no `resume`, no `fork_session`) with a prompt describing the commit task
- [ ] Given the agent is spawned, when it executes, then it operates from the workspace directory and has permission to run bash commands
- [ ] Given the agent prompt, then it explicitly instructs the agent to only use `git status`, `git diff`, `git add`, and `git commit` — not destructive commands (`git push`, `git branch`, `git reset`, `git checkout`, etc.)
- [ ] Given the agent is spawned, when the `query()` call fails (e.g., network error), then the post-processor raises an exception caught by the pipeline (per DLT-008 R1.3)

### Cohesive commit grouping (R2.2, R2.3)

- [ ] Given changes span multiple workspace subdirectories (e.g., `memories/episodic/` and `memories/facts/`), when the agent analyzes the diff, then it creates separate commits per cohesive group
- [ ] Given all changes belong to a single subdirectory or purpose, when the agent analyzes the diff, then it creates a single commit
- [ ] Given the agent creates commits, when each commit is written, then the commit message is descriptive and reflects the content of that group (e.g., "Update episodic memories for 2026-03-13")
- [ ] Given the agent successfully commits some groups but fails mid-way (e.g., agent error after committing 1 of 3 groups), then the partial commits remain as valid history and uncommitted changes are picked up on the next post-processing run
- [ ] Given the agent completes, when the post-processor resumes, then it checks `git status` and logs a warning if uncommitted changes remain

### No-op when clean (R4)

- [ ] Given no modified tracked files and no untracked files in the workspace, when the git post-processor runs, then it completes successfully without spawning an agent

### Repo initialization (R3, R3.1, R3.2, R6)

- [ ] Given no `.git` directory in the workspace, when the git bootstrap hook runs, then a git repo is initialized with an initial empty commit
- [ ] Given a fresh git init, when the hook completes, then the repo has repo-local `user.name` and `user.email` configured with a fixed identity (e.g., "Tachikoma" / "tachikoma@local")
- [ ] Given a fresh git init, when the hook completes, then no `.gitignore` file is created (users may add their own later)
- [ ] Given an existing `.git` directory with commits, when the hook runs, then it skips initialization (idempotent)
- [ ] Given the workspace directory exists but `.git` was deleted, when the hook runs, then it re-initializes the repo
- [ ] Given no global git config exists on the system, when the hook initializes the repo, then commits succeed using the repo-local identity
- [ ] Given git init fails (e.g., permission denied), when the hook runs, then a clear exception propagates with the failure reason
- [ ] Given the bootstrap system, when hooks are registered, then the git hook is registered after the workspace hook

### Linear history (R5)

- [ ] Given multiple sequential post-processing runs, then all commits are on the same default branch (linear history, no branching)
- [ ] Given the agent creates commits, then it does not create or switch branches

## Notes

- R1.1 (phased pipeline execution) is a new capability that extends DLT-008's pipeline. The pipeline currently runs all processors in parallel; phased execution adds sequential phases so finalization tasks like git commits can see all prior changes. This extension is implemented as part of DLT-020's work.
- Using Haiku keeps the cost low for what is essentially a housekeeping task. The agent's job is mechanical (inspect diff, group, commit) and doesn't need the reasoning depth of a larger model. The specific model identifier is a design decision.
- No gitpython dependency — bash git commands are sufficient and the agent can use them directly without a Python wrapper.
- Agent guardrails (destructive command prevention) are enforced via prompt instructions, not SDK-level tool restrictions. This is consistent with the DLT-008 memory processors which also rely on prompt-level constraints for file scope.
- Partial commits are by design — if the agent commits 1 of 3 groups and then fails, those commits are valid history. The remaining uncommitted changes will be picked up by the next post-processing run. This makes failure handling simple and avoids needing rollback logic.
- Cohesive grouping dimensions: the prompt should instruct the agent to group by workspace subdirectory/purpose. Primary grouping expected: each `memories/` subdirectory as a group, core context files as a group, other workspace files as a group. The exact prompt wording is a design concern — effectiveness may require iteration.
