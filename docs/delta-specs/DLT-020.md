# Delta Spec: DLT-020 - Git module for workspace version tracking

## Story

As the system, I need automatic git version tracking for all workspace file changes so that every modification to memories, context files, and skills is recorded with a descriptive commit, providing built-in history and rollback capability.

## What It Does

A git module that provides two things: (1) a bootstrap hook that initializes the workspace as a git repository on first run and validates it on subsequent runs, re-initializing if needed; and (2) a commit utility that other modules call after writing workspace files, passing specific file paths to stage and commit with descriptive messages. Uses gitpython for clean API and typed error handling. The bootstrap hook follows the standard hook contract (raises on failure). The commit utility returns a structured result object so both code and LLMs can interpret the outcome.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Git-backed version tracking for all workspace file changes | Core goal |
| R1 | Bootstrap hook that initializes workspace as a git repo on first run (git init + initial empty commit) | Must-have |
| R2 | Startup validation — re-initialize if workspace exists but has no .git; idempotent | Must-have |
| R3 | Commit utility that receives specific file paths from callers and creates commits with descriptive messages | Must-have |
| R3.1 | Commit utility returns a structured result (success/failure) rather than raising exceptions | Must-have |
| R4 | Linear history on a single branch — no branch operations are exposed | Must-have |
| R5 | Bootstrap hook errors propagate as exceptions (follows DLT-023 hook contract) | Must-have |
| R6 | Use gitpython library for git operations | Must-have |
| R7 | Automatic commits use a fixed identity (no dependency on global git config) | Must-have |

## Acceptance Criteria

### Repo initialization (R1, R2, R7)
- [ ] Given no .git directory in workspace, when the git bootstrap hook runs, then a git repo is initialized and an initial empty commit is created
- [ ] Given a fresh git init, when the hook completes, then the repo has one commit with HEAD pointing to it
- [ ] Given an existing .git directory with commits, when the git bootstrap hook runs, then it skips initialization (idempotent)
- [ ] Given the workspace dir was created but .git was deleted, when the git hook runs, then it re-initializes the repo
- [ ] Given no global git config exists on the system, when the hook initializes the repo, then commits use a fixed identity (e.g., "Tachikoma <tachikoma@local>") and succeed without error
- [ ] Given git init fails (e.g., permission denied on workspace directory), when the hook runs, then a clear exception propagates with the failure reason (R5)

### Commit utility (R3, R3.1, R4)
- [ ] Given a list of file paths with changes, when the commit utility is called, then those files are staged and committed with the provided message, and a success result is returned
- [ ] Given file paths that have no changes, when the commit utility is called, then it returns a result indicating nothing was committed (no exception)
- [ ] Given file paths that include a non-existent file, when the commit utility is called, then a failure result is returned indicating the file does not exist
- [ ] Given file paths that resolve outside the workspace directory, when the commit utility is called, then a failure result is returned
- [ ] Given a descriptive message, when a commit is created, then the commit message matches what was provided
- [ ] Given multiple sequential calls to the commit utility, then all commits are on the same branch (linear history)

### Integration (R1)
- [ ] Given the bootstrap system, when hooks are registered, then the git hook is registered after the workspace hook (depends on workspace directory existing)
