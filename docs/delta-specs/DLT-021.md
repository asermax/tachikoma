# Delta Spec: DLT-021 - Skill Detection and Context Injection

## Story

As the assistant, I want relevant skills automatically detected and injected into my context when a conversation starts, so that I have specialized knowledge and agents available only when they're actually needed — without wasting context on irrelevant skills.

## What It Does

Builds a skills context provider that plugs into the pre-processing pipeline. On a new session's first message, it queries the skill registry for all available skills, classifies which are relevant to the user's message, then injects the matched skills' instruction text and directory paths as an XML context block. The matched skills' agent definitions are also loaded into the SDK client for delegation. The pre-processing pipeline interface is extended to support returning structured data (agent definitions) alongside text context.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Detect relevant skills on new session start and inject their content + agents into the session | Core goal |
| R1 | Implement as a `ContextProvider` registered in the pre-processing pipeline | Must-have |
| R2 | Detection via LLM: load all skill names + descriptions into a forked Opus agent (low effort), LLM decides which are relevant to the message | Must-have |
| R3 | Inject matched skill content: SKILL.md body (no frontmatter) + skill directory path, as `<skills>` XML context block | Must-have |
| R4 | Load only detected skills' agent definitions into the SDK client — not all agents at startup | Must-have |
| R5 | Extend `ContextResult` with an optional extras field for structured data (e.g., agent definitions); existing providers that don't use extras continue working unchanged | Must-have |
| R6 | Detected skills persist for the session: agents remain available across subsequent messages without re-detection; on topic shift (new session), detection runs again | Must-have |
| R7 | Balance precision (don't waste context on irrelevant skills) with recall (don't miss applicable skills) | Must-have |
| R8 | Graceful error handling — detection failures never block the message; message proceeds with no skills/agents | Must-have |
| R9 | No changes to SKILL.md format or skill registry discovery mechanism | Must-have |
| R10 | When no skills exist in the registry, provider is a no-op (returns no context, no agents, no LLM call) | Must-have |
| R11 | Trigger patterns in SKILL.md | Out |
| R12 | Constrained execution (limiting what a skill's agents can do) | Out |
| R13 | Dynamic skill registry refresh at runtime (new skills require restart) | Out |

## Acceptance Criteria

### Skill Detection (R0, R1, R2)

- [ ] Given skills exist in the registry and a new session starts, when the first message arrives, then the skills context provider runs as part of the pre-processing pipeline
- [ ] Given the provider runs, when it queries the registry, then it loads all skill names and descriptions and sends them to a forked Opus agent (low effort) alongside the user message
- [ ] Given the forked agent completes classification, when it returns results, then the response is parsed as a list of skill names matching exactly the names from the registry; unrecognized names are discarded
- [ ] Given the provider runs on a subsequent message in the same session, when pre-processing is skipped (existing behavior), then the previously detected skills and agents remain active

### Detection Quality (R7)

- [ ] Given skills A (relevant to the message topic) and B (unrelated to the message topic) exist in the registry, when the detection agent classifies them, then skill A is included and skill B is not (precision)
- [ ] Given a skill exists whose description matches the user's message intent, when the detection agent classifies skills, then that skill is included in the results (recall)

### Skill Content Injection (R3)

- [ ] Given the detection agent identifies relevant skills, when the provider assembles the result, then it returns a `<skills>` XML context block containing each matched skill's SKILL.md body (no YAML frontmatter) and its directory path
- [ ] Given multiple skills are detected, when the context block is assembled, then each skill's content is clearly delineated with its name and directory path
- [ ] Given no skills are detected as relevant, when the provider completes, then it returns no text context (None) and no agent definitions

### Agent Loading (R4, R6)

- [ ] Given relevant skills are detected, when the provider returns results, then only those skills' agent definitions are included in the structured extras
- [ ] Given a new session starts and the pre-processing pipeline detects skills, when the coordinator creates the SDK client for that same first message, then the detected agents are included in the SDK client options
- [ ] Given agents are loaded for a session, when subsequent messages arrive in the same session, then the same agents are passed to the SDK client without re-detection
- [ ] Given no skills are detected, when the coordinator creates the SDK client, then no sub-agents are available for delegation
- [ ] Given a topic shift causes a new session, when the first message of the new session is processed, then skill detection runs again (it is a new session with fresh pre-processing)

### Pre-Processing Pipeline Extension (R5)

- [ ] Given `ContextResult` has an optional `extras` field, when a provider sets extras (e.g., agent definitions dict), then the pipeline collects both text context and extras separately
- [ ] Given multiple providers return results, when the pipeline assembles them, then text context is assembled as before (XML blocks prepended) and extras are collected separately for the coordinator
- [ ] Given existing providers (memory) that don't set extras, when the pipeline runs, then they continue to work unchanged — extras defaults to None (backward compatible)

### Error Handling (R8)

- [ ] Given the skills detection agent fails (SDK error, timeout), when the provider catches the error, then it logs the failure and returns None (no context, no agents)
- [ ] Given the detection agent returns an unrecognizable response (no valid skill names parseable), when the provider processes it, then it logs a warning and returns None
- [ ] Given all pre-processing providers fail, when the coordinator processes the message, then it proceeds with no context enrichment and no skill agents

### No Skills Available (R10)

- [ ] Given no skills exist in the registry, when the provider runs, then it returns immediately with no context and no agents (no LLM call made)
- [ ] Given the skills directory is empty, when the provider initializes, then it operates normally without error

### Format Preservation (R9)

- [ ] Given a SKILL.md file with the current frontmatter format (name, description, version), when the skills context provider reads it, then no additional frontmatter fields are required or consumed beyond what the skill registry already parses
