# Delta Spec: DLT-029 - Complete pending signals lifecycle with removal and auto-injection

## Story

As the system, I need a complete pending signals lifecycle — with auto-injection, removal, and updated prompt guidance — so that promoted or irrelevant signals are cleaned up immediately rather than accumulating until the 30-day expiry.

## What It Does

The context update post-processor currently provides two MCP tools for pending signals: `read_pending_signals` and `add_pending_signal`. When the forked agent promotes a recurring signal to a context file update, the original pending signal has no way to be removed — it lingers until it ages out after 30 days. This delta completes the lifecycle by: injecting current pending signals directly into the prompt (numbered S1..Sn) so the agent sees them without an extra tool call, replacing the read tool with this auto-injection, adding a `remove_pending_signal` tool that removes one or more signals by their injected index, and updating prompt instructions and tool descriptions to guide the agent through the full stage → promote+remove → cleanup lifecycle.

> **Note:** The DELTAS.md description says "by matching its text content" for the removal mechanism. During the spec interview, index-based removal was chosen instead (more reliable, no ambiguity). DELTAS.md will be updated during reconciliation.

## Acceptance Criteria

### Lifecycle Completion (R0)

- [ ] Given a pending signal has been promoted to a context file update, when the agent processes the promotion, then the agent calls `remove_pending_signal` with the promoted signal's index to remove it from `.tachikoma/pending-signals.md` in the same processor run
- [ ] Given the agent determines a pending signal is stale or irrelevant, when it decides to clean it up, then the signal is removed from the file via the remove tool
- [ ] Given a recurring signal detected by the agent, when it promotes the signal to a context file update, then the agent both updates the appropriate context file AND removes the promoted signal — promotion and removal happen in the same forked session

### Auto-Injection (R1)

- [ ] Given the pending signals file contains entries, when the processor builds the prompt before forking, then the entries are embedded directly in the prompt text as a numbered list (e.g., `S1: **2026-03-10**: User seemed to prefer shorter responses`)
- [ ] Given the pending signals file is empty or does not exist, when the processor builds the prompt, then the pending signals section of the prompt indicates no signals are present
- [ ] Given the pending signals file has entries, when the forked agent starts, then it can see all existing signals immediately in its prompt context without making any tool call

### Read Tool Removal (R2)

- [ ] Given the forked agent's available MCP tools, then `read_pending_signals` is not among them
- [ ] Given the `create_pending_signals_server()` factory, then it no longer registers a `read_pending_signals` tool

### Remove Tool — Single Removal (R3, R3.1)

- [ ] Given the prompt contains signals numbered S1, S2, S3, when the agent calls `remove_pending_signal` with index 2, then the signal that was labeled S2 in the prompt is removed from the file
- [ ] Given the agent provides an index that does not exist (e.g., 5 when only 3 signals were injected), when `remove_pending_signal` is called, then it returns an error indicating the index is invalid
- [ ] Given the agent removes S2 and then wants to remove S3, when it calls `remove_pending_signal` with index 3, then S3 is correctly removed — indices refer to the original numbering from the prompt, not shifted positions after earlier removals

> **Note:** The index-to-signal mapping is established at prompt build time (pre-fork) and remains stable for the entire duration of the forked session. The remove tool uses this snapshot to resolve indices, ensuring stability across multiple sequential removals.

### Remove Tool — Multiple Removal (R3, R3.2)

- [ ] Given the prompt contains signals S1 through S5, when the agent calls `remove_pending_signal` with indices [1, 3, 5], then all three signals are removed in a single call
- [ ] Given a batch removal request includes both valid and invalid indices, when `remove_pending_signal` is called, then it returns an error indicating which indices are invalid and does not remove any signals (atomic — all-or-nothing to prevent partial state)
- [ ] Given the agent calls `remove_pending_signal` with an empty indices list, when the call is processed, then it returns success as a no-op

### Prompt Instructions (R4)

- [ ] Given the updated prompt, when the agent processes a conversation with a recurring signal, then the instructions guide it to: update the context file AND remove the promoted signal via the remove tool
- [ ] Given the updated prompt, when the agent encounters a stale or irrelevant signal in the injected list, then the instructions guide it to remove it via the remove tool
- [ ] Given the updated prompt, when the agent encounters a new ambiguous signal, then the instructions guide it to stage it via `add_pending_signal` (existing behavior preserved)

### Tool Descriptions (R5)

- [ ] Given the `add_pending_signal` tool description, then it clearly explains its role as the staging step of the lifecycle
- [ ] Given the `remove_pending_signal` tool description, then it clearly explains its role as the cleanup/promotion step and that indices correspond to the numbered signals in the prompt

### Edge Cases

- [ ] Given all pending signals have been removed via the remove tool during a session, when the last removal completes, then the pending signals file is either deleted or left in a clean empty state (consistent with existing `clean_pending_signals` behavior of deleting empty files)
- [ ] Given the pending signals file is modified externally between prompt build and a `remove_pending_signal` call, when the tool executes, then it operates based on the pre-fork snapshot — the file is rewritten from the snapshot minus removed entries (externally added entries may be lost; this is acceptable given the single-user, sequential nature of the system)
