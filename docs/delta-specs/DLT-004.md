# Delta Spec: DLT-004 - Detect conversation boundaries via inactivity timeout

## Story

As a system, I need to detect when a user has been inactive for an extended period so that the current conversation session is closed and post-processing runs, even when the user simply walks away without changing topic.

## What It Does

Adds an inactivity pre-check to the boundary detection flow. Before running the LLM-based topic analysis on each incoming message, the coordinator compares the time elapsed since the last user message against a configurable threshold (default 1200 seconds / 20 minutes). If the threshold is exceeded, the system short-circuits with a boundary result signaling a topic shift — skipping both the recent session candidate query and the LLM call, and triggering the same session close and post-processing flow as a topic-based boundary. The check logic lives in the boundary module as a utility function. This serves as a fallback for the "user walked away and came back" case that topic analysis alone cannot detect.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Detect user inactivity and signal a conversation boundary when time since last message exceeds a configurable threshold (default: 1200 seconds) | Core goal |
| R1 | Inactivity check performed by the coordinator before calling LLM-based topic analysis; check logic lives in the boundary module as a utility function | Must-have |
| R2 | Use the timestamp of the last user message as the activity source (coordinator's `last_message_time` property) | Must-have |
| R3 | Timeout threshold configurable via config file (default 1200 seconds) | Must-have |
| R4 | Post-processing runs when a session closes due to inactivity, identical to any other session close | Must-have |
| R5 | Inactivity boundary skips session resumption matching — `resume_session_id` is always None | Must-have |
| R6 | Inactivity check failures are fail-open (proceed to topic analysis) | Must-have |
| R7 | Inactivity check only applies when boundary detection would already run (active session with summary and workspace directory) | Must-have |

## Acceptance Criteria

### Inactivity Detection (R0, R1, R2, R7)

- [ ] Given an active session with a summary and the time since the last user message exceeds the inactivity threshold, when a new message arrives, then the inactivity pre-check signals a boundary without querying recent session candidates or calling the LLM topic analyzer
- [ ] Given an active session with a summary and the time since the last user message is within the inactivity threshold, when a new message arrives, then the inactivity pre-check passes and LLM-based topic analysis runs normally
- [ ] Given no active session exists (first message ever), when a message arrives, then the inactivity check is skipped entirely (boundary detection as a whole is skipped)

> **Note**: The scenario "active session with no `last_message_time`" is structurally prevented — the coordinator sets `last_message_time` at the top of every `send_message()` call, so by the second message (when boundary detection first runs) it always has a value. Without a session registry, inactivity detection is also naturally bypassed since boundary detection requires an active session from the registry.

### Session Transition (R4, R5)

- [ ] Given the inactivity pre-check signals a boundary, when the coordinator handles the result, then the current session is closed, post-processing runs asynchronously, and a new session is created — identical to topic-shift behavior
- [ ] Given the inactivity pre-check signals a boundary, when the coordinator handles the result, then session resumption matching is skipped (`resume_session_id=None`) — the user walked away and came back, there is no topic to match against previous sessions
- [ ] Given a session was closed due to inactivity, when the user sends the next message, then a new session is created transparently with pre-processing (context enrichment) running on the first message
- [ ] Given a session was closed due to inactivity, when the new session starts, then the previous conversation summary is injected into the new session's system prompt (same as topic-shift behavior)

### Configuration (R3)

- [ ] Given the config file includes a boundary inactivity timeout setting, when the application starts, then the configured value is used as the inactivity threshold
- [ ] Given the config file does not include a boundary inactivity timeout setting, when the application starts, then the default threshold of 1200 seconds is used

### Graceful Degradation (R6)

- [ ] Given the inactivity pre-check encounters an error, when the error occurs, then it is logged and the system proceeds to LLM-based topic analysis (fail-open)
