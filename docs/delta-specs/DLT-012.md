# Delta Spec: DLT-012 — Configure application parameters and secrets

## Story

As a developer deploying Tachikoma, I want all tunable parameters and secrets managed in a single configuration file so that I can customize the assistant's behavior without modifying code.

> **Note:** The delta description in DELTAS.md specifies secrets via environment variables. After discussion during spec, this was deliberately changed to consolidate everything into the TOML config file for simplicity — a single source of truth. This is acceptable for a single-user, self-hosted deployment. Security hardening can be revisited later.

## What It Does

Introduces a typed configuration system backed by a single TOML file at `~/.config/tachikoma/config.toml`. All parameters (workspace path, agent settings, secrets) live in this file with sensible defaults. On first run, a commented default config file is auto-generated so users can see what's configurable. The system validates all values at startup and provides clear error messages.

The `ANTHROPIC_API_KEY` is not managed by this system — the Claude SDK reads it natively from the environment. Only Tachikoma-specific secrets (e.g. `telegram_bot_token`) go in the config file.

The `logging_level` parameter mentioned in the delta description is deferred to DLT-013 (structured logging), which will add it to the config model when needed.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Clean separation between operational configuration and code | Core goal |
| R1 | All parameters (settings + secrets) managed via a single TOML config file at `~/.config/tachikoma/config.toml` | Must-have |
| R2 | Sensible defaults for all non-secret parameters (workspace=`~/tachikoma`, model=SDK default, etc.) | Must-have |
| R3 | Startup validation with clear error messages for missing/invalid values | Must-have |
| R4 | Auto-generate a commented default config file when none exists at startup | Must-have |
| R5 | Easy extensibility — adding new config parameters should be low-friction | Nice-to-have |

## Acceptance Criteria

**Separation of config and code (R0)**
- [ ] Given the application is running, when any tunable parameter needs to change, then it can be changed by editing the config file without modifying source code

**Configuration loading (R1)**
- [ ] Given a valid TOML config file at `~/.config/tachikoma/config.toml`, when the application starts, then all parameters are loaded and available to components
- [ ] Given the config file contains secrets (e.g. `telegram_bot_token`), when loaded, then secrets are accessible like any other parameter
- [ ] Given no environment variable overrides exist, when the config loads, then only the TOML file determines values

**Defaults (R2)**
- [ ] Given a config file with no `[workspace]` section, when loaded, then `workspace.path` defaults to `~/tachikoma`
- [ ] Given a config file with no `[agent]` section, when loaded, then `agent.model` defaults to `None` (SDK default) and `agent.allowed_tools` defaults to `["Read", "Glob", "Grep"]`
- [ ] Given a completely empty config file, when loaded, then all non-secret parameters use their defaults and the application starts successfully

**Validation (R3)**
- [ ] Given a config file with an invalid value (e.g. `workspace.path = 123`), when the application starts, then it exits with a clear error message naming the field and expected type
- [ ] Given a config file with an unknown section or key, when loaded, then the unknown keys are ignored (forward-compatible)
- [ ] Given the config file contains invalid TOML syntax, when the application starts, then it exits with a clear parse error message
- [ ] Given the config file exists but is not readable (permission denied), when the application starts, then it exits with a clear error message indicating the permission issue
- [ ] Given the config path exists but is not a regular file (e.g. a directory), when the application starts, then it exits with a clear error message

**Auto-generation (R4)**
- [ ] Given no config file exists at `~/.config/tachikoma/config.toml`, when the application starts, then a default config file is created with all parameters commented out and annotated
- [ ] Given no `~/.config/tachikoma/` directory exists, when the application starts, then the directory is created before writing the default config
- [ ] Given the config directory cannot be created (permission denied), when the application starts, then it exits with a clear error message
- [ ] Given a config file already exists, when the application starts, then it is loaded as-is (never overwritten)

**Extensibility (R5)**
- [ ] Given a settings model with a new field that has a default value, when a config file without that field is loaded, then the new field uses its default value without error
