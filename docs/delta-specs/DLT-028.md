# Delta Spec: DLT-028 - Resume conversation on topic revisit

<!-- This spec describes WHAT the delta does, not HOW it's implemented. -->

## Story

As a user, I want the assistant to recognize when I'm returning to a previously discussed topic so that my earlier conversation context is restored instead of starting from scratch.

## What It Does

When the boundary detector identifies a topic shift, it now also checks whether the incoming message matches a recently closed session by comparing against their summaries. If a match is found, the system resumes that earlier conversation — reopening the session, restoring the SDK context, and injecting a bridging summary of what happened in between. When no match is found, the existing fresh-session behavior applies unchanged.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | On topic shift, compare the incoming message against recent closed sessions and resume matching ones instead of starting fresh | Core goal |
| R1 | Enhance boundary detection to include recent session summaries in prompt and return a matching session ID (or null) alongside the continuation classification | Must-have |
| R2 | Configurable time-based lookup window for recent sessions, defaulting to 1 day | Must-have |
| R3 | On match, reopen the matched session record and resume its SDK conversation context | Must-have |
| R4 | Track each resumption with a dedicated SessionResumption record | Must-have |
| R5 | Inject bridging context — chronologically concatenated summaries of intermediate sessions — into the system prompt on resumption | Must-have |
| R6 | Existing fresh-session transition behavior preserved when no resumption match is found (non-regression) | Must-have |
| R7 | Fail-open: matching/resumption failures fall back to current fresh-session behavior | Must-have |
| R8 | Post-processing prompts aware of resumed sessions to avoid re-extracting already-processed content | Nice-to-have |

## Acceptance Criteria

### Enhanced Boundary Detection (R0, R1)

- [ ] Given an active session with a summary and recent closed sessions within the lookup window, when a new message arrives, then boundary detection receives both the current session summary and the recent session summaries (with their IDs)
- [ ] Given the boundary detector identifies a topic shift that matches a recent closed session, when it returns the result, then the matched session's ID is included alongside the shift classification
- [ ] Given the boundary detector identifies a topic shift that does NOT match any recent session, when it returns the result, then no session ID is returned (null)
- [ ] Given the boundary detector identifies a continuation, when it returns the result, then no session matching is performed (same output as current behavior)
- [ ] Given no recent closed sessions exist within the lookup window, when boundary detection runs, then it behaves the same as current (shift classification only, no matching possible)
- [ ] Given the boundary detector identifies a topic shift that could match multiple recent sessions, when it selects a match, then it returns the single best match (the detector prompt instructs single-session selection)

### Session Lookup Window (R2)

- [ ] Given a configurable time-based lookup window (default: 1 day), when querying recent sessions for boundary detection, then only sessions closed within that window are included
- [ ] Given multiple closed sessions exist within the window, when queried, then all are provided to the boundary detector for matching
- [ ] Given sessions closed outside the time window, when queried, then they are excluded
- [ ] Given only interrupted sessions (no `sdk_session_id`) exist within the window, when queried, then they are excluded (sessions without an SDK session ID cannot be resumed because there is no SDK conversation context to restore)

### Session Resumption (R3)

- [ ] Given a topic shift with a matched session ID, when the transition executes, then the matched session is reopened (its `ended_at` is cleared) and it becomes the active session
- [ ] Given a session is resumed, when the SDK processes the next message, then it uses the matched session's `sdk_session_id` to resume the full prior conversation context
- [ ] Given the matched session cannot be found in the registry, when resumption is attempted, then it falls back to fresh-session behavior with a warning log
- [ ] Given the matched session is already open (edge case), when resumption is attempted, then it falls back to fresh-session behavior with a warning log
- [ ] Given a session is matched for resumption but the SDK rejects the session ID (expired, invalid, or error on resume), when the coordinator processes the next message, then it falls back to fresh-session behavior with the bridging context still injected and a warning logged
- [ ] Given a session is resumed, when the coordinator would normally run the pre-processing pipeline (first message of session), then pre-processing is skipped — the resumed SDK session already has full prior conversation context and the bridging summary covers intermediate sessions

### Resumption Tracking (R4)

- [ ] Given a session is successfully resumed, then a SessionResumption record is created capturing at minimum the session ID and resumption timestamp (exact schema determined in design)
- [ ] Given a session has been resumed multiple times, when its resumption history is queried, then all resumption records are available in chronological order — each resumption gets its own record
- [ ] Given a session is resumed, closed via topic shift, then resumed again, when its history is queried, then both resumption events are recorded independently

### Bridging Context (R5)

- [ ] Given a session is resumed and intermediate sessions occurred between the matched session's last close and now, when the system prompt is built for the first message, then a bridging context section listing intermediate session summaries chronologically is appended
- [ ] Given intermediate sessions have summaries, when bridging context is assembled, then each summary is listed as a separate item in chronological order
- [ ] Given intermediate sessions exist but some lack summaries, when bridging context is assembled, then only sessions with summaries are included
- [ ] Given no intermediate sessions exist (the resumed session was the immediately previous one), when the system prompt is built, then no bridging context is appended
- [ ] Given bridging context is injected, then it is only present for the first message of the resumed session and cleared afterward
- [ ] Given many intermediate sessions exist within the lookup window, when bridging context is assembled, then the total bridging context remains bounded in size (capped to a reasonable number of intermediate summaries to avoid excessive prompt consumption)

### Fallback to Current Behavior (R6)

- [ ] Given a topic shift with no matching session among recent candidates, when the transition executes, then current behavior applies: a fresh session is created and the previous conversation summary is injected into the system prompt
- [ ] Given the fallback path is taken, when the system prompt is built, then it matches current behavior exactly (no regression)

### Graceful Degradation (R7)

- [ ] Given session matching produces an invalid or unfound session ID, when the coordinator processes the transition, then it falls back to fresh-session behavior and logs a warning
- [ ] Given the boundary detector fails to return valid structured output (SDK error, timeout, malformed JSON), when the error is caught, then the message proceeds as a continuation (fail-open, same as current)
- [ ] Given the session registry fails during resumption (reopen fails, query fails), when the error is caught, then it falls back to fresh-session behavior and logs the error
- [ ] Given resumption tracking fails (SessionResumption record creation fails), when the error occurs, then the session is still resumed successfully — tracking is best-effort

### Post-Processing Awareness (R8)

- [ ] Given a resumed session eventually closes, when post-processing runs, then processors have access to information indicating the session was resumed and when
- [ ] Given a session was resumed, when post-processing runs memory extraction on the full transcript, then it does not produce duplicate extractions for conversation content that was already processed during the session's previous closure

- **docs/feature-specs/agent/sessions.md** - Modifies: session registry gains reopen capability (clearing `ended_at` on a closed session); Adds: new SessionResumption model for tracking resumption events
- **docs/feature-specs/agent/core-architecture.md** - Modifies: coordinator transition logic gains a resume path alongside the existing fresh-session path; system prompt injection gains a bridging context variant for resumed sessions
- **docs/feature-specs/configuration/config-system.md** - Adds: new configuration option for the session resumption lookup window (`[agent]` section or similar)
- **docs/feature-specs/agent/post-processing-pipeline.md** - Modifies: processors may receive resumption metadata to avoid duplicate extraction on resumed sessions
