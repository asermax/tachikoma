# Delta Spec: DLT-001 - Set up core agent architecture with terminal REPL

## Story

As a developer, I want a working agent skeleton with a terminal REPL so that I have a foundation to build features on and can interact with the agent immediately.

## What It Does

Sets up the core agent using the Claude Agent SDK Python package (`claude-agent-sdk`). The application wraps the SDK client with its own orchestration loop — receive a message, pass to the agent, return the response. A terminal REPL provides the first interactive channel, streaming responses token-by-token. Without any other deltas, the agent is a conversationalist with basic file access.

**Out of scope:** This delta does NOT implement pre-processing, post-processing, delegation, or idle task processing — it establishes the core loop that future deltas extend.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Core agent loop: receive a user message, pass it to the Claude agent, return the response | Core goal |
| R1 | Project structure: pyproject.toml with dependencies, src package layout, entry point | Must-have |
| R2 | Programmatic entry point that channels (REPL, Telegram) can call to send messages and get streamed responses | Must-have |
| R3 | Terminal REPL that reads user input, sends it through the agent loop, and streams the response | Must-have |

## Acceptance Criteria

**R0 — Core agent loop**
- [ ] Given a user message, when passed to the agent, then the agent responds using the Claude model via the SDK
- [ ] Given the agent is in a conversation, when the user sends a follow-up message, then the agent has context from prior messages in the same conversation
- [ ] Given the agent has SDK built-in file tools (Read, Glob, Grep), when the user asks about files, then the agent can explore the process's working directory
- [ ] Given the agent receives a message, when the Claude API call fails with a transient error (network, rate limit), then the user sees a meaningful error message and the conversation remains usable
- [ ] Given the agent starts, when the API key is missing or invalid, then the agent fails fast with a clear error message

**R1 — Project structure**
- [ ] Given a fresh clone, when running `uv sync`, then all dependencies are installed and the project is ready to run
- [ ] Given the project structure, when inspected, then it follows the src layout convention (`src/tachikoma/`)
- [ ] Given the pyproject.toml, when inspected, then `claude-agent-sdk` is listed as a dependency

**R2 — Programmatic entry point**
- [ ] Given a channel implementation, when it calls the entry point with a user message, then it receives an async iterator that yields response content as it streams
- [ ] Given the entry point, when a new conversation starts, then a new SDK session is created
- [ ] Given an active conversation, when subsequent messages arrive, then they are sent to the same SDK session

**R3 — Terminal REPL**
- [ ] Given the REPL is started, when the user types a message and presses enter, then the agent's response is streamed token-by-token to the terminal
- [ ] Given the REPL is running, when the user sends multiple messages, then conversation context is preserved between them
- [ ] Given the REPL is running, when the user presses Ctrl+C or types an exit command, then streaming stops, no stack trace is printed, and the process exits cleanly
- [ ] Given the REPL is running, when the user submits an empty or whitespace-only message, then no request is sent to the agent and the REPL prompts for the next input
- [ ] Given the agent is streaming a response, when the API connection drops mid-stream, then partial output remains visible and an error message is shown
