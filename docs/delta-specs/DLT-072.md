# Delta Spec: DLT-072 - Fix task management MCP tool bugs

## Story

As a user, I want the agent to manage my tasks reliably through its MCP tools so that I don't encounter broken updates, missing information, or confusing error messages when creating, listing, or modifying tasks.

## What It Does

Fixes multiple bugs in the task management MCP tools that currently force the agent to fall back to raw SQLite queries. Adds missing task IDs to list output, exposes the `task_type` parameter in `update_task`, improves error messages to surface root causes, corrects misleading `notify` documentation, and enriches both tool descriptions and the system preamble with accurate parameter guidance.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Agent can manage tasks entirely through MCP tools without falling back to raw SQL | Core goal |
| R1 | `list_tasks` output includes task IDs so the agent can reference them in `update_task`/`delete_task` | Must-have |
| R2 | `update_task` exposes `task_type` parameter to allow changing between "session" and "background" | Must-have |
| R3 | `update_task` error messages surface the underlying cause, not just a generic wrapper | Must-have |
| R4 | Preamble accurately describes `notify` default behavior: silent on success, notifies on failure | Must-have |
| R5 | Tool `@tool()` descriptions include parameter types, validations, and usage examples | Must-have |
| R6 | Preamble Tasks section describes when to use each tool, task behavior, and cross-references tools | Must-have |

## Acceptance Criteria

### Task ID visibility (R1)

- [ ] Given the agent calls `list_tasks`, when definitions exist, then each entry in the output includes the task's `id` field
- [ ] Given the agent calls `list_tasks` and sees an ID, when it passes that ID to `update_task` or `delete_task`, then the operation succeeds without the agent needing any other discovery mechanism

### Task type updates (R2)

- [ ] Given the agent calls `update_task` with `task_type="session"` on a background task, then the definition's `task_type` is updated to "session"
- [ ] Given the agent calls `update_task` with `task_type="background"` on a session task, then the definition's `task_type` is updated to "background"
- [ ] Given the agent calls `update_task` with an invalid `task_type` value, then a clear error is returned indicating the valid options ("session" or "background")

### Error message clarity (R3)

- [ ] Given `update_task` fails due to a repository error, when the error is returned to the agent, then the message includes the underlying cause (e.g., the database error), not just "Failed to update task definition {id}"
- [ ] Given `update_task` receives an invalid value for a known field (e.g., non-boolean for `enabled`), then the error message identifies which field failed and what values are accepted

### Notify documentation accuracy (R4)

- [ ] Given the preamble Tasks section describes the `notify` parameter, then it states that failures always generate a notification regardless of the `notify` field
- [ ] Given the preamble describes `notify`, then it states that `notify` controls the success notification message — when omitted, successful background tasks complete silently
- [ ] Given the `create_task` tool description references `notify`, then it matches the corrected behavior (not "run silently" for all cases)
- [ ] Given the preamble describes `notify`, then it clarifies that the `notify` field has no effect on failure notifications (failures use an error template, ignoring `notify`)

### Tool description enrichment (R5)

- [ ] Given each MCP tool's `@tool()` description, then it includes parameter names with their types and whether they are required or optional
- [ ] Given `create_task`'s description, then it includes an example schedule (cron and one-shot) and valid `type` values
- [ ] Given `update_task`'s description, then it documents which fields can be updated and that `task_id` is required (obtained from `list_tasks`)
- [ ] Given `list_tasks`'s description, then it explains the `archived` parameter and default behavior (shows active tasks)
- [ ] Given `delete_task`'s description, then it notes that `task_id` is required and the operation is permanent

### Preamble accuracy (R6)

- [ ] Given the preamble Tasks section, then it describes task types (session vs background) with clear guidance on when to use each
- [ ] Given the preamble, then it describes scheduling formats with examples
- [ ] Given the preamble, then it lists each tool with a description of purpose, key parameters, and cross-references to related tools (e.g., "use `list_tasks` to get task IDs for `update_task`/`delete_task`")
- [ ] Given the preamble, then the `notify` field description matches the corrected behavior from R4
