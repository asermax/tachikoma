# Delta Spec: DLT-024 - Package and install agent as a uv tool

## Story

As a user, I want to install Tachikoma as a standalone CLI tool via `uv tool install tachikoma` so that I can run it from anywhere without managing a source checkout or Python environment.

## What It Does

Packages Tachikoma for distribution on PyPI so that users can install it with `uv tool install tachikoma` and run it via the `tachikoma` command. Includes project packaging configuration (build system, entry points, metadata), CLI restructuring to support subcommands (`tachikoma run`), a GitHub Actions CD pipeline with automatic semantic versioning and PyPI publishing, and installation documentation.

## Requirements

| ID | Requirement | Status |
|----|-------------|--------|
| R0 | Tachikoma installable as standalone CLI via `uv tool install tachikoma` from PyPI | Core goal |
| R1 | CLI exposes `tachikoma run` subcommand (with `--channel` flag); bare `tachikoma` defaults to `run`; extensible for future subcommands | Must-have |
| R2 | Package metadata and `uv_build` build system configured for PyPI (name, version, description, MIT license, classifiers, URLs) | Must-have |
| R3 | Non-Python data files (builtin skill `.md` files) included in distributed package | Must-have |
| R4 | GitHub Actions CD workflow publishes to PyPI via trusted publisher (OIDC) on version bump; includes quality gates (lint, typecheck, test) before publishing | Must-have |
| R5 | Automatic semantic versioning via python-semantic-release — determines version from conventional commits, stamps pyproject.toml, creates git tag, generates CHANGELOG.md | Must-have |
| R6 | Installation documentation for end users (install, configure, upgrade) | Must-have |
| R7 | Development workflow (`just run`, `just test`, etc.) preserved for contributors | Must-have |

## Acceptance Criteria

### Packaging & Installation (R0, R2, R3)

- [ ] Given pyproject.toml is configured with `[build-system]`, `[project.scripts]` entry, and PyPI metadata, when `uv build` is run, then a wheel and sdist are produced successfully
- [ ] Given the built package, when inspecting the wheel contents, then non-Python data files (`.md` files in `tachikoma/skills/builtin/`) are included
- [ ] Given a user runs `uv tool install tachikoma`, when installation completes, then the `tachikoma` command is available on PATH
- [ ] Given a fresh install with no prior config, when the user runs `tachikoma`, then the config file is auto-generated at `~/.config/tachikoma/config.toml` and the bootstrap sequence runs normally
- [ ] Given pyproject.toml, when inspected, then it includes: name, version, description, `readme`, MIT license, requires-python, classifiers, project URLs, and dependencies
- [ ] Given a LICENSE file at the project root, when inspected, then it contains a valid MIT license

### CLI Structure (R1)

- [ ] Given the installed CLI, when the user runs `tachikoma run`, then the agent starts with the REPL channel (default)
- [ ] Given the installed CLI, when the user runs `tachikoma run --channel telegram`, then the agent starts with the Telegram channel
- [ ] Given the installed CLI, when the user runs `tachikoma` with no subcommand, then it defaults to `tachikoma run`
- [ ] Given the installed CLI, when the user runs `tachikoma --help`, then available subcommands are listed (including `run`)
- [ ] Given the installed CLI, when the user runs `tachikoma run --help`, then the `--channel` flag and its options are documented
- [ ] Given the user runs `tachikoma run --channel invalid`, then a clear error is shown listing valid channel options
- [ ] Given the user runs `tachikoma unknown-subcommand`, then a helpful error message is shown with available subcommands

### Continuous Delivery (R4, R5)

- [ ] Given a push to master with a `feat:` commit, when the CD workflow completes successfully, then a new minor version tag exists, a GitHub Release is created, and the package is available on PyPI at the new version
- [ ] Given a push to master with a `fix:` commit, when the CD workflow completes, then a new patch version tag and release are created
- [ ] Given a push to master with only `docs:` or `chore:` commits, when the CD workflow runs, then no release is created and no publish occurs
- [ ] Given a version bump is triggered, when the workflow runs, then CHANGELOG.md is updated with the new release's changes
- [ ] Given a quality gate failure (lint, typecheck, or test), when the CD workflow runs, then no release or publish occurs
- [ ] Given the CD workflow, when two pushes to master happen in quick succession, then only one release workflow runs at a time (concurrency control)
- [ ] Given a version bump succeeded but the PyPI publish step fails, then the GitHub Release is still created and the error is surfaced in workflow logs

### Documentation (R6)

- [ ] Given the README.md, when a user reads the installation section, then it documents: prerequisites (uv, Python >=3.12), install command (`uv tool install tachikoma`), first-run behavior (config generation), `ANTHROPIC_API_KEY` environment variable setup, channel selection (`tachikoma run --channel telegram`), and upgrade command (`uv tool upgrade tachikoma`)

### Development Workflow (R7)

- [ ] Given a developer clones the repository, when they run `just install && just run`, then the agent starts in development mode
- [ ] Given a developer runs `just test`, then tests execute successfully
- [ ] Given a developer runs `just check`, then lint, typecheck, and tests all pass
