# Design: DLT-024 - Package and install agent as a uv tool

**Delta Spec**: [../delta-specs/DLT-024.md](../delta-specs/DLT-024.md)
**Status**: Approved

## Purpose

This document explains the design rationale for this delta: the modeling choices, data flow, system behavior, and architectural approach.

After implementation, the "Detected Impacts" section will guide reconciliation into feature design docs.

## Problem Context

Tachikoma currently runs only from a source checkout via `PYTHONPATH=src uv run python -m tachikoma`. This ties users to cloning the repository and managing the Python environment manually. To enable broader usage and automatic updates, the project needs to be packaged for distribution on PyPI and installable as a standalone CLI tool.

**Constraints:**
- The project uses uv as its package manager (ADR-001) and cyclopts for CLI parsing
- Non-Python data files (`.md` files in builtin skills) must be included in the distribution
- The existing development workflow (`just run`, `just test`, etc.) must be preserved
- The project is at v0.1.0 — versioning should follow conventional commits semantics
- PyPI publishing must use OIDC trusted publisher (no stored API tokens)

**Interactions:**
- Core architecture (`__main__.py`): entry point restructured from `@app.default` to `@app.command` with a `[project.scripts]` callable
- Configuration system: `--channel` flag moves under `tachikoma run` subcommand
- Telegram channel: same behavioral change in CLI invocation path

## Design Overview

The design spans four concerns: **packaging** (making the project buildable and installable), **CLI restructuring** (subcommand-based interface), **continuous delivery** (automated versioning and publishing), and **documentation** (installation guide).

```
┌─────────────────────────────────────────────────────────┐
│                    pyproject.toml                         │
│  [build-system] → uv_build backend                       │
│  [project.scripts] → tachikoma = __main__:cli            │
│  [tool.semantic_release] → version + changelog config    │
└──────────────────┬──────────────────────────────────────┘
                   │
        ┌──────────┴──────────┐
        ▼                     ▼
┌───────────────┐   ┌─────────────────────────────┐
│  CLI Layer    │   │  CD Pipeline                 │
│  __main__.py  │   │  .github/workflows/          │
│               │   │  release.yml                 │
│  cli()        │   │                              │
│    └─ app()   │   │  quality ──► release         │
│  @app.command │   │  (lint,       (PSR version,  │
│    run()      │   │   typecheck,   uv build,     │
│  @app.default │   │   test)        PyPI publish, │
│    → run()    │   │                GH release)   │
└───────────────┘   └─────────────────────────────┘
```

## Components

### Implementation Structure

| Layer/Component | Responsibility | Key Decisions |
|-----------------|----------------|---------------|
| `pyproject.toml` | Package definition, build system, metadata, PSR config | `uv_build` backend; `[project.scripts]` for CLI entry point; PSR `"conventional"` parser |
| `src/tachikoma/__main__.py` | CLI entry point with subcommand structure | cyclopts `@app.command` for `run`, parameterless `@app.default` delegates to `run()` with defaults; `def cli(): app()` wrapper for `[project.scripts]` |
| `.github/workflows/release.yml` | CD pipeline: quality gates → version → build → publish | Two-job: `quality` (lint/typecheck/test) → `release` (PSR + PyPI OIDC + GH release) |
| `LICENSE` | MIT license text | Standard MIT with copyright holder |
| `README.md` | Installation and usage documentation | Separate sections for end-user install and developer setup |
| `justfile` | Development task recipes | Change `run` recipe to `uv run tachikoma {{args}}` |

### Cross-Layer Contracts

**Entry point chain:**

```
uv tool install tachikoma
  → [project.scripts] tachikoma = "tachikoma.__main__:cli"
    → cli() calls app()
      → cyclopts dispatches to run() or default()
        → default() calls run() with defaults
```

**`python -m tachikoma` (dev mode):**

```
python -m tachikoma
  → __main__.py executed as script
    → if __name__ == "__main__": cli()
      → same chain as above
```

**CD pipeline flow:**

```
push to master
  → quality job: checkout → uv setup → uv sync --all-groups → ruff check → ty check → pytest
    → release job (needs: quality):
      → checkout (fetch-depth: 0) → uv setup → uv sync --all-groups
      → PSR action: analyze commits → bump version → update CHANGELOG.md → commit → tag → push → GH release
      → if released == 'true':
        → uv build: produce wheel + sdist in dist/
        → pypa/gh-action-pypi-publish: upload to PyPI via OIDC
        → publish-action: attach dist/ to GitHub Release
```

### Shared Logic

None — each component is self-contained. The CLI entry point (`cli()`) is the only shared interface, consumed by both `[project.scripts]` and `if __name__ == "__main__"`.

## Modeling

No new domain models. This delta modifies project configuration and restructures the existing CLI entry point. The only new code artifact is the `cli()` wrapper function in `__main__.py`.

## Data Flow

### Package Build Flow

```
pyproject.toml (version, metadata, build-system)
  → uv build
    → uv_build backend reads [project] + [build-system]
    → Collects Python files from src/tachikoma/
    → Collects non-Python files from src/tachikoma/ (including .md in skills/builtin/)
    → Produces dist/tachikoma-{version}-py3-none-any.whl
    → Produces dist/tachikoma-{version}.tar.gz
```

### Version Bump Flow

```
Conventional commits on master
  → PSR analyzes commits since last tag
    → feat: → minor bump
    → fix:/perf: → patch bump
    → BREAKING CHANGE footer → minor bump (major_on_zero = false)
    → docs:/chore: only → no release
  → Stamps new version in pyproject.toml:project.version
  → Runs build_command: uv lock && uv build
    → uv.lock updated with new version
    → dist/ produced
  → Updates CHANGELOG.md (update mode — prepends new entry, preserves existing)
  → Commits version + lock + changelog
  → Creates git tag (v{version})
  → Pushes to origin
  → Creates GitHub Release with changelog as notes
```

## Key Decisions

### Build Backend: uv_build

**Choice**: `uv_build` as the `[build-system]` backend
**Why**: The project already uses uv for dependency management (ADR-001). Using uv's native build backend avoids introducing a separate build tool (setuptools, hatchling) and ensures tight integration with the existing toolchain.
**Sources**: uv docs — `https://docs.astral.sh/uv/concepts/build-backend/`
**Options Researched**:
- `uv_build`: Native uv backend, auto-includes files in package directory, minimal config
- `hatchling`: Popular modern backend, more config options, but adds a dependency outside the uv ecosystem
- `setuptools`: Legacy backend, requires `setup.cfg`/`setup.py`, more boilerplate
**Why This Over Alternatives**: `uv_build` requires zero extra configuration for data file inclusion (`.md` files in package dir are auto-included), aligns with ADR-001, and keeps the toolchain unified.
**Consequences**:
- Pro: No extra build-time dependencies beyond uv
- Pro: Data files in package directory automatically included — no `package_data` or `MANIFEST.in`
- Con: `uv_build` version must be pinned (`>=0.10.9,<0.11.0`) as it follows uv's rapid release cadence

### CLI Default Command: Parameterless @app.default Delegation

**Choice**: Parameterless `@app.default` function that calls `run()` with defaults. Bare `tachikoma` starts the agent with default settings. The `--channel` flag is only available via `tachikoma run --channel`. `tachikoma --channel telegram` is not supported — this follows standard CLI conventions where flags belong to their subcommand (like `docker run --env`, `cargo build --release`).
**Why**: Cyclopts has no built-in "redirect bare invocation to subcommand" mechanism. The alternatives are duplicating the function signature (which forwards flags) or using a clean delegation pattern.
**Sources**: cyclopts docs — `https://cyclopts.readthedocs.io/en/stable/`
**Options Researched**:
- Parameterless delegation: `@app.default` calls `run()` with no args — clean, no duplication
- Signature duplication: Both `@app.default` and `@app.command` accept `--channel` — allows `tachikoma --channel telegram` but creates maintenance burden and blurs the subcommand boundary
- No default (show help): Bare `tachikoma` shows help page — conflicts with spec R1 requiring bare invocation defaults to `run`
**Why This Over Alternatives**: Parameterless delegation keeps the CLI interface clean — flags belong to their subcommand, which is the standard pattern for extensible CLIs. Avoids parameter duplication and drift.
**Consequences**:
- Pro: Single source of truth for the `run` command implementation
- Pro: Future subcommands (e.g., `tachikoma config`, `tachikoma version`) just add `@app.command` functions
- Pro: Clean CLI structure — `tachikoma --help` shows subcommands, `tachikoma run --help` shows run-specific flags
- Con: Users must type `tachikoma run --channel telegram` instead of `tachikoma --channel telegram`

### Semantic Versioning: python-semantic-release

**Choice**: python-semantic-release v10.x with `"conventional"` commit parser
**Why**: Automates the entire release cycle (version bump, changelog, tag, release) from commit messages, eliminating manual version management.
**Sources**: PSR docs — `https://python-semantic-release.readthedocs.io/en/latest/`; GitHub Action v10.x
**Options Researched**:
- python-semantic-release: Full automation, GH Action available, conventional commit support
- `commitizen`: Similar automation, but PSR has better GH Actions integration and a dedicated publish-action
- Manual versioning with `bump2version`: Requires explicit version commands, no changelog generation
**Why This Over Alternatives**: PSR's official GitHub Action handles the full pipeline (version → build → tag → release) in a single step, with output variables that chain cleanly to PyPI publish. The `"conventional"` parser is current (the `"angular"` parser is deprecated and will be removed in v11).
**Consequences**:
- Pro: Zero-touch releases — push to master with conventional commits triggers everything
- Pro: CHANGELOG.md generated automatically
- Con: Requires discipline with conventional commit format (enforced by CI culture, not tooling in this delta)

### PyPI Publishing: Trusted Publisher (OIDC)

**Choice**: `pypa/gh-action-pypi-publish` with OIDC trusted publisher, no stored API tokens
**Why**: Eliminates long-lived secrets in the repository. The OIDC flow exchanges short-lived (15-minute) tokens between GitHub Actions and PyPI.
**Sources**: PyPI docs — `https://docs.pypi.org/trusted-publishers/`; `pypa/gh-action-pypi-publish` — `https://github.com/pypa/gh-action-pypi-publish`
**Options Researched**:
- OIDC trusted publisher: No secrets, short-lived tokens, recommended by PyPI
- API token in GitHub secrets: Works but requires manual token rotation and risks exposure
- `twine upload` with credentials: Legacy approach, manual, no OIDC support
**Why This Over Alternatives**: OIDC is PyPI's recommended approach, eliminates secret management, and the `pypa/gh-action-pypi-publish` action handles the token exchange transparently.
**Consequences**:
- Pro: No API tokens stored in repository secrets
- Pro: Token automatically scoped to the specific workflow/environment
- Con: Requires one-time PyPI-side setup (pending publisher registration)
- Con: Does not work from reusable workflows (`workflow_call`) — must be in a standard workflow file

## System Behavior

### Scenario: First-time Installation

**Given**: A user with uv installed and Python >=3.12
**When**: They run `uv tool install tachikoma`
**Then**: uv downloads the package from PyPI, creates an isolated virtual environment, and symlinks the `tachikoma` command to `~/.local/bin/`. Running `tachikoma` starts the agent with REPL channel, auto-generating config at `~/.config/tachikoma/config.toml`.
**Rationale**: The `[project.scripts]` entry point and cyclopts default command chain ensure bare invocation works identically to `tachikoma run`.

### Scenario: Feature Commit Triggers Release

**Given**: A `feat: add new capability` commit is pushed to master
**When**: The CD workflow runs
**Then**: The quality job passes (lint + typecheck + test), PSR determines a minor version bump (e.g., 0.1.0 → 0.2.0), stamps pyproject.toml, updates CHANGELOG.md, creates a `v0.2.0` tag, pushes, creates a GitHub Release, builds the wheel, publishes to PyPI, and attaches dist to the release.
**Rationale**: Full automation from commit to published package with quality gates as a prerequisite.

### Scenario: Non-releasable Commit

**Given**: A `docs: update README` commit is pushed to master
**When**: The CD workflow runs
**Then**: The quality job passes. PSR analyzes commits and determines no version bump is needed (`steps.release.outputs.released == 'false'`). The build, PyPI publish, and GH release upload steps are skipped (gated on `released == 'true'`). The release job completes successfully with no artifacts produced.
**Rationale**: Only `feat:`, `fix:`, and `perf:` commits (or `BREAKING CHANGE` footers) trigger releases. The conditional gate ensures no build or publish runs when there's nothing to release.

### Scenario: Quality Gate Failure

**Given**: A commit is pushed to master that breaks a test
**When**: The CD workflow runs
**Then**: The quality job fails. The release job never executes. No version bump, no publish.
**Rationale**: The release job depends on the quality job (`needs: quality`), ensuring broken code is never released.

### Scenario: Concurrent Pushes to Master

**Given**: Two pushes to master happen within seconds
**When**: Both trigger the CD workflow
**Then**: The concurrency group (`group: release`, `cancel-in-progress: false`) ensures only one release workflow runs at a time. The second queues behind the first.
**Rationale**: Prevents race conditions in version determination and tag creation.

### Scenario: PyPI Publish Failure

**Given**: PSR successfully creates a version tag and GitHub Release, but PyPI upload fails (network issue, etc.)
**When**: The `pypa/gh-action-pypi-publish` step fails
**Then**: The GitHub Release still exists with the correct tag and changelog. The PyPI publish step can be retried by re-running the failed workflow job.
**Rationale**: PSR commits, tags, and creates the GH release before the PyPI publish step. A publish failure doesn't roll back the release — it's a separate, retriable step.

### Scenario: Developer Workflow Preservation

**Given**: A developer clones the repo and runs `just install`
**When**: `uv sync --all-groups` runs
**Then**: The package is installed in editable mode (since `package = true`). `just run` works without `PYTHONPATH=src`, `just test` and `just check` work as before.
**Rationale**: Switching from `package = false` to `package = true` means `uv sync` installs the package in editable mode, making the `src` directory importable without `PYTHONPATH` manipulation.
