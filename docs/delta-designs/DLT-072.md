# Design: DLT-072 - Fix task management MCP tool bugs

**Delta Spec**: [../delta-specs/DLT-072.md](../delta-specs/DLT-072.md)
**Status**: Approved

## Purpose

This document explains the design rationale for this delta: the modeling choices, data flow, system behavior, and architectural approach.

After implementation, the "Detected Impacts" section will guide reconciliation into feature design docs.

## Problem Context

The task management MCP tools (`list_tasks`, `create_task`, `update_task`, `delete_task`) have several bugs and documentation gaps that force the agent to fall back to raw SQLite queries instead of using the tools as intended:

1. `list_tasks` omits task IDs from output, so the agent cannot reference tasks in `update_task`/`delete_task`
2. `update_task` doesn't expose `task_type`, preventing the agent from changing a task between "session" and "background"
3. Error messages wrap the root cause in a generic string, hiding actionable information from the agent
4. `@tool()` descriptions are minimal one-liners lacking parameter types, validation rules, and examples
5. The preamble's `notify` documentation is misleading — it says "if omitted, background tasks run silently" without clarifying that failures always generate notifications regardless of the `notify` field

**Constraints:**
- All changes are localized to `tasks/tools.py` and `context/loading.py` — no new modules, no new external dependencies
- Tool return format (`{"content": [...]}` / `{"is_error": true, "content": [...]}`) is unchanged (DES-006)
- Repository API (`update_definition(**fields)`) already accepts `task_type` — no repository changes needed

## Design Overview

Five localized fixes across two files. The tools module (`tasks/tools.py`) gets: task IDs in list output, `task_type` on `UpdateTaskArgs`, structured error handling with cause surfacing, and enriched `@tool()` descriptions. The preamble constant in `context/loading.py` gets a rewritten Tasks section with corrected `notify` behavior and tool cross-references.

No new abstractions, no new modules, no dependency changes.

### Cross-Layer Contracts

**Error response contract (unchanged shape, improved content):**

All four tools follow the same error handling structure after this delta:

```python
try:
    # ... tool logic ...
except TaskRepositoryError as exc:
    # Surface both wrapper and root cause
    cause = f" Cause: {exc.__cause__}" if exc.__cause__ else ""
    return {"is_error": True, "content": [{"type": "text", "text": f"{exc}{cause}"}]}
except Exception as exc:
    # Unexpected errors — generic fallback
    return {"is_error": True, "content": [{"type": "text", "text": f"Unexpected error: {exc}"}]}
```

`ValidationError` is caught earlier (before tool logic) in `create_task`, `update_task`, and `delete_task` — these already produce field-level messages via Pydantic's built-in error formatting. No changes needed to those existing `ValidationError` catches.

**Note on `list_tasks` structural difference:** Unlike the other three tools, `list_tasks` currently wraps all logic (including `model_validate`) in a single `except Exception` block rather than catching `ValidationError` separately. For S3, normalize `list_tasks` to match: catch `ValidationError` first for arg parsing, then wrap the repository call in `TaskRepositoryError` + `Exception` catches. This gives `list_tasks` the same two-tier error structure as the other tools.

**Integration Points:**
- `UpdateTaskArgs.task_type` → `repository.update_definition(id, task_type=value)` — repository already accepts this field via `**fields`
- `list_tasks` output format change — no downstream consumers; only the agent reads this output. Since the agent processes tool output per-turn (not persisted), there is no backward compatibility concern with the format change

### S4: Example Enriched Tool Description

To set the tone and detail level for all four tools, here is an example enriched description for `create_task`:

```
Create a new scheduled task definition.

Parameters:
- name (str, required): Human-readable task name
- schedule (str, required): Cron expression (e.g., "0 9 * * *" for daily at 9 AM)
  or ISO datetime for one-shot (e.g., "2026-04-01T14:00:00Z")
- type (str, required): "session" (delivered during idle) or "background" (isolated execution)
- prompt (str, required): Instruction the agent follows when the task fires
- notify (str, optional): Success notification instruction — when set, generates a
  user-facing message on completion. Omit for silent success. Failures always notify
  regardless of this field.
- enabled (bool, optional, default true): Whether the task is active
```

The other three tools follow the same pattern: parameter list with types, required/optional, valid values, and behavioral notes where relevant.

## Modeling

### UpdateTaskArgs (changed)

```
UpdateTaskArgs (Pydantic BaseModel)
├── task_id: str                                          (required)
├── name: str | None = None                               (optional)
├── schedule: str | None = None                           (optional)
├── task_type: Literal["session", "background"] | None    (optional — NEW)
├── prompt: str | None = None                             (optional)
├── notify: str | None = None                             (optional)
└── enabled: bool | None = None                           (optional)
```

Using `Literal["session", "background"]` (matching `TaskType` from `model.py`) rather than a plain `str` with runtime validation. This gives:
- Pydantic validates automatically at parse time, producing a clear error like `Input should be 'session' or 'background'`
- JSON schema includes `enum: ["session", "background"]` so the agent sees valid values in the tool schema
- No manual validation code needed in the handler

## Data Flow

### S1: Task listing with IDs

```
1. Agent calls list_tasks
2. Tool queries repository (unchanged)
3. For each definition, output line format changes:
   Before: "- **{name}** [{task_type}] {status}"
   After:  "- [{id}] **{name}** [{task_type}] {status}"
4. Agent can now extract ID and pass to update_task/delete_task
```

### S2: Task type update flow

```
1. Agent calls update_task with task_type="session"
2. Pydantic validates task_type against Literal["session", "background"]
   - Invalid value → Pydantic raises ValidationError with clear message
3. Handler adds task_type to updates dict (same path as other optional fields)
4. repository.update_definition(id, task_type="session") updates the record
5. Success response returned to agent
```

### S3: Error surfacing flow

```
1. Tool logic raises (e.g., repository operation fails)
2. Repository wraps as TaskRepositoryError with __cause__ = original exception
3. Tool catches TaskRepositoryError specifically
4. Error message includes both: "Failed to update task definition {id}" + "Cause: {original}"
5. Agent sees actionable error (e.g., database constraint violation) instead of opaque wrapper
```

## Key Decisions

### Literal type for task_type validation

**Choice**: Use `Literal["session", "background"] | None` on `UpdateTaskArgs` instead of `str | None` with manual validation
**Why**: Pydantic handles validation automatically, produces clear error messages including the valid values, and generates a JSON schema with `enum` — the agent sees valid options in the tool schema without needing to parse the description.

**Alternatives Considered**:
- `str` with manual `if parsed.task_type not in (...)` check: Same approach used for `type` in `CreateTaskArgs` today. Works but duplicates validation logic that Pydantic can handle, and doesn't expose valid values in the JSON schema.

**Consequences**:
- Pro: Zero manual validation code for this field
- Pro: JSON schema self-documents valid values
- Pro: Error message is consistent with other Pydantic validation errors
- Con: Introduces a minor inconsistency — `CreateTaskArgs.type` is still `str` with manual validation. This inconsistency is acceptable since `CreateTaskArgs` is not in scope for this delta.

### Consistent error surfacing across all tools

**Choice**: Apply `TaskRepositoryError`-aware error handling to all four tools, not just `update_task`
**Why**: All four tools catch generic `Exception` today and all wrap repository errors the same way. Fixing only `update_task` would leave the other three with opaque error messages for the same class of failures.

**Alternatives Considered**:
- Fix only `update_task` (spec-scoped): Minimal change, but inconsistent — agent would get good errors from `update_task` and opaque errors from `create_task` for the same underlying failure.

**Consequences**:
- Pro: Consistent error quality across all task tools
- Pro: All repository errors surface root cause to the agent
- Con: Slightly larger change than spec-scoped minimum

## System Behavior

### Scenario: Agent lists tasks and uses ID to update

**Given**: Task definitions exist
**When**: Agent calls `list_tasks`, then `update_task` with an ID from the output
**Then**: The ID is visible in the list output, and passing it to `update_task` succeeds
**Rationale**: This is the core workflow that was broken — the agent couldn't discover IDs

### Scenario: Agent changes task type via update

**Given**: A background task definition exists
**When**: Agent calls `update_task` with `task_type="session"`
**Then**: The definition's `task_type` is updated to "session"
**Rationale**: Previously impossible — `task_type` was not exposed on `UpdateTaskArgs`

### Scenario: Agent passes invalid task_type to update

**Given**: A task definition exists
**When**: Agent calls `update_task` with `task_type="invalid"`
**Then**: Pydantic returns a `ValidationError` naming the field and valid values (`'session'` or `'background'`)
**Rationale**: Literal type validation produces clear, actionable error messages automatically

### Scenario: Repository error surfaces root cause

**Given**: A repository operation fails (e.g., database locked)
**When**: The tool catches `TaskRepositoryError`
**Then**: The agent sees both the wrapper message ("Failed to update task definition {id}") and the cause ("database is locked")
**Rationale**: Previously, the agent only saw the wrapper — the cause was hidden

### Scenario: Agent reads preamble and understands notify correctly

**Given**: The preamble Tasks section describes `notify`
**When**: The agent reads it
**Then**: It understands that `notify` controls the success notification message only, and that failures always generate a notification regardless of `notify`
**Rationale**: The previous wording ("if omitted, background tasks run silently") implied failures would also be silent
