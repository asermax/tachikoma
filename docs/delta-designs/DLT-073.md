# Design: DLT-073 - Block Claude Code built-in cron tools in default config

**Delta Spec**: [../delta-specs/DLT-073.md](../delta-specs/DLT-073.md)
**Status**: Draft

## Purpose

This document explains the design rationale for this delta: the modeling choices, data flow, system behavior, and architectural approach.

After implementation, the "Detected Impacts" section will guide reconciliation into feature design docs.

## Problem Context

Claude Code ships built-in cron tools (`CronCreate`, `CronDelete`, `CronList`) that create ephemeral in-memory scheduled jobs. Tachikoma has its own persistent task system (backed by SQLAlchemy + cron scheduling) that survives restarts. If the agent uses the built-in cron tools instead of the persistent task system, scheduled work silently vanishes when the process exits.

**Constraints:**
- The fix must be default-safe — new installations block cron tools without manual configuration
- Existing users who have explicitly set `disallowed_tools` in their config must not be affected
- The auto-generated default config must reflect the updated deny list

## Design Overview

Extend the default value of `AgentSettings.disallowed_tools` from `["AskUserQuestion"]` to `["AskUserQuestion", "CronCreate", "CronDelete", "CronList"]`. No new code paths, modules, or configuration keys are introduced. The existing config system infrastructure — Pydantic defaults, TOML override semantics, and dynamic default config generation — handles all requirements without modification.

### Cross-Layer Contracts

No new contracts. The `disallowed_tools` list flows through the existing path:

```
AgentSettings.disallowed_tools (default or TOML override)
  → Coordinator reads settings.agent.disallowed_tools
  → Passes to ClaudeAgentOptions
  → SDK enforces tool blocking
```

### Shared Logic

No new shared logic. The existing `_generate_default_config` function dynamically iterates `AgentSettings.model_fields` and formats list defaults as commented TOML arrays — the new entries appear automatically.

## Data Flow

No change to data flow. The `disallowed_tools` list follows the same path as before:

1. `AgentSettings` provides the default (now including cron tools)
2. If the user's TOML has an explicit `disallowed_tools`, Pydantic uses that value instead (complete replacement, not merge)
3. `Coordinator` reads the final value and passes it to `ClaudeAgentOptions`
4. `_generate_default_config` renders the default as a commented TOML array

## Key Decisions

### Append to existing default list rather than separate deny-list mechanism

**Choice**: Add the three cron tool names directly to the `disallowed_tools` default list
**Why**: The config system already supports a tool deny list with the exact semantics needed. Creating a separate "system-blocked tools" concept would add unnecessary complexity for a static list of tool names.
**Alternatives Considered**:
- Separate `system_disallowed_tools` field merged at runtime: Adds a new field, merge logic, and documentation for no functional benefit
- Hardcoded filtering in Coordinator: Bypasses the config system, making the blocking invisible to config inspection

**Consequences**:
- Pro: Zero new code paths — only a default value change
- Pro: Users can still override the full list if they have a reason to allow cron tools
- Con: Users who override `disallowed_tools` to block only specific tools will need to manually include the cron tools if they want them blocked

### Complete replacement semantics for TOML overrides

**Choice**: Rely on Pydantic's existing behavior where an explicit TOML value replaces the default entirely (no merge)
**Why**: This is how the config system already works for all list fields (`allowed_tools`, `disallowed_tools`). Introducing merge semantics for one field would be inconsistent and surprising.

**Consequences**:
- Pro: Consistent with existing config behavior
- Pro: Users have full control over the deny list
- Con: A user who explicitly sets `disallowed_tools = ["AskUserQuestion"]` will not get cron tool blocking — but this is intentional per R2

## System Behavior

### Scenario: Fresh installation

**Given**: No config file exists
**When**: The application starts for the first time
**Then**: `_generate_default_config` creates a commented config file showing `disallowed_tools = ["AskUserQuestion", "CronCreate", "CronDelete", "CronList"]`. The agent runs with this default, blocking all four tools.

### Scenario: Existing installation with no explicit disallowed_tools

**Given**: A config file exists but has no `disallowed_tools` entry
**When**: The application starts
**Then**: `AgentSettings` uses the new default. The agent blocks `AskUserQuestion`, `CronCreate`, `CronDelete`, and `CronList`.

### Scenario: Existing installation with explicit disallowed_tools override

**Given**: A config file with `disallowed_tools = ["AskUserQuestion"]`
**When**: The application starts
**Then**: Pydantic uses the explicit value. Only `AskUserQuestion` is blocked. Cron tools remain available (user's choice takes precedence per R2).
