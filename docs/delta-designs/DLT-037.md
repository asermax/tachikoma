# Design: DLT-037 - Ensure Telegram push notifications for streamed responses

**Delta Spec**: [../delta-specs/DLT-037.md](../delta-specs/DLT-037.md)
**Status**: Approved

## Purpose

This document explains the design rationale for this delta: the modeling choices, data flow, system behavior, and architectural approach.

After implementation, the "Detected Impacts" section will guide reconciliation into feature design docs.

## Problem Context

Tachikoma streams Telegram responses via progressive message editing: an initial `send_message` creates the message, then `edit_message_text` updates it as tokens arrive. This means the user's push notification fires for the incomplete initial message, not the final response. Worse, if the user has already closed the app by the time streaming starts, they see the notification for an empty or partial message — or none at all if the initial send happens while they're still in-chat (Telegram suppresses notifications for open chats).

The core problem: **edits don't trigger push notifications** — only new messages do.

**Constraints:**
- The Telegram Bot API's `copyMessage` is the only way to create a new message from an existing one without the "Forwarded from" header
- `copyMessage` returns only `MessageId` (not a full `Message`), which is sufficient for our needs
- `copyMessage` preserves all text entities automatically (confirmed via Telegram Bot API docs) — no manual entity passing needed
- `disable_notification` is available on `send_message` and `copy_message` but not on `edit_message_text` (edits never trigger notifications anyway)
- The `ResponseRenderer` already tracks `_current_message_id` which correctly points to the last message after splits

**Interactions:**
- `ResponseRenderer` (telegram.py): Owns message lifecycle — will gain `notify()` method and silent sending
- `TelegramChannel._process_through_coordinator` (telegram.py): Calls `notify()` after `finalize()` on Result events
- `TelegramSettings` (config.py): Gains `push_notifications` toggle

## Design Overview

The solution has two complementary parts:

1. **Silent streaming**: All messages created during the streaming phase (`send_message` calls for initial message, splits, and status) are sent with `disable_notification=True`. This suppresses all push notifications during streaming.

2. **Post-finalize copy+delete**: After the response is fully streamed and the final edit is flushed, the renderer copies the last message via `bot.copy_message` (creating a new message that triggers a push notification), then deletes the original. The copy inherits all formatting entities. Safe ordering ensures the original is preserved if copy fails.

The net effect: the user receives exactly one push notification per agent response — for the final, complete message. As a side benefit, the copied message appears after any steering messages the user sent during processing, preserving correct chronological order.

## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | Pass `disable_notification=True` on all `send_message` calls during streaming: initial message (`_flush`), splits (`_split_and_send`), and status messages (`handle_status`). Error messages are conditional: silent if text was already streamed (`_current_message_id is not None`), normal notification if the error is the only response. | |
| **S2** | Add a `notify()` method to `ResponseRenderer` that calls `bot.copy_message` to create a new message (triggers push), then `bot.delete_message` on the original, with safe ordering (copy first, skip delete on failure). Called from `_process_through_coordinator` after `finalize()` on `Result` event, and in the exception path. | |
| **S3** | `_current_message_id` already tracks the last message — `_split_and_send` sets it to `None` before sending the remainder, then `_flush` assigns the new message ID. No additional tracking needed. | |
| **S4** | `notify()` wraps both `copy_message` and `delete_message` in try/except, logs failures, does not propagate. If copy fails, original is preserved. If delete fails, duplicate is acceptable degradation. | |
| **S5** | Add `push_notifications: bool = True` to `TelegramSettings`. When disabled, skip `notify()` and send all messages with default notification behavior (current behavior). Renderer receives this flag via constructor. | |

## Components

### Implementation Structure

| Layer/Component | Responsibility | Key Decisions |
|-----------------|----------------|---------------|
| `ResponseRenderer` (telegram.py) | Silent message sending + post-finalize copy+delete via `notify()` | Owns entire message lifecycle; receives `push_notifications` flag via constructor |
| `TelegramChannel._process_through_coordinator` (telegram.py) | Calls `renderer.notify()` after `finalize()` on Result event | Orchestrates the renderer; also handles exception path |
| `TelegramSettings` (config.py) | `push_notifications: bool = True` toggle | Defaults to enabled; when disabled, reverts to current behavior |

### Cross-Layer Contracts

The existing sequence diagram from the Telegram feature design gains one additional step after `finalize()`:

```
Result event arrives
  → Channel calls renderer.finalize()     (final edit flushed)
  → Channel calls renderer.notify()       (copy+delete for push notification)
  → Channel calls renderer.reset()        (state cleared for next turn)
```

`notify()` is a no-op when:
- `push_notifications` is disabled in config
- No message was sent (`_current_message_id is None`) — handles empty responses and error-only responses

### Shared Logic

No new shared logic. All changes are contained within the Telegram channel module and config model.


The `_push_notifications` flag controls two behaviors:
- When `True`: all `send_message` calls pass `disable_notification=True`, and `notify()` performs copy+delete
- When `False`: all `send_message` calls use default notification behavior (no `disable_notification`), and `notify()` is a no-op

```
TelegramSettings
├── bot_token: str
├── authorized_chat_id: int

```

## Data Flow

### Happy path: streaming response with push notification

```
1. User sends message
2. Coordinator processes, agent starts streaming
3. ResponseRenderer._flush() creates initial message
   → send_message(..., disable_notification=True)     ← silent
4. Progressive edits update the message (no notifications on edits anyway)
5. If message splits:
   → split message sent with disable_notification=True  ← silent
6. Result event arrives
7. renderer.finalize() — final edit flushed
8. renderer.notify():
   a. bot.copy_message(chat_id, chat_id, _current_message_id)
      → Returns MessageId (new message, triggers push notification)
   b. bot.delete_message(chat_id, _current_message_id)
      → Original removed
9. renderer.reset() — state cleared
```

### Status message during streaming

```
1. Status event ("Thinking...") arrives before any text
2. renderer.handle_status(message):
   → send_message(..., disable_notification=True)     ← silent
3. Status message gets replaced when first TextChunk arrives
   (existing behavior — the status message_id becomes the edit target)
```

### Error as only response

```
1. Agent returns only an Error event (no text streamed)
2. renderer.handle_error(error):
   → _current_message_id is None (no text was sent)
   → send_message(...) WITHOUT disable_notification    ← triggers push
3. Result event arrives
4. renderer.notify():
   → _current_message_id is None → no-op (error message already triggered push)
5. renderer.reset()
```

### Error alongside streamed text

```
1. Agent streams some text, then an error
2. Text was sent with disable_notification=True
3. renderer.handle_error(error):
   → _current_message_id is not None (text was already sent)
   → send_message(..., disable_notification=True)     ← silent
   Note: handle_error does NOT update _current_message_id — it remains
   pointing at the text message, not the error message
4. Result event arrives
5. renderer.notify() → copy+delete on the text message → push notification
   The copied text message appears after both the error message and any
   user steering messages in the chat. This ordering is acceptable:
   the error is contextual, the text is the substantive response.
```

### Exception in _process_through_coordinator (no prior text)

```
1. Processing raises an unexpected exception before any text was streamed
2. _current_message_id is None
3. Exception handler sends error message:
   → send_message(...) WITHOUT disable_notification    ← triggers push
4. No notify() call needed — the error message itself is the notification
```

### Exception in _process_through_coordinator (after partial text)

```
1. Some text was streamed silently (_current_message_id is set)
2. Processing raises an unexpected exception
3. Exception handler calls renderer.notify() first:
   → copy+delete on the silently-sent text message → push notification
4. Exception handler sends error message:
   → send_message(..., disable_notification=True)      ← silent (notify already triggered push)
5. User receives push for the partial text, plus a visible error message
```

Note: The exception handler must check whether `_active_renderer` exists and has a `_current_message_id` before calling `notify()`. If no text was streamed, it falls through to the "no prior text" path above.

### Copy failure (graceful degradation)

```
1. Response streams normally, finalize() completes
2. renderer.notify():
   a. bot.copy_message() raises TelegramAPIError
   b. Exception caught, logged as warning
   c. delete_message() is NOT called
   d. Original message preserved — user still has the response
   e. No push notification for this response (acceptable degradation)
```

### Delete failure (acceptable duplicate)

```
1. Response streams normally, finalize() completes
2. renderer.notify():
   a. bot.copy_message() succeeds → new message created, push fires
   b. bot.delete_message() raises TelegramAPIError
   c. Exception caught, logged as warning
   d. User sees both original and copy (duplicate — acceptable degradation)
```

## Key Decisions

### copyMessage over delete+re-send

**Choice**: Use `bot.copy_message` to create the notification-triggering message, rather than deleting the original and re-sending with explicit text and entities.
**Why**: `copyMessage` is a server-side operation that preserves all text entities automatically. No need to store the final rendered text or manage entity serialization. Simpler and more reliable.
**Sources**: Telegram Bot API docs (copyMessage specification), aiogram 3.x Bot.copy_message signature
**Options Researched**: `copyMessage`, manual `deleteMessage` + `sendMessage` with stored text/entities, `forwardMessage` (adds "Forwarded from" header — rejected)
**Why This Over Alternatives**: `copyMessage` avoids the risk of entity serialization mismatches and doesn't require storing the final message text. `forwardMessage` adds an unwanted attribution header.

**Consequences**:
- Pro: Server-side entity preservation — zero risk of formatting loss
- Pro: No need to store or replay the final rendered text
- Pro: Atomic from the user's perspective — the copy appears as a normal message
- Con: Returns only `MessageId`, not full `Message` — sufficient for our needs

### ResponseRenderer owns notify() lifecycle

**Choice**: Place the `notify()` method on `ResponseRenderer`, called from `_process_through_coordinator` after `finalize()`.
**Why**: The renderer already manages the full message lifecycle (create, edit, split, finalize). Adding copy+delete as a post-finalize step keeps message lifecycle cohesive in one place. The renderer has direct access to `_current_message_id` — no need to expose internals.

**Consequences**:
- Pro: Message lifecycle cohesion — create/edit/split/finalize/notify all in one class
- Pro: No new abstractions — just one additional method on an existing class
- Con: Renderer's responsibility grows slightly beyond "rendering" — but it already manages deletion of status messages

### Direct disable_notification=True over state flag

**Choice**: Pass `disable_notification=True` directly on each `send_message` call during streaming, rather than using a state flag or context manager.
**Why**: The parameter is always `True` during streaming (when `push_notifications` is enabled). A state flag adds indirection for something that doesn't vary within a response. The `_push_notifications` constructor flag already controls whether to apply it.

**Consequences**:
- Pro: Explicit and readable — each call site clearly shows its notification behavior
- Pro: No hidden state that could get out of sync
- Con: Repeated parameter at multiple call sites — acceptable given there are only 3-4 send calls

### Conditional silence for error messages

**Choice**: Error messages are sent silently only when text has already been streamed (`_current_message_id is not None`). When the error is the only response, it sends with normal notification behavior.
**Why**: If an error is the sole response, it IS the notification — the user needs to know the agent responded. If text was already streamed, the copy+delete of the text message will provide the push notification; the error should stay silent to avoid double-notification.

**Consequences**:
- Pro: User always gets exactly one notification per response
- Pro: Error-only responses still trigger push (user isn't left waiting)

## System Behavior

### Scenario: Normal response with push notification

**Given**: Push notifications enabled (default), agent streams a text response
**When**: The Result event arrives
**Then**: `finalize()` sends the final edit, `notify()` copies the message (triggering push) and deletes the original. The user receives one push notification for the complete response.
**Rationale**: This is the core flow — silent streaming + post-finalize copy replaces the current behavior where the initial (incomplete) message triggers the notification.

### Scenario: Split response with push notification

**Given**: The agent streams a response that exceeds 3800 characters
**When**: The response splits into multiple messages and the Result event arrives
**Then**: All split messages are sent silently. Only the last message (`_current_message_id`) is copy+deleted. Earlier split messages remain in place unchanged.
**Rationale**: The user only needs one push notification. The last message is the one being actively edited and is the most relevant to copy.

### Scenario: User sends steering message during streaming

**Given**: The agent is streaming a response via progressive edits
**When**: The user sends a message mid-stream
**Then**: All streaming messages were sent silently. After `finalize()`, `notify()` copies the last message — the copy appears AFTER the user's steering message in the chat, preserving chronological order.
**Rationale**: Side benefit of copy+delete — the new message's position reflects when the response completed, not when it started.

### Scenario: Error-only response

**Given**: The agent returns only an Error event (no text streamed)
**When**: The renderer handles the error
**Then**: `_current_message_id` is `None`, so the error is sent with normal notification behavior (triggers push). `notify()` is a no-op because there's no message to copy.
**Rationale**: The error message itself serves as the notification.

### Scenario: Push notifications disabled

**Given**: `push_notifications = false` in config
**When**: The agent streams a response
**Then**: All `send_message` calls use default notification behavior (no `disable_notification`). `notify()` is a no-op. Behavior matches the current (pre-delta) implementation.
**Rationale**: Provides a clean escape hatch if the copy+delete mechanism causes issues.

### Scenario: Copy fails, original preserved

**Given**: The response streams normally
**When**: `bot.copy_message()` raises a `TelegramAPIError`
**Then**: The exception is caught and logged as a warning. `delete_message()` is NOT called. The original message remains — the user has the response but no push notification for it.
**Rationale**: Safe ordering ensures the user never loses a response. Missing one push notification is acceptable degradation.

### Scenario: Delete fails, duplicate visible

**Given**: The response streams normally and copy succeeds
**When**: `bot.delete_message()` raises a `TelegramAPIError`
**Then**: The exception is caught and logged as a warning. The user sees both the original and the copy (duplicate messages). Push notification was already triggered by the copy.
**Rationale**: A duplicate message is a minor cosmetic issue. The user received the notification and can read the response.

## Notes

- Telegram Bot API `copyMessage` docs: preserves all text entities (bold, italic, code, links, etc.) automatically for text messages. Caption entities on media are preserved only if no new caption is provided.
- aiogram 3.x `Bot.copy_message()` returns `MessageId` (single `.message_id` field), not a full `Message` object.
- `disable_notification=True` silences push notification sound/banner but the message still appears in the chat immediately.
- Task notifications (`_handle_notification`) are NOT affected by this delta — they are standalone messages outside the streaming flow that should always trigger normal notifications.
- `notify()` catches `TelegramAPIError` (the base class) without special-casing `TelegramRetryAfter`. A single copy+delete is not worth retry logic — if rate-limited, the original message is preserved (copy failure path) or the duplicate is acceptable (delete failure path).
