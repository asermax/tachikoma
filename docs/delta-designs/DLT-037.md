## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | Send all messages created during streaming (initial, splits, status) silently — suppress push notifications during the streaming phase | |
| **S2** | Add a post-finalize step that copies the last response message to create a new message (triggers push), then deletes the original, with safe ordering (copy first, skip delete on failure) | ⚠️ |
| **S3** | Track which message ID is the "last" message across single-message and split-message responses to know which message to copy+delete | ⚠️ |
| **S4** | Handle failures gracefully — catch exceptions from both copy and delete, log them, and do not propagate | |

### Flagged Unknowns

#### S2 — Formatting preservation through copy
`copyMessage` copies messages server-side, but it needs verification that entity-based formatting (markdown converted to Telegram entities via `telegramify_markdown`) is faithfully preserved through the copy. If formatting is lost, the design may need to fall back to delete+re-send with explicit text and parse mode.

#### S3 — Last message tracking for split responses
The current `ResponseRenderer` tracks `_current_message_id` which gets updated on each split. Need to verify that after splits, `_current_message_id` correctly points to the last (most recent) message. If it does, no additional tracking is needed.
