## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | Context update processor — a `PostProcessor` subclass that forks the SDK session, sends the update prompt, and handles the forked agent lifecycle | |
| **S2** | Context update prompt — comprehensive prompt instructing the forked agent to: read all three context files, read pending signals (via tool), analyze the conversation, classify signals as clear vs ambiguous, update files directly for clear signals, check recurrence and promote or stage for ambiguous signals | |
| **S3** | Pending signals data model and storage — structured file at `.tachikoma/pending-signals.md` with dated entries, managed programmatically by processor code | |
| **S4** | Pending signals agent tools — two tools exposed to the forked agent: read (returns full list with dates) and add (appends new entry with current date); agent cannot modify or delete entries | ⚠️ |
| **S5** | Pending signals auto-cleanup — processor-side logic that clears entries older than 1 month on every run, before the forked session starts | |
| **S6** | Prompt-driven forked processor DES — extract the shared pattern into a DES document describing: PostProcessor subclass with prompt constant, session forking via `fork_and_consume()`, agent-managed file I/O | |
| **S7** | Refactor existing processors — refactor episodic, facts, preferences processors to use the generalized abstraction from the DES pattern | |
| **S8** | Pipeline registration — register the new processor in `__main__.py` in the main phase, alongside existing memory processors | |

### Flagged Unknowns

#### S4 — Pending signals agent tools
The current `fork_and_consume()` pattern gives the forked agent standard Claude Code tools with full workspace access via `permission_mode="bypassPermissions"`. The pending signals requirement says the agent should interact with the file only via read and add tools. The likely approach is passing extra options to `fork_and_consume()` (e.g., custom tool configuration via `ClaudeAgentOptions`), but the exact mechanism needs investigation during design. The constraint is prompt-enforced and tool-design-reinforced — the tools make the intended interaction pattern clear, while the prompt instructs the agent not to access the file directly.
