## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | `LoggingSettings` pydantic model in `config.py`: `level` field (default `"INFO"`), added to `Settings` as `[logging]` section. Config auto-generation includes commented `[logging]` section. | |
| **S2** | `configure_logging()` function in new `logging.py` module: removes loguru default handler, adds file handler (`{data_path}/logs/tachikoma.log`) with rotation/retention, optionally adds stderr handler for dev. Reads level from settings. | |
| **S3** | Logging bootstrap hook: registered in `__main__.py` after workspace hook, calls `configure_logging()` with settings and workspace data path. Creates `logs/` subdirectory if missing. | |
| **S4** | Module-level bound loggers: each instrumented module creates `_log = logger.bind(component="name")` following DES-002 conventions (keyword args, no f-strings). | |
| **S5** | Instrument existing components: add log statements to `__main__.py` (startup/shutdown), `coordinator.py` (session lifecycle, message send/receive). In `config.py`, only add a post-load success log — error-path `print()` calls remain since they run before logging is configured. | |
| **S6** | `InterceptHandler`: redirect third-party stdlib logging (SDK, libraries) through loguru, following shinsenkyo's pattern. | |

### Flagged Unknowns

None — loguru is a known library with well-documented APIs, DLT-023's bootstrap system is already designed, and the mechanisms are straightforward adaptations of patterns from shinsenkyo.
