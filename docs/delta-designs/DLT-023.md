# Design: DLT-023 - Bootstrap agent workspace on first run

**Delta Spec**: [../delta-specs/DLT-023.md](../delta-specs/DLT-023.md)
**Status**: Approved

## Purpose

This document explains the design rationale for this delta: the modeling choices, data flow, system behavior, and architectural approach.

After implementation, the "Detected Impacts" section will guide reconciliation into feature design docs.

## Problem Context

The agent currently has a single ad-hoc `settings.workspace.path.mkdir(parents=True, exist_ok=True)` line in `__main__.py`. As more modules need first-run setup (git init, core context files, skills directory), this approach doesn't scale — each module would add its own ad-hoc initialization, creating ordering issues and making the startup sequence fragile.

Additionally, hooks may need to prompt the user for configuration values during first-time setup (e.g., choosing a workspace path). Those values must be persisted to the config file so they survive restarts, and subsequent hooks in the same bootstrap run must see the updated values. The current config system is read-only — it loads settings but cannot write them back.

The bootstrap needs to:
- Run a formal initialization sequence on every launch
- Let modules register idempotent setup hooks without coupling them to `__main__.py`
- Provide hooks with shared context (settings, data folder, user prompting, settings persistence)
- Ensure the agent operates from the workspace directory via the SDK's `cwd` option

**Constraints:**
- Must be idempotent — safe to run on every launch, including after partial failures
- Must fail fast with clear error naming the failing hook
- No external dependencies — pure Python patterns are sufficient
- Settings writes must preserve existing comments in the TOML file (tomlkit handles this)

## Design Overview

A `SettingsManager` class wraps the configuration system with read-write access: it loads settings on construction and provides `update(section, key, value)` + `save()` to persist changes back to the TOML file. The `.settings` property always returns the current frozen `Settings` snapshot, reloaded after each save.

A `Bootstrap` registry class owns a `BootstrapContext` and an ordered list of named hooks. Callers register hooks via `register(name, hook)` and trigger execution via `run()`. Each hook is a plain callable (`Callable[[BootstrapContext], None]`) that receives the context and self-determines whether it needs to act. Hooks access and mutate settings via `ctx.settings_manager`.

The workspace root directory and `.tachikoma/` data folder are created by a standard hook (not special-cased), making the bootstrap mechanism completely generic. After bootstrap completes, `__main__.py` reads the final settings from `settings_manager.settings` and passes the workspace path to the Coordinator as `cwd`.

```
__main__.py
    │
    ▼
SettingsManager()
    │
    ▼
Bootstrap(settings_manager, prompt=input)
    │
    ├── register("workspace", workspace_hook)
    ├── register("git", git_hook)        ← future (DLT-020)
    │
    ▼
bootstrap.run()
    │  ┌──────────────────────────────────┐
    │  │ workspace_hook(ctx)              │
    │  │  ├─ workspace dir exists? → skip │
    │  │  └─ create dir + .tachikoma/     │
    │  └──────────────────────────────────┘
    │  ┌──────────────────────────────────┐
    │  │ git_hook(ctx)                    │ ← future
    │  │  └─ ...                          │
    │  └──────────────────────────────────┘
    │
    ▼
settings = settings_manager.settings  (final snapshot)
Coordinator(cwd=settings.workspace.path, ...)
```

### Cross-Layer Contracts

```
__main__.py ──creates──▶ SettingsManager(config_path)
                              │
                              │ .settings → frozen Settings
                              │ .update(section, key, value)
                              │ .save()
                              ▼
__main__.py ──creates──▶ Bootstrap(settings_manager, prompt=input)
                              │
                              │  .register(name, hook)
                              │  .run()
                              │
                              ▼
                         BootstrapContext
                         ┌────────────────────────────────────┐
                         │ settings_manager: SettingsManager   │
                         │ prompt: Callable[[str], str]        │
                         └────────────────────────────────────┘
                              │
                              │ passed to each hook
                              ▼
                         workspace_hook(ctx) ──creates──▶ dirs
                                            ──may call──▶ ctx.settings_manager.update/save

__main__.py ──reads───▶ settings_manager.settings (final frozen snapshot)
           ──creates──▶ Coordinator(cwd=settings.workspace.path, ...)
                              │
                              ▼
                         ClaudeAgentOptions(cwd=workspace_path, ...)
```

**Integration Points:**
- `SettingsManager` ↔ `config.py`: uses existing `load_settings()` internally, adds TOML write via tomlkit
- `Bootstrap` ↔ hooks: hooks receive `BootstrapContext`, raise exceptions on failure
- `Bootstrap` ↔ `__main__.py`: `run()` raises `BootstrapError` on hook failure
- `Coordinator` ↔ SDK: `cwd` forwarded via `ClaudeAgentOptions`

### Shared Logic

- **`SettingsManager`**: Shared between `__main__.py` (construction, final read) and hooks via `BootstrapContext`. Single point of truth for current settings.
- **`BootstrapContext`**: Shared between bootstrap module and all hooks. Provides the contract that hooks depend on.

## Modeling

```
WorkspaceSettings (frozen Pydantic model)
├── path: Path (workspace root)
└── data_path: Path (computed property: self.path / ".tachikoma")

SettingsManager
├── owns config_path: Path (TOML file location, defaults to CONFIG_PATH)
├── owns _settings: Settings (frozen, reloaded after save)
├── property settings → Settings (current frozen snapshot)
├── update(section, key, value) → modifies in-memory TOML doc; raises KeyError on invalid section/key
└── save() → writes to file, reloads frozen Settings via load_settings()

Bootstrap
├── has one BootstrapContext (created at construction)
└── has ordered list of (name: str, hook: BootstrapHook) tuples

BootstrapContext (frozen dataclass)
├── settings_manager: SettingsManager (mutable — can update/save settings)
└── prompt: Callable[[str], str] (for user input)

BootstrapHook = Callable[[BootstrapContext], None]

BootstrapError
└── names the failing hook in its message
```

The `BootstrapContext` is frozen (fields can't be reassigned), but `settings_manager` is a mutable object — hooks can call `.update()` and `.save()` on it. This gives hooks write access to settings while keeping the context itself stable.

`data_path` lives on `WorkspaceSettings` as a computed property — it's available to any consumer of Settings (not just bootstrap hooks) and always reflects the current workspace path, even after settings changes during bootstrap.

## Data Flow

### Bootstrap execution (every launch)

```
1. __main__.py creates SettingsManager(config_path)
   └─ SettingsManager calls load_settings() internally
2. __main__.py creates Bootstrap(settings_manager, prompt=input)
   └─ Bootstrap builds BootstrapContext internally:
      ├─ settings_manager = settings_manager
      └─ prompt = input (or test stub)
3. __main__.py registers hooks: bootstrap.register("workspace", workspace_hook)
4. __main__.py calls bootstrap.run()
5. For each (name, hook) in registration order:
   a. Call hook(context)
   b. Hook self-determines if action needed
   c. Hook may call ctx.settings_manager.update() + .save() to persist values
   d. If hook raises → wrap in BootstrapError naming hook (chained via `from exc`), propagate
6. __main__.py reads settings_manager.settings (final snapshot after all hooks)
7. __main__.py creates Coordinator(cwd=settings.workspace.path, ...)
8. Agent operates from workspace directory via SDK cwd
```

### SettingsManager update flow

```
1. Hook calls ctx.settings_manager.update("workspace", "path", "~/custom-workspace")
2. SettingsManager reads current TOML file with tomlkit (preserves comments/formatting)
3. Updates the specified section/key in the tomlkit document
4. Stores updated doc in memory (not yet written to disk)

5. Hook calls ctx.settings_manager.save()
6. SettingsManager writes tomlkit document to config file
7. SettingsManager reloads frozen Settings via load_settings()
8. Subsequent access to .settings returns the updated snapshot
```

### Workspace hook logic

```
1. Receive BootstrapContext
2. Read workspace path from ctx.settings_manager.settings.workspace.path
3. Check if workspace path exists and is a directory
   ├─ exists and is dir → skip (idempotent)
   ├─ exists but is file → raise with clear error
   └─ doesn't exist → create with parents
4. Check if .tachikoma/ data folder exists (via settings.workspace.data_path)
   ├─ exists → skip
   └─ doesn't exist → create
5. If mkdir fails (PermissionError) → raise with clear error
```

## Key Decisions

### SettingsManager for read-write config access

**Choice**: Add a `SettingsManager` class that wraps settings loading and provides `update()`/`save()` for write-back, instead of keeping the config system read-only
**Why**: Hooks may need to persist user-provided values (e.g., a custom workspace path chosen during first-run setup). Without write-back, those values would be lost on restart. The `SettingsManager` uses tomlkit (already a dependency for default config generation) to read-modify-write the TOML file while preserving comments and formatting. The `.settings` property always returns a frozen `Settings` instance, so consumers outside of bootstrap still get the immutable guarantee.
**Alternatives Considered**:
- Read-only settings + separate config writer utility: Splits the concern but requires coordinating two objects that manage the same file
- Mutable Settings during bootstrap, freeze after: Pydantic frozen models can't be unfrozen without hacks; creating new instances on each change is complex

**Consequences**:
- Pro: Hooks can prompt for values and persist them in one flow
- Pro: Frozen `Settings` guarantee preserved for all consumers
- Pro: tomlkit preserves comments — user edits to config file are not destroyed
- Pro: Subsequent hooks see updated settings immediately after save
- Con: Adds write responsibility to the config module (previously read-only)
- Note: The `update(section, key, value)` API assumes a two-level TOML structure matching the current Settings model. Would need revision if deeper nesting is added (e.g., `workspace.git.auto_commit`)

### SettingsManager uses update() + save() (not auto-save)

**Choice**: Separate `update()` (in-memory) and `save()` (write to disk + reload) methods
**Why**: Hooks may want to batch multiple updates before writing to disk, or update without saving if they determine the change isn't needed after all. Explicit save also makes it clear when file I/O happens, which is important for error handling and testing.
**Alternatives Considered**:
- Auto-save on each update: Simpler API but causes unnecessary disk I/O for multi-field changes and makes it harder to reason about when the file changes

**Consequences**:
- Pro: Hooks control when file I/O happens
- Pro: Multiple updates can be batched before a single write
- Con: Hooks must remember to call save() — forgetting means values are lost

### Bootstrap as a registry class (not a plain function)

**Choice**: `Bootstrap` class with `register()`/`run()` methods instead of a `bootstrap(hooks, context)` function
**Why**: A registry pattern where the class owns the context internally and hooks are registered by name. This keeps context construction encapsulated and provides a natural place for hook naming (for error messages).
**Alternatives Considered**:
- Plain function taking a list of hooks: Simpler, but context must be built externally and hook names require tuples or a Protocol
- Decorator-based auto-discovery: Over-engineered for a small, explicit hook set

**Consequences**:
- Pro: Clean API — construct, register, run
- Pro: Hook names stored naturally alongside hooks
- Pro: Context construction encapsulated (callers don't build BootstrapContext)
- Con: Slightly more ceremony than a plain function

### Plain callable for hooks (not Protocol)

**Choice**: `BootstrapHook = Callable[[BootstrapContext], None]` type alias
**Why**: Any function with the right signature works. No need for hooks to implement a class — the bootstrap registry handles naming. This aligns with the project's preference for simplicity.
**Alternatives Considered**:
- Protocol with `name` property + `__call__`: Adds a `name` method to each hook, but the registry already stores names via `register(name, hook)`
- ABC base class: Too much ceremony for a callable

**Consequences**:
- Pro: Zero boilerplate — plain functions work
- Pro: Easy to test — pass any callable
- Con: No way to introspect a hook's name from the hook itself (not needed — registry has it)

### Prompt callback for user input (not direct input())

**Choice**: Hooks receive a `prompt` callable via `BootstrapContext` instead of calling `input()` directly
**Why**: Direct `input()` calls require monkeypatching `builtins.input` in tests. A callback is explicit, injectable, and testable without global state manipulation.
**Alternatives Considered**:
- Direct `input()`: Simpler interface but hard to test

**Consequences**:
- Pro: Fully testable — inject a lambda or mock
- Pro: Explicit dependency — clear from the type what hooks can do
- Con: One extra field on context

### SDK cwd option for workspace directory (not os.chdir)

**Choice**: Pass `workspace_path` to Coordinator, which forwards it as `cwd` in `ClaudeAgentOptions`, instead of calling `os.chdir()`
**Why**: `os.chdir()` is a global side effect that changes the working directory for the entire process. The SDK's `ClaudeAgentOptions.cwd` parameter (`str | Path | None`) sets the agent's working directory without affecting the host process. This is cleaner and avoids surprises for other code that depends on the process CWD.
**Sources**: Confirmed from installed SDK at `claude_agent_sdk/types.py:994` — `ClaudeAgentOptions` has `cwd: str | Path | None = None`.
**Alternatives Considered**:
- `os.chdir()` after bootstrap: Global side effect, affects entire process, harder to reason about

**Consequences**:
- Pro: No global side effects — process CWD unchanged
- Pro: SDK natively supports it — no workarounds needed
- Pro: Coordinator explicitly declares its working directory
- Con: Requires adding a `cwd` parameter to Coordinator's constructor

### Workspace hook co-located with bootstrap module

**Choice**: Define `workspace_hook` in `bootstrap.py` alongside the bootstrap mechanism
**Why**: The workspace hook is tightly coupled to the bootstrap — it's the foundational hook that creates the directories other hooks depend on. Keeping it in the same module avoids a separate file for a small function. Future hooks with more complex logic (e.g., git init) will live in their own modules.
**Alternatives Considered**:
- Separate `hooks/` package: Over-engineered for one built-in hook

**Consequences**:
- Pro: Everything related to bootstrap in one module
- Pro: No extra files for a ~15-line function
- Con: If many built-in hooks accumulate, may need extraction (unlikely — most hooks come from other deltas)

## System Behavior

### Scenario: First launch — no workspace exists

**Given**: No directory at `settings.workspace.path`
**When**: Bootstrap runs the workspace hook
**Then**: The workspace directory and `.tachikoma/` data folder are created. Subsequent hooks can use `settings.workspace.data_path` safely.
**Rationale**: The workspace hook runs first by registration order, ensuring the directory structure exists before any other hook needs it.

### Scenario: Subsequent launch — workspace already exists

**Given**: Workspace directory and `.tachikoma/` already exist
**When**: Bootstrap runs the workspace hook
**Then**: The hook detects existing directories and returns immediately with no side effects. Startup completes quickly.
**Rationale**: Each hook self-determines whether it needs to act (R3), making the full sequence safe to run every time.

### Scenario: Hook updates settings during bootstrap

**Given**: A hook prompts the user for a custom workspace path
**When**: The hook calls `ctx.settings_manager.update("workspace", "path", user_value)` and `ctx.settings_manager.save()`
**Then**: The config file is updated (preserving comments), the frozen Settings snapshot is reloaded, and subsequent hooks see the new workspace path via `ctx.settings_manager.settings`.
**Rationale**: R4.1 requires persisting user-provided values. The SettingsManager handles write-back via tomlkit, and reloading ensures the frozen snapshot is always current.

### Scenario: Partial failure recovery

**Given**: A previous launch was interrupted after workspace creation but before a later hook completed
**When**: The app is restarted
**Then**: The workspace hook skips (already done), and the incomplete hook re-runs its setup.
**Rationale**: Idempotency per-hook (R3) means each hook independently tracks its own state — no global "initialization complete" flag needed.

### Scenario: Hook failure

**Given**: A registered hook raises an exception (e.g., PermissionError creating directories)
**When**: Bootstrap's `run()` is executing
**Then**: The exception is caught and re-raised as `BootstrapError` naming the failing hook. Startup aborts immediately. No subsequent hooks run.
**Rationale**: Fail-fast (R6) prevents cascading errors. The hook name in the error message tells the user exactly what went wrong.

### Scenario: Workspace path is a file

**Given**: `settings.workspace.path` resolves to an existing regular file
**When**: The workspace hook runs
**Then**: The hook raises an error indicating the path exists but is not a directory.
**Rationale**: Clear error messaging (R9) instead of a confusing `NotADirectoryError` from `mkdir`.

### Scenario: No hooks registered

**Given**: Bootstrap is created but no hooks are registered
**When**: `run()` is called
**Then**: It completes successfully (no-op).
**Rationale**: The bootstrap mechanism is generic — zero hooks is a valid state.

### Scenario: Agent working directory

**Given**: The workspace path is configured to `~/my-workspace`
**When**: The Coordinator is created with `cwd=Path("~/my-workspace").expanduser()`
**Then**: The SDK agent operates from that directory — file tools resolve paths relative to it.
**Rationale**: R8 requires the agent to operate from the workspace. The SDK's `cwd` option achieves this without `os.chdir()`.

## Notes

- `SettingsManager` uses tomlkit for TOML document manipulation (read-modify-write with comment preservation). `load_settings()` continues to use stdlib `tomllib` for reading and is called by `SettingsManager` internally to reload the frozen `Settings` snapshot after each save
- `WorkspaceSettings.data_path` is a computed property (`self.path / ".tachikoma"`) — available to any consumer of Settings, always current even after settings changes during bootstrap
- `SettingsManager.__init__` accepts an optional `config_path` parameter (defaults to `CONFIG_PATH`), matching the existing `load_settings()` pattern for testability
- Future hooks (DLT-020 git module, DLT-005 core context files) will register themselves in `__main__.py` alongside the workspace hook
- The hook registration order in `__main__.py` serves as the explicit documentation of initialization sequence — no magic discovery
- `load_settings()` continues to exist as a standalone function for backward compatibility and for `SettingsManager` to use internally
