# Design: DLT-038 - Hot-reload skills at runtime

**Delta Spec**: [../delta-specs/DLT-038.md](../delta-specs/DLT-038.md)
**Status**: Draft

## Purpose

This document explains the design rationale for this delta: how the skill registry re-scans the filesystem on each new session so that skills created or modified at runtime become available without an application restart.

After implementation, the "Detected Impacts" section will guide reconciliation into feature design docs.

## Problem Context

The `SkillRegistry` loads skills once during `SkillsContextProvider.__init__()` and never refreshes. Since the agent can create and modify skill files during execution (e.g., via the skill authoring guide), newly authored or updated skills are invisible until the next restart. The registry needs a way to pick up filesystem changes before each detection cycle.

**Constraints:**
- The re-scan must not change the `SkillsContextProvider` or coordinator API — it's internal to the registry
- Session stability must be preserved — skills detected at session start must not change mid-session
- Error handling must remain graceful — a failed re-scan retains the previous valid state
- The mechanism must handle additions, modifications, and deletions

**Interactions:**
- `SkillsContextProvider.provide()` is the sole consumer of the registry and the natural trigger point for refresh
- The existing `is_new_session` guard in the coordinator means `provide()` only runs on the first message of each new session, so `refresh()` naturally runs only at session boundaries

## Design Overview

Minimal two-part change: the `SkillRegistry` gains a `refresh()` method that re-runs discovery on fresh dictionaries and atomically swaps the results, and the `SkillsContextProvider` calls `refresh()` at the start of `provide()` before reading any skill data.

The refresh always performs a full re-scan — no staleness detection. For a typical skills directory (< 20 skills), the filesystem I/O cost is negligible relative to the LLM classification call that follows.

```
┌─────────────────────────────────────────────────────┐
│  SkillsContextProvider.provide(message)             │
│                                                     │
│  1. registry.refresh()          ← NEW               │
│     ├─ Save old _skills, _agents references         │
│     ├─ Assign fresh empty dicts to self._skills/    │
│     │  self._agents                                 │
│     ├─ Call _discover(skills_path) → populates      │
│     │  instance dicts directly                      │
│     └─ On failure: restore old references           │
│                                                     │
│  2. Read registry.skills        (existing flow)     │
│  3. Classify via LLM            (existing flow)     │
│  4. Assemble context + agents   (existing flow)     │
└─────────────────────────────────────────────────────┘
```

## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | Add a `refresh()` method to `SkillRegistry` that saves old dict references, assigns fresh empty dicts to `self._skills` and `self._agents`, calls `_discover()` (which populates the instance dicts directly), and restores old references on failure | |
| **S2** | `refresh()` always performs a full re-scan — no change detection mechanism needed (R2 satisfied by design, see Key Decisions) | |
| **S3** | `SkillsContextProvider.provide()` calls `registry.refresh()` at the start, before reading `registry.skills` | |

## Components

### Implementation Structure

| Layer/Component | Responsibility | Key Decisions |
|-----------------|----------------|---------------|
| `src/tachikoma/skills/registry.py` | `SkillRegistry.refresh()`: re-discovers skills from filesystem | Builds fresh dicts, then swaps; reuses existing `_discover()` |
| `src/tachikoma/skills/context_provider.py` | Calls `registry.refresh()` at start of `provide()` | Single line addition before existing `if not self._registry.skills` check |

### Cross-Layer Contracts

**No contract changes.** The `SkillRegistry` public API (`skills` property, `get_agents()`, `get_agents_for_skill()`) is unchanged. The `SkillsContextProvider.provide()` signature is unchanged. The coordinator and pipeline are unaffected.

**Integration Points:**
- `SkillRegistry.refresh()` ↔ filesystem: re-reads `workspace/skills/` directory, same paths as `__init__`
- `SkillsContextProvider` ↔ `SkillRegistry`: adds one `refresh()` call before existing reads — no other changes

## Modeling

No new entities or data types. The existing `Skill` dataclass and `SkillRegistry` class are unchanged structurally. The registry gains one public method (`refresh()`) and stores the `skills_path` as an instance attribute so it can be re-used across calls.

```
SkillRegistry
├── _skills_path: Path              ← NEW (stored from __init__)
├── _agents: dict[str, AgentDefinition]
├── _skills: dict[str, Skill]
├── refresh() → None                ← NEW
├── get_agents() → dict[str, AgentDefinition]
├── get_agents_for_skill(skill_name) → dict[str, AgentDefinition]
└── skills (property) → dict[str, Skill]
```

## Data Flow

### Refresh Flow (new)

```
1. SkillsContextProvider.provide(message) called on new session
2. Provider calls registry.refresh()
3. refresh() saves references: old_skills = self._skills, old_agents = self._agents
4. refresh() assigns fresh empty dicts: self._skills = {}, self._agents = {}
5. refresh() calls _discover(self._skills_path)
   → _discover() populates self._skills and self._agents directly (existing behavior)
   ├─ Directory doesn't exist → dicts stay empty (valid state)
   ├─ Directory exists → scan subdirectories as before
   │   ├─ Valid skills → added to self._skills, agents to self._agents
   │   └─ Invalid skills → logged and skipped
   └─ Exception → restore old_skills and old_agents, log warning
6. Provider continues with existing flow (read registry.skills, classify, etc.)
```

### Deletion Handling

Deletions are handled naturally by the fresh-dict approach. Since `refresh()` assigns fresh empty dicts before calling `_discover()`, only skills that currently exist on the filesystem get added. Any previously-loaded skill whose directory was deleted simply doesn't get re-added.

## Key Decisions

### Always Re-scan Over Staleness Detection

**Choice**: Always perform a full re-scan on every `refresh()` call — no mtime-based or hash-based change detection.
**Why**: For a typical skills directory (< 20 skills), re-scanning involves ~20 `stat()` + `frontmatter.load()` calls, completing in single-digit milliseconds. The LLM classification call that follows takes seconds. The re-scan cost is negligible (< 0.1% of the total `provide()` time). Adding staleness detection would introduce complexity (mtime edge cases on Linux where directory mtime doesn't reflect nested file edits) for zero perceptible benefit.
**Alternatives Considered**:
- Top-level directory mtime: Only catches additions/deletions of skill directories, not modifications to files within them (e.g., editing `SKILL.md`). Misses a key use case (R1: modifications).
- Walk all file mtimes: Catches everything but adds complexity and its own I/O cost, approaching the cost of just re-scanning.
- Content hashing: Most expensive option, no benefit over re-scanning.

**Consequences**:
- Pro: Simplest possible implementation — no edge cases, no state to maintain
- Pro: Trivially correct — always reflects current filesystem state
- Pro: R2 is satisfied by design — "change detection" is effectively "always assume changed," which is an acceptable deviation from R2's literal wording given the negligible cost of re-scanning
- Con: Does redundant work when nothing changed — acceptable given negligible cost

### Save-and-Restore Over In-Place Mutation

**Choice**: `refresh()` saves old dict references, assigns fresh empty dicts, calls `_discover()`, and restores old references on failure — rather than clearing and rebuilding in-place.
**Why**: Since `_discover()` populates `self._skills` and `self._agents` directly (via `_load_skill` and `_load_agent`), the simplest approach is to swap to fresh empty dicts before calling it. Saving old references first means we can restore them if `_discover()` throws, preserving the previous valid state (R3). Clearing in-place would leave the registry empty or partial on failure.
**Alternatives Considered**:
- Clear and rebuild in-place: Simpler but leaves registry in broken state on failure
- Refactor `_discover()` to accept target dicts: Achieves cleaner separation but requires changing `_discover()`, `_load_skill()`, and `_load_agent()` signatures — unnecessary churn for this delta
- Copy-on-write with rollback: Unnecessarily complex for this use case

**Consequences**:
- Pro: Failure-safe — previous state preserved on error
- Pro: Zero changes to `_discover()` internals
- Con: Briefly doubles memory for skill data during refresh (trivial — skill metadata is small)

### Store skills_path as Instance Attribute

**Choice**: Store `skills_path` (derived from `workspace_path / "skills"`) as `self._skills_path` in `__init__` so `refresh()` can reuse it.
**Why**: Currently `_discover()` receives `skills_path` as a parameter computed in `__init__`. For `refresh()` to call `_discover()` again, it needs the same path. Storing it avoids recomputing and keeps `_discover()` signature unchanged.

**Consequences**:
- Pro: Minimal change — one new instance attribute
- Pro: `_discover()` remains a pure method that takes a path parameter

## System Behavior

### Scenario: New skill created during execution

**Given**: The agent creates a new skill directory with valid SKILL.md during a session
**When**: The next new session starts and `provide()` is called
**Then**: `refresh()` re-scans, discovers the new skill, swaps it into the registry. Classification includes the new skill.
**Rationale**: Core use case — skills authored at runtime become immediately available.

### Scenario: Skill modified during execution

**Given**: An existing skill's SKILL.md description is updated by the agent
**When**: The next new session starts and `provide()` is called
**Then**: `refresh()` re-scans, loads the updated description. Classification uses the new description.
**Rationale**: Updated skill metadata must be reflected for accurate classification.

### Scenario: Skill deleted during execution

**Given**: A skill directory is removed from the filesystem
**When**: The next new session starts and `provide()` is called
**Then**: `refresh()` builds fresh dicts from current filesystem — deleted skill is absent. Old skill references are replaced entirely.
**Rationale**: Fresh-dict approach handles deletions naturally without explicit removal logic.

### Scenario: Re-scan fails (permission error, corrupted directory)

**Given**: The skills directory becomes unreadable between sessions
**When**: `refresh()` calls `_discover()` and it raises an exception
**Then**: `refresh()` restores the saved old dict references, so `_skills` and `_agents` revert to their pre-refresh state. A warning is logged.
**Rationale**: Save-and-restore pattern ensures the registry degrades gracefully to its last known good state.

### Scenario: Skills directory deleted entirely

**Given**: The entire `workspace/skills/` directory is deleted
**When**: `refresh()` runs
**Then**: `_discover()` detects the directory doesn't exist, returns with empty dicts. Registry swaps to empty — no skills, no agents.
**Rationale**: The `_discover()` method already handles this case (early return when directory doesn't exist). Empty registry is a valid state.

### Scenario: No changes between sessions

**Given**: No skill files have changed since the last scan
**When**: `refresh()` runs
**Then**: Full re-scan occurs, producing identical results. New dicts replace old ones with the same content.
**Rationale**: Always re-scanning is the simplest approach; redundant work is negligible.
