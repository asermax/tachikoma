## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | Skills context provider — implements `ContextProvider`, uses forked Opus agent (low effort) to classify which skills are relevant based on registry metadata and user message | |
| **S2** | Extended pipeline return type — add optional `extras` field to `ContextResult` so providers can return structured data (agent definitions dict) alongside text content | |
| **S3** | Coordinator per-session agent tracking — store detected agents on the coordinator's session state so they persist across subsequent messages; cleared on session transition so new sessions trigger re-detection | |
| **S4** | Pipeline-to-coordinator handoff — mechanism for the coordinator to extract agent definitions from pipeline results and pass them to the per-message SDK client | |
| **S5** | Startup wiring changes — skill registry still discovers all skills at startup, but agents flow through the provider (registry → provider → coordinator) instead of directly (registry → coordinator) | |
