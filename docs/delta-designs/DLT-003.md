# Design: Skill System Foundation and Sub-Agent Delegation

<!-- This design describes the current implementation approach. Updated through delta reconciliation. -->

**Feature Spec**: [../feature-specs/agent/skills.md](../../feature-specs/agent/skills.md)
**Status**: ⧗ Design (in progress)

## Purpose

This document outlines the design approach for implementing the skill system foundation and sub-agent delegation mechanism. It establishes how skills are structured, discovered, registered, and integrated with the coordinator to enable the SDK's internal orchestrator to delegate work to specialized sub-agents.

## Problem Context

The coordinator needs a way to make specialized sub-agents available for the SDK's internal orchestrator to delegate tasks to. Currently, the coordinator only interacts with the main agent. Skills provide a structured way to package sub-agents with their definitions, making them discoverable and loadable at startup.

**Constraints:**
- Skills must be directory-based (not single files) to accommodate future expansion
- Agent definitions must be loadable from markdown files with metadata
- Agents must be discoverable and loaded at startup before conversations begin
- Invalid or missing skills/agents should not crash the system
- Agents are passed to the SDK as a dictionary and persist for the session lifetime

**Interactions:**
- Bootstrap process creates the skills directory (via skills hook)
- Skill registry discovers all skills and agents at startup
- Coordinator retrieves agents from registry and passes them to SDK
- SDK's internal orchestrator uses agents for delegation decisions (application does not control delegation)

## Design Overview

Four-layer architecture with clear separation of concerns:

```
┌─────────────────────────────────────────────────────────┐
│                 Coordinator Layer                        │
│  ┌──────────────────────────────────────────────┐        │
│  │  Coordinator                                 │        │
│  │  - Retrieves agents from registry            │        │
│  │  - Passes agents to ClaudeAgentOptions       │        │
│  └────┬─────────────────────────────────────────┘        │
├────────┼────────────────────────────────────────────────┤
│        │                                                  │
│        ▼                                                  │
│  ┌──────────────────────────────────────────────┐        │
│  │  Skill Registry                              │        │
│  │  - Discovers skills at startup               │        │
│  │  - Loads agents from each skill              │        │
│  │  - Provides agents dict to coordinator       │        │
│  └────┬─────────────────────────────────────────┘        │
├────────┼────────────────────────────────────────────────┤
│        │                                                  │
│        ▼                                                  │
│  ┌──────────────────────────────────────────────┐        │
│  │  Skills Directory Structure                  │        │
│  │  workspace/skills/                           │        │
│  │  ├── skill-name/                             │        │
│  │  │   ├── SKILL.md (metadata)                 │        │
│  │  │   └── agents/                             │        │
│  │  │       ├── agent-1.md                      │        │
│  │  │       └── agent-2.md                      │        │
│  │  └── another-skill/                          │        │
│  │      └── agents/                             │        │
│  └──────────────────────────────────────────────┘        │
├─────────────────────────────────────────────────────────┤
│              Bootstrap Layer                             │
│  ┌──────────────────────────────────────────────┐        │
│  │  Skills Hook                                 │        │
│  │  - Creates skills/ directory if missing      │        │
│  │  - Idempotent execution                      │        │
│  └──────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────┘
```

The **Skill Registry** discovers and loads all skills at startup, building a dictionary of agents indexed by namespace. The **Coordinator** retrieves this dictionary and passes it to the SDK. The **Bootstrap** creates the skills directory and registers the loading hook.

## Shape Parts

### S1: Skill Directory Structure (R0, R3)

**Location**: `workspace/skills/skill-name/`

**Files**:
- `SKILL.md` — metadata file with YAML frontmatter containing name (must match folder name), description, version, and optional fields
- `agents/` — subdirectory containing agent definitions (optional; skill can exist without agents)
- `agents/agent-name.md` — individual agent definition files (markdown with YAML frontmatter); agent name derived from filename stem (minus `.md`)

**Key Properties**:
- Skill folder name is the unique identifier and must match the `name` field in SKILL.md frontmatter
- Empty or missing descriptions cause the skill to be logged as invalid and skipped
- Each skill is self-contained and can be developed independently
- Agents are nested within the skill for organizational clarity and namespacing
- Non-.md files in `agents/` are ignored (only `.md` files are loaded)
- An agent with empty markdown body is valid (contains empty system prompt)

**Validation**:
- SKILL.md must be parseable YAML frontmatter
- SKILL.md name field must be non-empty and match the folder name
- SKILL.md description field must be non-empty
- SKILL.md with non-matching name field causes skill to be skipped (warning logged)

### S2: Agent Definition Format (R3, R4, R7)

**File**: `agents/agent-name.md` (agent-name derived from filename stem)

**Structure**:
```yaml
---
description: "What this agent does"
model: "sonnet"       # optional; valid: sonnet, opus, haiku, inherit
tools:                # optional list of tool names
  - Read
  - Glob
---

# Agent system context

The markdown body below the frontmatter becomes the agent's system prompt.
It can contain instructions, guidelines, examples, or any information
the agent needs to perform its specialized role.
```

**Key Properties**:
- YAML frontmatter contains: description (required), model (optional), tools (optional list)
- Description field must be non-empty (empty descriptions cause agent to be skipped with warning)
- Model must be one of: `sonnet`, `opus`, `haiku`, `inherit` (SDK short names, not full model IDs)
- Invalid/unsupported model values cause agent to be skipped with warning
- Markdown body (everything after `---`) becomes the agent's system prompt (passed to SDK as `prompt` field)
- Agent is uniquely identified by namespace: `skill-name/agent-name`
- Model defaults to SDK default if omitted
- Tools list is passed to AgentDefinition; SDK enforces tool scoping at invocation time
- Empty markdown body is valid (creates agent with empty system prompt)

### S3: Skill Registry (R1, R2, R9)

**Component**: `SkillRegistry` class

**Responsibilities**:
- Discover all subdirectories in `workspace/skills/` at initialization; treat each as potential skill
- Load `SKILL.md` from each directory and parse YAML frontmatter
- Validate skill metadata (name, description present and non-empty; name matches folder name)
- For each valid skill, discover agent definitions from `agents/` subdirectory (if present)
- Build and maintain agents dictionary indexed by namespace: `{skill-name/agent-name: AgentDefinition, ...}`
- Gracefully handle all error conditions: log warnings, skip invalid skills/agents, continue with valid ones
- Provide `get_agents()` method returning the agents dictionary
- Handle the case where `workspace/skills/` does not exist (return empty agents dictionary)

**Error Handling**:
- Missing `workspace/skills/` directory → return empty agents dictionary (bootstrap hook will create it)
- Empty `skills/` directory → return empty agents dictionary
- Directory with no SKILL.md → warning logged, directory skipped
- SKILL.md with unparseable YAML → warning logged, skill skipped
- SKILL.md with missing/empty name field → warning logged, skill skipped
- SKILL.md with missing/empty description field → warning logged, skill skipped
- SKILL.md with name ≠ folder name → warning logged, skill skipped
- Agent definition with unparseable YAML frontmatter → warning logged, agent skipped
- Agent with missing/empty description field → warning logged, agent skipped
- Agent with unsupported model value → warning logged, agent skipped
- All errors logged with skill/agent name and filesystem path for operator diagnostics

### S4: Coordinator Integration (R5, R6, R7)

**Changes to Coordinator Initialization**:
1. Receive `skill_registry: SkillRegistry` from bootstrap extras (pattern: created by skills bootstrap hook, passed via extras dict)
2. Call `agents_dict = skill_registry.get_agents()` during initialization
3. Pass `agents_dict` to `ClaudeAgentOptions.agents` when constructing the SDK client
4. Agents remain available for the session lifetime (SDK client property)

**Key Properties**:
- Agents are loaded at initialization, before any conversations begin
- Same agents available throughout the session (no dynamic reloading mid-session)
- SDK orchestrator has full visibility of all available agents
- AgentDefinition objects passed to SDK include: description, prompt (markdown body), model (if specified), tools (if specified)

### S5: Bootstrap Hook (R8)

**Hook**: `skills_bootstrap_hook`

**Responsibilities**:
- Execute during bootstrap sequence in registration order (after workspace hook, before logging hook)
- Check if `workspace/skills/` directory exists
- Create it if missing
- Log a message indicating creation or idempotent skip
- Idempotent: safe to run every launch

**Context and Return Value**:
- Receives `BootstrapContext` with settings_manager and workspace path
- Creates SkillRegistry instance and stores it in `bootstrap.extras['skill_registry']` for coordinator to retrieve
- Logs any errors; does not raise (bootstrap handles hook errors)

**Integration**:
- Registered in `__main__.py` during bootstrap hook registration
- Placement in sequence: after workspace hook (ensures workspace root exists), before logging hook (ensures logs configured before registry loads)
- Coordinator retrieves SkillRegistry from bootstrap extras: `skill_registry = bootstrap.extras.get('skill_registry')`
- Bootstrap hook failure (e.g., cannot create directory) causes startup to abort with error message naming the failing hook

## Implementation Strategy

1. **Phase 0: Dependencies**
   - Add `python-frontmatter` to `pyproject.toml` (for YAML frontmatter parsing)
   - Note: python-frontmatter wraps PyYAML and handles markdown + frontmatter natively

2. **Phase 1: Core Registry**
   - Create `SkillRegistry` class with directory scanning and YAML parsing using python-frontmatter
   - Implement graceful error handling: catch all parse errors, log warnings, skip invalid skills/agents, continue with partial results
   - Build agents dictionary indexed by `skill-name/agent-name` namespace
   - Handle edge cases: missing skills/ directory, empty skills/ directory, skills without agents/ subdirectory, agents with empty markdown bodies

3. **Phase 2: Bootstrap Hook**
   - Create `skills_bootstrap_hook` function in skills subsystem module
   - Instantiate SkillRegistry and store in `bootstrap.extras['skill_registry']`
   - Create `workspace/skills/` directory if missing
   - Implement idempotency: check existence before creating, log appropriately
   - Register in `__main__.py` bootstrap hook sequence (after workspace, before logging)

4. **Phase 3: Coordinator Integration**
   - Modify Coordinator initialization to retrieve SkillRegistry from bootstrap extras
   - Call `get_agents()` and pass result to `ClaudeAgentOptions.agents`
   - Ensure AgentDefinition objects are correctly constructed with description, prompt, model, tools fields

5. **Phase 4: Testing & Validation**
   - Unit tests for SkillRegistry: discovery, parsing, error handling, edge cases
   - Integration tests for coordinator integration with agents passed to SDK
   - Test error scenarios: malformed SKILL.md, malformed agent markdown, missing fields, unsupported model values
   - Test edge cases: missing skills/ directory, empty skills/ directory, skill with no agents, agent with empty body

## Data Flow

### Startup Sequence

```
1. __main__.py runs
2. Bootstrap executes skills_hook
   → Creates workspace/skills/ if missing
3. SkillRegistry initializes
   → Scans workspace/skills/
   → Loads all SKILL.md files
   → Discovers and loads all agents/
   → Builds agents dictionary
4. Coordinator initializes
   → Requests agents from registry
   → Creates ClaudeAgentOptions with agents
   → Connects SDK client (agents now available)
5. Coordinator ready for messages
   → SDK orchestrator can delegate to any loaded agent
```

### Agent Discovery Process

```
For each directory in workspace/skills/:
  1. Check if SKILL.md exists
  2. Parse SKILL.md YAML frontmatter
  3. Validate skill metadata (name, description)
  4. Create Skill object or log warning if invalid
  5. For each .md file in skill/agents/:
     a. Parse agent markdown with YAML frontmatter
     b. Extract: description, model, tools, markdown body
     c. Create AgentDefinition with namespace "skill-name/agent-name"
     d. Add to agents dictionary or log warning if invalid
6. Return agents dictionary to coordinator
```

## Key Decisions

### Directory-based Skills over Single Files

**Choice**: Skills are directories (`skills/skill-name/`) containing SKILL.md and agents/ subdirectory, not single files.

**Why**: Directories allow for future expansion (instructions, resources, configurations) without breaking the structure. Single files would be brittle and preclude adding skill-level components later.

**Consequences**:
- Pro: Extensible foundation for future skill components
- Pro: Clear organizational hierarchy
- Con: More filesystem operations needed

### YAML Frontmatter for Metadata

**Choice**: Skill and agent metadata is embedded in markdown files using YAML frontmatter (Jekyll/Hugo/static site generator convention).

**Why**: Markdown is human-readable, and YAML frontmatter is a widely-adopted convention in static site generators, documentation tools, and publishing platforms. Metadata stays with the file it describes, making skills self-contained and portable.

**Alternatives Considered**:
- Separate JSON/YAML files: Decoupled but requires more files per skill
- Database/registry file: Centralized but less portable and requires external state management
- Python files with dataclasses: Programmatic but less accessible to non-developers

**Consequences**:
- Pro: Self-contained metadata with markdown body
- Pro: Human-friendly format (no special tooling to edit)
- Pro: Portable (entire skill is just a directory with markdown files)
- Con: Parsing overhead on every startup (mitigated by startup-only discovery)

### Agents Passed to SDK at Initialization

**Choice**: All agents are discovered and passed to the SDK at coordinator initialization via `ClaudeAgentOptions.agents`.

**Why**: The SDK's orchestrator requires agents to be known upfront for delegation decisions. Agents must be available for the entire session.

**Alternatives Considered**:
- Dynamic agent loading mid-session: Complex, SDK doesn't support mid-session agent updates
- Application-controlled delegation: Tighter coupling, duplicates SDK functionality

**Consequences**:
- Pro: Single load at startup, simple lifecycle
- Pro: Aligns with SDK's design
- Con: Cannot add agents during conversation

### Graceful Error Handling

**Choice**: Invalid skills/agents are logged as warnings; registry continues loading other skills.

**Why**: A single malformed skill file should not crash the entire system. Partial functionality is better than complete failure.

**Consequences**:
- Pro: System resilience
- Pro: Operator sees what went wrong (diagnostic logging)
- Con: Silent skipping could hide typos (mitigated by explicit warning logs)

## Components

| Component | Type | Responsibility | Lifecycle |
|-----------|------|-----------------|-----------|
| **SkillRegistry** | Class | Discover and load all skills/agents at startup; provide agents dictionary | Singleton; initialized at coordinator startup; persists for session |
| **Skill** | Data Structure | Container for skill metadata (name, description, version); holds reference to agents | Created during registry discovery; discarded after agent extraction |
| **AgentDefinition** | Data Transfer | SDK-compatible agent definition with description, model, tools, and system prompt | Created from parsed markdown; passed to SDK; immutable during session |
| **Skills Hook** | Bootstrap Callback | Create `workspace/skills/` directory if missing | Called during bootstrap sequence; idempotent |
| **Skills Directory** | Filesystem | Root directory for all skill packages; discoverable by registry | Must exist before coordinator initialization |
| **Skill Directory** | Filesystem | Single skill package; contains SKILL.md and agents/ subdirectory | User-created; discovered by registry |
| **Agent Markdown File** | Filesystem | Single agent definition with YAML frontmatter and system context | User-created within skill; parsed and loaded by registry |

## Modeling

### Agent Definition Transformation

```
Agent Markdown File
  ↓ [Parse YAML frontmatter + markdown body]
  ↓ [Extract: description, model, tools, markdown body]
  ↓ [Validate: description non-empty, model in {sonnet,opus,haiku,inherit} if specified]
AgentDefinition (SDK type)
  - description: str
  - prompt: str (from markdown body)
  - model: Literal["sonnet", "opus", "haiku", "inherit"] | None
  - tools: list[str] | None
  ↓ [Passed to ClaudeAgentOptions.agents]
ClaudeAgentOptions
  ↓ [Agents passed to SDK at initialization]
SDK Orchestrator
  ↓ [Uses agents dict for delegation decisions during conversation]
```

### Skill Discovery Process (State Machine)

```
[No Skills Loaded]
  ↓ scan(workspace/skills/)
[Scanning Directories]
  ├─→ SKILL.md exists? ──no──→ [Skip Directory]
  │   ├─→ SKILL.md valid? ──no──→ [Log Warning, Skip]
  │   └─→ SKILL.md valid? ──yes─→ [Load Skill Metadata]
  │                                 ↓
  │                         [Load agents/ subdirectory]
  │                           ├─→ agents/ exists? ──no──→ [Valid (no agents)]
  │                           └─→ agents/ exists? ──yes─→ [Parse agent files]
  │                                                        ↓
  │                                                [For each .md file]
  │                                                  ├─→ Valid? ──no──→ [Log Warning, Skip]
  │                                                  └─→ Valid? ──yes─→ [Add to agents dict]
  └─→ [All directories processed]
      ↓
[Agents Dictionary Complete]
  ↓ [Returned to coordinator]
[Coordinator -> SDK]
```

### Error Handling Flow

```
Parse Error (invalid YAML, missing fields)
  ↓
[Catch exception]
  ↓
[Log warning with: skill/agent name, filepath, error details]
  ↓
[Continue processing other skills/agents]
  ↓
[Return available agents; system remains operational with partial skill set]
```

## System Behavior

### Invariants

1. **Agent Uniqueness by Namespace**: Each agent in the agents dictionary has a unique namespace key (skill-name/agent-name). Because skill names are folder names (unique by filesystem constraint) and agent names are filename stems (unique within a skill by filesystem constraint), namespace collisions cannot occur in normal operation. If a duplicate agent is somehow present within a single skill, the last one loaded overwrites the previous (logged as warning).

2. **Session Stability**: Once agents are passed to the SDK at initialization, the set of available agents does not change for the session duration. New skills added to the filesystem during runtime are not discovered until the next application restart.

3. **Graceful Degradation**: Invalid skills or agents do not cause the system to fail. Registry returns whatever agents it successfully loaded; coordinator proceeds with that partial set. Empty registry (no skills or all invalid) is a valid state.

4. **Metadata Integrity**: An AgentDefinition passed to the SDK contains exactly the metadata extracted from the agent markdown file. No transformation, inference, or injection occurs (except markdown parsing for the prompt field).

### Initialization Protocol

```
1. Application starts
2. Bootstrap sequence executes:
   a. Skills hook runs → creates workspace/skills/ if missing
3. Coordinator initializes:
   a. Creates SkillRegistry instance
   b. Calls registry.get_agents() → blocks until discovery complete
   c. Receives agents dictionary (possibly partial if errors occurred)
   d. Creates ClaudeAgentOptions with agents parameter
   e. Initializes SDK client with options
4. Session begins; agents available for orchestrator delegation
5. If registry encountered errors during startup:
   - Coordinator logs summary of what loaded successfully
   - Available agents provided to SDK (subset of what could have loaded)
   - Coordinator continues normally
```

### Skill Discovery Protocol

The registry scans the skills directory exactly once, at initialization:

```
Input: workspace_path (resolved to workspace/skills/)
Output: agents dictionary {namespace: AgentDefinition, ...}

1. List all directories in workspace/skills/
2. For each directory (in filesystem order):
   a. Check for SKILL.md
      - Not found: log skip (skill directory but no metadata)
      - Found: parse YAML frontmatter
   b. Validate skill metadata (name, description present)
      - Invalid: log warning, skip this skill
      - Valid: proceed to agent discovery
   c. Check for agents/ subdirectory
      - Not found: skill valid but has no agents; continue
      - Found: list all .md files in agents/
   d. For each agent .md file:
      - Parse markdown with YAML frontmatter
      - Extract: description, model, tools, markdown body
      - Validate: description required, model must be sonnet|opus|haiku|inherit if specified
      - Create AgentDefinition with namespace "skill-name/agent-name"
      - Add to agents dictionary
      - If invalid: log warning, skip this agent
3. Return complete agents dictionary to caller
```

## Detected Impacts

### Critical External Dependencies

- **python-frontmatter library**: This delta REQUIRES adding `python-frontmatter` to `pyproject.toml`. Without it, YAML frontmatter parsing will fail at startup and the entire skill system will be non-functional. Alternative: manually parse frontmatter with PyYAML (already a transitive dependency) if the team prefers to avoid adding a new dependency, but this increases implementation complexity.

- **Bootstrap System (DES-003)**: Skills hook plugs into the existing bootstrap hook sequence. The pattern (create shared objects in hooks, pass via extras dict) is established by DES-003. If bootstrap system is unavailable or the extras dict mechanism is removed, integration fails.

- **Coordinator Structure**: Coordinator must retrieve SkillRegistry from bootstrap extras and pass agents to `ClaudeAgentOptions`. Current coordinator implementation needs modification to accept the agents parameter.

- **SDK Type Compatibility**: Requires Claude Agent SDK types `AgentDefinition` and `ClaudeAgentOptions.agents`. If SDK API changes (e.g., agents field renamed or removed), compatibility breaks.

### Side Effects & Ripple Points

- **DLT-021 (Skill Detection & Injection)**: Will layer a skills context provider on top of this delta. Depends on SkillRegistry being functional and accessible (likely via bootstrap extras or singleton pattern). Currently DLT-021 will need to extend the registry to retain skill metadata (currently discarded after agent extraction). No breaking changes to DLT-003 design, but DLT-021 may require extending SkillRegistry interface.

- **Skill Package Format**: Once adopted, changing from YAML frontmatter + markdown to a different format (e.g., JSON, TOML) will require migration tooling for existing skill packages and SkillRegistry rewrite.

- **Agent Namespacing Convention**: The "skill-name/agent-name" format is visible to DLT-021, future skill detection logic, and users creating skills. Changing the separator or format breaks those layers and requires skill package migration.

- **Bootstrap Hook Ordering**: The skills hook placement (after workspace, before logging) is critical. Moving it breaks dependency order (logging must be configured before registry logging occurs) and may cause bootstrap failures.

- **Error Visibility**: Logging warnings for invalid skills means operators must monitor logs. No UI or dashboard feedback if a skill fails to load. Operators might not notice degraded skill availability.

- **Coordinator Dependency**: The coordinator now depends on SkillRegistry for initialization. If any coordinator code path tries to run before bootstrap completes, runtime errors occur.

### Known Limitations

- **No Dynamic Reloading**: Skills cannot be added/removed mid-session. Operator must restart application to pick up new skills or changes to skill definitions.

- **No Skill Validation Schema**: No JSON Schema or validation spec exists for SKILL.md or agent markdown formats. Validation is implemented in code (loose/permissive). This makes skill creation error-prone without tooling.

- **No Skill Dependencies**: Skills cannot depend on each other or declare required tools/capabilities. Each skill is treated independently. Complex skills that would benefit from composition cannot express that relationship.

- **Operator Visibility**: No CLI tool or dashboard to inspect loaded agents, validate agent definitions, or test agent invocations without running a full conversation.

- **Tool Scoping Advisory Only**: Agent definitions specify tools in frontmatter, but the SDK enforces tool scoping at invocation time. The application has no way to validate tool access before delegating to the SDK. An agent that specifies tools but tries to use a tool not in its list will fail at SDK invocation time.

- **No Skill Metadata Persistence**: SkillRegistry discards Skill objects after extracting agents. Future features (DLT-021) that need skill metadata will require either re-parsing SKILL.md files or extending the registry to retain metadata in memory.

## Integration Points

- **Coordinator**: Accepts SkillRegistry, passes agents to SDK
- **Bootstrap**: Skills hook creates skills/ directory in registration order (after workspace, before logging)
- **SDK**: Receives agents dict via `ClaudeAgentOptions.agents`, manages delegation logic
- **Future DLT-021**: Skills context provider will layer on top for automatic detection/injection; depends on SkillRegistry being available

## Notes

- The SDK's orchestrator makes all delegation decisions based on agent descriptions and conversation context. This delta provides the infrastructure; the intelligence comes later.
- Agent scoping (tools field) is advisory to the SDK. The SDK enforces tool access at invocation time.
- The design deliberately keeps DLT-003 (infrastructure) separate from DLT-021 (intelligence), allowing each to evolve independently.
