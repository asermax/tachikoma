# Design: DLT-034 - Summarize agent actions instead of generic tool markers

**Delta Spec**: [../delta-specs/DLT-034.md](../delta-specs/DLT-034.md)
**Status**: Approved

## Purpose

This document explains the design rationale for this delta: the modeling choices, data flow, system behavior, and architectural approach.

After implementation, the "Detected Impacts" section will guide reconciliation into feature design docs.

## Problem Context

The Telegram channel's `ResponseRenderer` currently inserts a static `_🔧 Ran tools_` marker at each tool→text transition. This tells the user that tools ran but not *what* they did, forcing them to either ignore the marker or mentally reconstruct what happened from surrounding context.

**Constraints:**
- The summary must fit on a single line and follow the existing visual style (`_🔧 <text>_` — italic with emoji prefix)
- The `ToolActivity` dataclass carries `tool_name` and `tool_input` (a dict) — all detail extraction must come from these fields
- The summary generator must be channel-agnostic (in `display.py`) so other channels can adopt it later
- Tool types and their input schemas are defined by the Claude Agent SDK and may expand over time — unknown tools must degrade gracefully

**Interactions:**
- `ResponseRenderer` in `telegram.py`: owns the tool→text transition logic that triggers summary insertion
- `display.py`: already contains `TOOL_DISPLAY` (live tool status line formatting); the summary generator joins it here
- `ToolActivity` in `events.py`: the domain event carrying tool name and input data

## Design Overview

A **rule-based summary generator** function in `display.py` takes a list of `ToolActivity` objects collected between two text boundaries and produces a compact, human-readable action description. The `ResponseRenderer` accumulates activities in a list, and at each tool→text transition calls the generator instead of inserting static text.

```
┌─────────────────────────────────────────────────────────┐
│  ResponseRenderer (telegram.py)                         │
│                                                         │
│  handle_tool(activity)                                  │
│    → append to _tool_activities list                    │
│                                                         │
│  handle_text(chunk)                                     │
│    → if _tool_activities non-empty:                     │
│        summary = summarize_tool_activity(activities)    │
│        insert "_🔧 {summary}_\n" into buffer            │
│        clear _tool_activities                           │
│    → append chunk to buffer                             │
│                                                         │
│  finalize()                                             │
│    → clear _tool_activities (shutdown safety)            │
│                                                         │
│  reset()                                                │
│    → clear _tool_activities                             │
└────────────────────────┬────────────────────────────────┘
                         │ calls
                         ▼
┌─────────────────────────────────────────────────────────┐
│  display.py                                             │
│                                                         │
│  TOOL_DISPLAY    (existing — live status line format)   │
│  TOOL_SUMMARY    (new — summary verb-phrase format)     │
│  summarize_tool_activity(activities) → str              │
└─────────────────────────────────────────────────────────┘
```

## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | Tool activity collector — add a `_tool_activities: list[ToolActivity]` to `ResponseRenderer` that accumulates activities per tool→text boundary. `handle_tool()` appends to the list. `handle_text()` passes the list to the summary generator and resets it after each summary insertion. `reset()` clears the list. | |
| **S2** | Summary formatter map — a `TOOL_SUMMARY` dict in `display.py` mapping tool names to lambdas that produce compact verb-phrase strings from `tool_input` (e.g., `"Read": lambda inp: f"read {basename}"`, `"Bash": lambda inp: description or truncated command`). Separate from `TOOL_DISPLAY` (live status) because the formats differ (imperative vs. progressive, no ellipsis). Unknown tools fall back to `"used {tool_name}"`. Malformed input falls back to default placeholders (e.g., "read a file"). | |
| **S3** | Summary generator function — `summarize_tool_activity(activities: list[ToolActivity]) -> str` in `display.py`. Groups activities by tool type, applies per-tool formatting via `TOOL_SUMMARY`, uses adaptive aggregation (>2 same type → count phrase like "read 3 files"; ≤2 → list details individually), joins phrases by commas with "and" before the last (2 items: "A and B"; 3+: "A, B, and C"), capitalizes the first word of the final string, caps at 5 groups with trailing "and more". Returns empty string for empty input. | |
| **S4** | Renderer integration — in `handle_text()`, replace the hardcoded `"_🔧 Ran tools_\n"` with `f"_🔧 {summarize_tool_activity(self._tool_activities)}_\n"`. Remove `_had_tools` and `_tools_marker_inserted` flags, replace with `_tool_activities` list emptiness check. `handle_tool()` only appends to the list and sets the live tool line — no flag management. `finalize()` clears `_tool_activities` (needed for the shutdown path which calls `finalize()` without `reset()`). | |

### Flagged Unknowns

None — all mechanisms are well-understood.

## Components

### Implementation Structure

| Layer/Component | Responsibility | Key Decisions |
|-----------------|----------------|---------------|
| `src/tachikoma/display.py` | `TOOL_SUMMARY` map + `summarize_tool_activity()` function. Channel-agnostic summary generation from `ToolActivity` lists | Separate map from `TOOL_DISPLAY`; function returns plain string (no markdown wrapping — caller adds formatting) |
| `src/tachikoma/telegram.py` | `ResponseRenderer` collects `ToolActivity` objects in `_tool_activities`, calls `summarize_tool_activity()` at tool→text transitions, inserts formatted summary into buffer | Replaces `_had_tools` / `_tools_marker_inserted` flags with list emptiness check |

### Cross-Layer Contracts

**Summary generator contract:**

```
summarize_tool_activity(activities: list[ToolActivity]) -> str
```

- Input: list of `ToolActivity` objects (may be empty)
- Output: plain text summary string (no markdown wrapping), or empty string if input is empty
- The caller is responsible for wrapping in italic markdown and emoji prefix

**Per-tool formatter contract (TOOL_SUMMARY entries):**

Each entry is a callable `(tool_input: dict) -> str` that returns a lowercase verb-phrase fragment. If `tool_input` is missing expected keys, the formatter returns a generic fallback.

### Shared Logic

- **`TOOL_SUMMARY` map**: shared between the summary generator and potentially future channels. Lives in `display.py` alongside `TOOL_DISPLAY` for discoverability.
- **`summarize_tool_activity()` function**: the single entry point for any channel to generate a tool summary. The Telegram renderer calls it; the REPL could adopt it later (R7 is currently Out).

## Modeling

No new entities or models are introduced. The design operates entirely on the existing `ToolActivity` dataclass:

```
ToolActivity
├── tool_name: str          (e.g., "Read", "Grep", "Bash")
├── tool_input: dict        (tool-specific parameters)
└── result: str             (not used by summary generation)
```

The `TOOL_SUMMARY` map defines per-tool formatting rules:

```
TOOL_SUMMARY: dict[str, Callable[[dict], str]]
├── "Read"       → "read {basename(file_path)}" | "read a file"
├── "Grep"       → "searched for '{pattern}'" | "searched for a pattern"
├── "Glob"       → "globbed '{pattern}'" | "globbed a pattern"
├── "Bash"       → "{description}" | "{command[:40]}..." | "ran a command"
├── "Edit"       → "edited {basename(file_path)}" | "edited a file"
├── "Write"      → "wrote {basename(file_path)}" | "wrote a file"
├── "ToolSearch"  → "searched tools" (no variable detail needed)
└── (unknown)    → "used {tool_name}"
```

**Bash formatting priority:**
1. `tool_input["description"]` if present — first character lowercased for sentence flow (e.g., "Run tests" → "run tests"), preserving casing of the rest (proper nouns, paths)
2. `tool_input["command"]` truncated to 40 chars with "..." if longer
3. "ran a command" as final fallback

**Read file_path formatting:** extracts basename from the path for brevity (e.g., `src/tachikoma/display.py` → `display.py`). When aggregated (>2), uses count ("read 3 files").

## Data Flow

### Summary generation at tool→text transition

```
1. Agent uses tools → SDK emits ToolUseBlock messages
2. Adapter converts to ToolActivity events
3. ResponseRenderer.handle_tool() receives each ToolActivity:
   a. Appends activity to _tool_activities list
   b. Sets _tool_line for live status display (existing behavior, unchanged)
   c. Flushes (throttled edit showing live tool line)

4. Agent emits text → SDK emits TextBlock
5. Adapter converts to TextChunk event
6. ResponseRenderer.handle_text() receives the chunk:
   a. Checks: is _tool_activities non-empty?
   b. YES → tool→text transition detected:
      i.   summary = summarize_tool_activity(_tool_activities)
      ii.  Appends "_🔧 {summary}_\n" to _buffer
      iii. Clears _tool_activities list
      iv.  Clears _tool_line (marker replaces it)
   c. Appends chunk text to _buffer
   d. Flushes (throttled edit)
```

### summarize_tool_activity() internal flow

```
1. Receive list of ToolActivity objects
2. Return "" if list is empty
3. Group activities by tool_name (preserving first-seen order)
4. For each group:
   a. Count = number of activities in group
   b. If count > 2 → use aggregated form (e.g., "read 3 files")
   c. If count ≤ 2 → format each individually via TOOL_SUMMARY, collect phrases
5. If more than 5 tool-type groups → keep first 5 groups, append "and more"
6. Flatten all phrases from kept groups into a single ordered list
7. Join phrases: 1 item → as-is; 2 items → "A and B"; 3+ items → "A, B, and C" (Oxford comma)
8. Capitalize the first character of the joined string
9. Return the capitalized string
```

### Aggregated form generation

For tools with count > 2, the aggregated phrasing depends on the tool type:

```
"Read"       → "read {count} files"
"Grep"       → "ran {count} searches"
"Glob"       → "ran {count} glob searches"
"Bash"       → "ran {count} commands"
"Edit"       → "edited {count} files"
"Write"      → "wrote {count} files"
"ToolSearch"  → "ran {count} tool searches"
(unknown)    → "used {tool_name} {count} times"
```

## Key Decisions

### Separate TOOL_SUMMARY map over parameterized TOOL_DISPLAY

**Choice**: Add a new `TOOL_SUMMARY` dict in `display.py` rather than parameterizing `TOOL_DISPLAY` to support both modes.
**Why**: The two formats serve fundamentally different purposes. `TOOL_DISPLAY` produces present-progressive labels for live status lines ("Reading src/main.py..."), while `TOOL_SUMMARY` produces past/imperative verb phrases for post-hoc summaries ("read main.py"). The field extraction differs too — summaries use basenames for brevity, status lines use full paths for specificity.
**Alternatives Considered**:
- Parameterized `TOOL_DISPLAY` with mode argument: Adds complexity to every existing lambda, couples two independent concerns, and makes each formatter harder to read for marginal DRY benefit.

**Consequences**:
- Pro: Each map is simple, single-purpose, and independently evolvable
- Pro: Adding a new tool type means adding one entry to each map — straightforward
- Con: Two places to update when a new tool type is added — but the formats differ enough that this is intentional, not duplicated

### Summary returns plain text (caller wraps)

**Choice**: `summarize_tool_activity()` returns a plain text string; the caller adds italic markdown wrapping and emoji prefix.
**Why**: Channel-agnostic output. The Telegram renderer wraps as `_🔧 {summary}_` (Telegram markdown italic), but another channel might use different formatting (ANSI colors, HTML, plain text).

**Consequences**:
- Pro: Function is testable with plain string assertions
- Pro: Reusable across channels with different formatting conventions
- Con: Caller must remember to wrap — but there's only one call site today

### Adaptive aggregation threshold at >2

**Choice**: Aggregate by count when more than 2 tools of the same type run; list individually for 1-2.
**Why**: With 1-2 tools, the specific details are more useful than a count ("read main.py and config.py" > "read 2 files"). At 3+, listing each detail becomes noisy and the count is more helpful.

**Consequences**:
- Pro: Short sequences stay detailed, long sequences stay concise
- Pro: Matches the spec's R5 requirement exactly

### Bash description field preference

**Choice**: For Bash tools, prefer `tool_input["description"]` (first character lowercased) over `tool_input["command"]`.
**Why**: The description field is a human-written summary of what the command does (e.g., "Run tests", "Install dependencies"). It's more meaningful than a raw shell command that may be long or opaque. Only the first character is lowercased to preserve proper nouns and paths in the rest of the string.
**Fallback chain**: description → truncated command (40 chars) → "ran a command"

**Consequences**:
- Pro: Summaries read naturally ("run tests" vs. "ran: pytest tests/test_...")
- Pro: Graceful degradation when description is absent
- Pro: Preserves proper nouns and paths within descriptions

## System Behavior

### Scenario: Single tool before text

**Given**: The agent reads one file, then emits text
**When**: `handle_text()` is called and `_tool_activities` has one entry (Read of `src/main.py`)
**Then**: Summary is "Read main.py", marker inserted as `_🔧 Read main.py_`
**Rationale**: Single tool with detail is more informative than a count

### Scenario: Multiple tools of the same type

**Given**: The agent reads 4 files, then emits text
**When**: `handle_text()` is called with 4 Read activities
**Then**: Summary is "Read 4 files" (aggregated because count > 2)
**Rationale**: Adaptive aggregation keeps the summary concise

### Scenario: Mixed tool types

**Given**: The agent reads 3 files and runs 1 grep, then emits text
**When**: `handle_text()` is called with 3 Read + 1 Grep activities
**Then**: Summary is "Read 3 files and searched for 'pattern'"
**Rationale**: Read is aggregated (>2), Grep is listed individually (≤2)

### Scenario: Two tools of the same type listed individually

**Given**: The agent reads `src/a.py` and `src/b.py`, then emits text
**When**: `handle_text()` is called with 2 Read activities
**Then**: Summary is "Read a.py and read b.py"
**Rationale**: ≤2 threshold means details are shown individually

### Scenario: More than 5 tool groups

**Given**: The agent uses Read, Grep, Glob, Bash, Edit, Write, and ToolSearch before text
**When**: `handle_text()` is called with 7 distinct tool types
**Then**: Summary shows first 5 groups joined by commas with "and more" at the end (e.g., "Read main.py, searched for 'pattern', globbed '\*\*/\*.py', ran a command, edited file.py, and more")
**Rationale**: Cap prevents excessively long summaries

### Scenario: Unknown tool type

**Given**: The agent uses a tool called "CustomMCP" not in `TOOL_SUMMARY`
**When**: The summary generator processes it
**Then**: Falls back to "used CustomMCP"
**Rationale**: Graceful degradation for SDK tool types added in the future

### Scenario: Malformed tool input

**Given**: A Read activity arrives with no `file_path` key in `tool_input`
**When**: The summary formatter for Read is called
**Then**: Falls back to "Read a file" (generic placeholder, capitalized as first word)
**Rationale**: Defensive — `tool_input` structure is SDK-defined and may vary

### Scenario: Multiple tool→text transitions in one response

**Given**: Agent uses tools, emits text, uses more tools, emits more text
**When**: Each tool→text transition occurs
**Then**: Each transition generates its own independent summary from only the tools in that segment. The `_tool_activities` list is cleared after each summary insertion.
**Rationale**: Each summary reflects only its corresponding tool segment

### Scenario: Tools run but no text follows (response ends with tools)

**Given**: The agent runs tools but then the Result event arrives without intervening text
**When**: `finalize()` is called
**Then**: The tool line is cleared and `_tool_activities` is cleared (existing behavior preserved — no summary marker inserted). The `_tool_activities` clear in `finalize()` is important for the shutdown path (`_on_shutdown`) which calls `finalize()` without a subsequent `reset()`.
**Rationale**: A trailing summary with no following text would look odd; the live tool line already communicated what was happening

### Scenario: Empty tool activity list

**Given**: `summarize_tool_activity()` is called with an empty list
**When**: The function executes
**Then**: Returns empty string, no marker inserted
**Rationale**: Defensive guard — should not happen in normal flow but prevents edge case issues

## Notes

- The `TOOL_SUMMARY` map entries mirror the tool types currently handled by the Claude Agent SDK adapter (`adapter.py`): Read, Grep, Glob, Bash, ToolSearch — plus Edit and Write which the SDK also supports
- The summary generator is a pure function with no side effects — easy to unit test with constructed `ToolActivity` lists
- Group ordering preserves first-seen order (insertion order of a dict), so the summary reads in the order tools actually ran
- The 5-group cap is a soft limit — it applies to the number of distinct tool-type groups, not individual activities. A response using Read (×10) and Grep (×5) produces only 2 groups.
