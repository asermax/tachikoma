# Design: DLT-025 - Display formatted markdown in REPL responses

**Delta Spec**: [../delta-specs/DLT-025.md](../delta-specs/DLT-025.md)
**Status**: Approved

## Purpose

This document explains the design rationale for this delta: how the Renderer is migrated from raw `print()` + ANSI escape codes to `rich`-based styled and markdown-aware output.

After implementation, the "Detected Impacts" section will guide reconciliation into feature design docs.

## Problem Context

The coordinator agent produces markdown-formatted text (headings, bold, code blocks, lists), but the REPL's `Renderer` prints it as raw syntax via `print(text, end="", flush=True)`. Tool activity lines use manual ANSI escape codes (`\033[3;90m`) for styling, making rendering fragile and inconsistent.

**Constraints:**
- The change is scoped to the `Renderer` class in `repl.py` — the event system, adapter, and coordinator are unaffected
- TextChunk events correspond to complete SDK TextBlocks (not streaming tokens), so markdown syntax is never split across chunks
- The REPL is a developer tool — formatting should be pleasant but doesn't need to be pixel-perfect

## Design Overview

Replace all manual terminal output in `Renderer` with `rich.console.Console`. Text chunks are rendered as `rich.markdown.Markdown` objects (block rendering), tool activity uses `Console.print()` with style strings, and errors go through a separate stderr Console. The `_needs_newline` line-state tracking is removed entirely since block rendering guarantees each output ends with a newline.

## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | Add `rich` as a project dependency in `pyproject.toml` | |
| **S2** | Accept optional `Console` instances via `Renderer.__init__` (defaults to stdout/stderr consoles); reuse avoids terminal-detection overhead and enables test injection | |
| **S3** | Render `TextChunk.text` via `Console.print(Markdown(text, code_theme="dracula"))` — each chunk is a complete block | |
| **S4** | Render `ToolActivity` via `Console.print(label, style="dim italic grey50")` using the same Console | |
| **S5** | Remove `TOOL_STYLE`/`RESET` ANSI constants and `_needs_newline` state — block rendering makes line-state tracking unnecessary | |
| **S6** | Render errors via `Console(stderr=True).print(...)` with `style="bold red"` | |
| **S7** | Change prompt string from `"you> "` to `"> "` | |

## Components

### Implementation Structure

| Layer/Component | Responsibility | Key Decisions |
|-----------------|----------------|---------------|
| `Renderer` | Event rendering via rich Console | Owns two Console instances (stdout, stderr). No line-state tracking. |
| `Repl` | Input loop, control flow | Only change: prompt string `"you> "` → `"> "` |

### Event Rendering (updated)

| Event Type | Rendering | Notes |
|------------|-----------|-------|
| `TextChunk` | `console.print(Markdown(text, code_theme="dracula"))` | Block rendering — `Markdown` constructed with `code_theme`, then printed via Console |
| `ToolActivity` | `console.print(label, style="dim italic grey50")` | Same display format as before (tool-specific labels), styled via rich instead of ANSI |
| `Result` | `console.print()` | Empty print for blank line separator before next prompt |
| `Error` | `err_console.print(f"Error: {message}", style="bold red")` | Printed to stderr via dedicated Console |

### What's Removed

- `TOOL_STYLE` and `RESET` ANSI escape code constants
- `_needs_newline` boolean state and all associated conditional logic
- All raw `print()` calls in `Renderer`

### What's Preserved

- `TOOL_DISPLAY` dictionary (tool-specific label formatting lambdas)
- `Renderer.render()` return value contract (`True` = continue, `False` = exit)
- `Repl` class structure, PromptSession configuration, exit handling

### Testability

The `Renderer` accepts Console instances via constructor injection, enabling tests to pass `Console(file=StringIO(), force_terminal=True)` instead of real terminal consoles. This lets tests capture rendered output as strings and assert on content (e.g., checking that tool labels appear, errors go to the stderr console). Existing `capsys`-based tests must be migrated to this pattern since `Console` does not write to `sys.stdout` by default.

`Renderer.__init__` signature:

```
def __init__(self, console: Console | None = None, err_console: Console | None = None)
```

When `None`, defaults to `Console()` and `Console(stderr=True)` for production use. Tests inject StringIO-backed consoles.

## Modeling

No new domain entities. The change is purely in the rendering layer — `AgentEvent` types and their fields remain unchanged. The `Renderer` class keeps its existing interface (`render(event) -> bool`) but changes its internal rendering mechanism.

## Data Flow

The data flow is unchanged from the current design:

```
AgentEvent stream → Renderer.render(event) → Console → Terminal
```

The only difference is the rendering step: instead of `print()` with ANSI codes, events pass through `rich.console.Console` which handles formatting, styling, and markdown parsing before writing to the terminal.

## Key Decisions

### rich library for terminal rendering

**Choice**: Use `rich` (Textualize, v13.x) for all terminal output
**Why**: Dominant Python terminal rendering library. Provides Console (styled output), Markdown (CommonMark rendering with syntax-highlighted code blocks), and Text (programmatic styling) — covering all requirements in a single, actively maintained dependency.
**Sources**: Rich 13.x stable docs (rich.readthedocs.io), PyPI (actively maintained by Textualize)
**Options Researched**:
- `rich` — Full-featured terminal rendering, Markdown + styling + syntax highlighting
- `streamdown` — Streaming markdown CLI tool, beta status, not a library API
- `richify` — Thin wrapper around rich.Live for streaming, adds a dependency layer without benefit here
- Manual ANSI codes — Current approach, fragile, no markdown support
**Why This Over Alternatives**: Chunks are complete text blocks (not streaming tokens), so streaming-specific tools (streamdown, richify) solve a problem we don't have. Rich provides everything needed with a single well-maintained dependency.

**Consequences**:
- Pro: Markdown rendering, syntax highlighting, and styled output from one library
- Pro: Eliminates manual ANSI codes entirely
- Pro: Extensible (themes, custom styles) for future channels
- Con: Adds ~3MB dependency

### Block rendering per TextChunk (not Live accumulation)

**Choice**: Render each `TextChunk` as a standalone `Markdown` object via `Console.print()`
**Why**: The spec establishes that TextChunk events correspond to complete SDK TextBlocks — markdown syntax is never split across chunks. This means each chunk is a self-contained markdown document that can be rendered independently. No accumulation buffer or `rich.Live` needed.
**Sources**: Delta spec DLT-025 assumption; Rich docs (Markdown class parses full string at construction)
**Options Researched**:
- Block rendering (chosen) — `Console.print(Markdown(chunk))` per TextChunk. Simplest approach.
- Live accumulation — Buffer chunks, re-render via `rich.Live` on each arrival. Used by Pydantic AI for token-level streaming. Adds complexity (buffer management, Live context) for no benefit when chunks are already complete.
- Buffer-then-render — Accumulate all chunks, render as single Markdown on Result. Delays all output until response is complete — defeats the purpose of streaming.
**Why This Over Alternatives**: Block rendering is the simplest approach that works correctly given the spec's assumption about chunk completeness. It requires no state management and renders each chunk immediately.

**Consequences**:
- Pro: Simplest implementation — no buffers, no Live context, no state
- Pro: Each chunk renders immediately as it arrives
- Pro: Eliminates `_needs_newline` state tracking (Markdown block rendering always terminates with newline)
- Con: If two chunks are part of the same logical paragraph, they'll render as separate blocks (acceptable per spec assumption)

### Dracula code theme

**Choice**: Use `"dracula"` Pygments theme for fenced code block syntax highlighting
**Why**: User preference. Popular dark theme with softer purple/green palette, good contrast in most terminals.

**Consequences**:
- Pro: Visually distinct code blocks with familiar developer aesthetics
- Con: May not match all terminal color schemes (acceptable for developer tool)

## System Behavior

### Scenario: Agent responds with markdown

**Given**: The agent responds with `"# Summary\n\nHere is **important** info"` as a TextChunk
**When**: The Renderer processes the event
**Then**: The Console renders a bold/colored heading "Summary" followed by a paragraph with "important" in bold, then a trailing newline

### Scenario: Agent responds with a code block

**Given**: The agent responds with a fenced code block (` ```python\nprint("hello")\n``` `)
**When**: The Renderer processes the TextChunk
**Then**: The code block renders with Dracula-themed syntax highlighting, language-appropriate coloring

### Scenario: Agent uses a tool between text chunks

**Given**: The agent has sent a TextChunk, then a ToolActivity, then another TextChunk
**When**: Each event is rendered
**Then**: The first TextChunk renders as a Markdown block (with trailing newline), the ToolActivity renders as a dim gray italic line, and the second TextChunk renders as another Markdown block. No manual newline management needed.

### Scenario: Plain text without markdown

**Given**: The agent responds with plain text (no markdown syntax)
**When**: The Renderer processes the TextChunk
**Then**: The text displays normally as a paragraph — Markdown class treats plain text as a paragraph block without artifacts

### Scenario: Error during response

**Given**: An error event arrives during response streaming
**When**: The Renderer processes the Error event
**Then**: The error message appears in bold red on stderr (separate Console), not mixed with stdout markdown output
