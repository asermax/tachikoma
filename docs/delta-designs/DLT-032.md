# Design: DLT-032 - Guide the assistant through skill authoring

**Delta Spec**: [../delta-specs/DLT-032.md](../delta-specs/DLT-032.md)
**Status**: Approved

## Purpose

This document explains the design rationale for this delta: how the skill authoring guide is structured, deployed as a built-in skill, and integrated with the existing skill system through a multi-source registry.

After implementation, the "Detected Impacts" section will guide reconciliation into feature design docs.

## Problem Context

Developers extending the system need the assistant to guide them through creating well-structured skills. Currently, the agent has no way to produce valid skills without external documentation — the system prompt's Skills section describes the structure but lacks the depth needed for authoring (detection tuning, writing best practices, agent definition format). Additionally, skills can only come from the workspace directory, so there's no mechanism to ship built-in skills with the package.

**Constraints:**
- The guide must be comprehensive enough for the agent to produce valid skills without external docs
- The skill description must trigger on broad authoring-adjacent intents (not just "create a skill")
- Built-in skills must coexist with workspace skills, with workspace taking precedence
- The project currently has `package = false` in pyproject.toml; path resolution must work now and after DLT-024 packages the project
- System prompt trimming must retain enough awareness for the agent to recognize skill-authoring intents and trigger detection

**Interactions:**
- Skill registry discovers both built-in and workspace skills at startup via bootstrap hook
- Skills context provider receives the registry via injection and classifies/injects skills as before
- System prompt preamble provides reduced but sufficient skill awareness
- DLT-038 (hot-reload) will need to handle both sources when implemented
- DLT-033 and DLT-057 depend on the built-in source concept introduced here

## Design Overview

Four-part approach: a content-only built-in skill (SKILL.md + agent reference file) provides comprehensive authoring guidance, a multi-source registry mechanism enables shipping skills with the package, and the system prompt is trimmed to avoid duplicating structural details now covered by the guide.

The registry interface generalizes from a single workspace path to an ordered list of source directories. The bootstrap hook resolves both paths (built-in from package, workspace from settings), creates the registry, and exposes it via extras — matching the established pattern used by database, sessions, and tasks. The provider receives the registry via constructor injection instead of creating it internally.

```
┌─────────────────────────────────────────────────────────┐
│  Bootstrap                                              │
│  ┌───────────────────────────────────────────────┐      │
│  │  skills_hook                                  │      │
│  │  1. Create workspace/skills/ directory         │      │
│  │  2. Resolve built-in path (Path(__file__))     │      │
│  │  3. Create SkillRegistry([builtin, workspace]) │      │
│  │  4. ctx.extras["skill_registry"] = registry    │      │
│  └───────────────────────────────────────────────┘      │
├─────────────────────────────────────────────────────────┤
│  __main__.py (after bootstrap)                          │
│  ┌───────────────────────────────────────────────┐      │
│  │  skill_registry = bootstrap.extras[...]        │      │
│  │  SkillsContextProvider(agent_defaults,          │      │
│  │                        registry=skill_registry) │      │
│  └───────────────────────────────────────────────┘      │
├─────────────────────────────────────────────────────────┤
│  SkillsContextProvider (per-session detection)          │
│  ┌───────────────────────────────────────────────┐      │
│  │  Uses injected registry (no longer creates it) │      │
│  │  Classification + injection flow unchanged      │      │
│  └───────────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────┘
```

### Cross-Layer Contracts

**Bootstrap → __main__.py → Provider contract:**

```
skills_hook(ctx)
    │
    ├── workspace_skills_path = workspace_path / "skills"
    ├── Creates workspace_skills_path directory (idempotent)
    ├── Resolves builtin_path = Path(__file__).parent / "builtin"
    │   ├─ Exists → include in sources
    │   └─ Missing → log warning, skip (graceful)
    ├── Assembles skill_sources: [builtin_path, workspace_skills_path]
    ├── Creates SkillRegistry(skill_sources)
    └── ctx.extras["skill_registry"] = registry

__main__.py (after bootstrap.run())
    │
    ├── skill_registry = bootstrap.extras["skill_registry"]
    └── SkillsContextProvider(agent_defaults, registry=skill_registry)

SkillsContextProvider.provide(message)
    │
    ├── Uses self._registry (injected, not created)
    ├── Classification via query() [unchanged]
    └── Returns ContextResult [unchanged]
```

**Integration Points:**
- skills_hook ↔ Bootstrap: registered as standard hook (DES-003); writes `"skill_registry"` to extras
- __main__.py ↔ extras: reads `"skill_registry"` after bootstrap; passes to provider constructor
- SkillsContextProvider ↔ SkillRegistry: injected dependency; provider reads `skills` property and calls `get_agents_for_skill()`
- SkillRegistry ↔ filesystem: scans each source path for skill directories (unchanged scanning logic)
- SYSTEM_PREAMBLE ↔ Agent: trimmed Skills section provides awareness without structural detail

### Shared Logic

- **Skill discovery logic (`_discover`, `_load_skill`, `_load_agents`)**: Unchanged — called once per source path in the `skill_sources` list. The same validation, error handling, and agent loading applies regardless of source.
- **Last-wins precedence**: Achieved naturally via dict assignment — scanning built-in first, then workspace means workspace entries overwrite built-in entries for the same skill name. On skill name collision, `_load_skill` first removes any existing agents for that skill from `_agents` (by prefix), then loads the new skill and its agents. This ensures complete replacement — no orphaned agents from earlier sources remain.

## Modeling

No new domain entities. Changes to existing types:

```
SkillRegistry (modified constructor + collision handling)
├── __init__(skill_sources: list[Path])   # was: __init__(workspace_path: Path)
│   └── for source in skill_sources:
│         self._discover(source)          # reuses existing _discover()
├── _agents: dict[str, AgentDefinition]   # unchanged
├── _skills: dict[str, Skill]             # unchanged
├── _load_skill(skill_dir)               # modified: if skill name already in _skills,
│                                        #   remove its agents from _agents before loading
├── get_agents() → dict[str, AgentDefinition]           # unchanged
├── get_agents_for_skill(skill_name) → dict[...]        # unchanged
└── skills (property) → dict[str, Skill]                # unchanged

SkillsContextProvider (modified constructor)
├── __init__(agent_defaults: AgentDefaults,
│            registry: SkillRegistry)     # was: no registry param
│   └── self._registry = registry         # was: SkillRegistry(agent_defaults.cwd)
└── provide(message) → ContextResult | None  # unchanged
```

The `Skill` dataclass remains frozen and unchanged. A built-in skill and a workspace skill with the same name produce identical `Skill` objects — the workspace version completely replaces the built-in in the `_skills` dict and its agents replace the built-in's agents in `_agents` (previous agents are cleared first to prevent orphaned entries).

## Data Flow

### Startup (modified)

```
1. Bootstrap runs hooks in registration order:
   ...
   → skills_hook:
     a. workspace_skills_path = workspace_path / "skills"
        Create workspace_skills_path directory if missing (unchanged)
     b. Resolve builtin_path = Path(__file__).parent / "builtin"
        ├─ Does not exist → log warning, omit from sources
        └─ Exists → include
     c. Assemble skill_sources = [builtin_path, workspace_skills_path]
        (built-in first so workspace overwrites)
     d. Create SkillRegistry(skill_sources)
        (each source is a directory containing skill subdirectories)
        → For each source path, call _discover():
          → Scan directory for skill subdirectories
          → Load SKILL.md, agents/ for each
          → Later sources overwrite earlier by skill name
     e. Store registry in ctx.extras["skill_registry"]
   ...

2. __main__.py retrieves skill_registry from bootstrap.extras
3. Creates SkillsContextProvider(agent_defaults, registry=skill_registry)
4. Registers provider in pre-processing pipeline
```

### Per-Message (unchanged)

```
1. User sends message
2. PreProcessingPipeline.run(message) executes providers in parallel
3. SkillsContextProvider.provide(message):
   a. Check registry.skills — if empty, return None
   b. Build skills list (names + descriptions) from registry
   c. Classify via query() [Opus low effort, DES-007]
   d. For each detected skill:
      - Read skill.body from registry
      - Wrap in <skill> XML block
      - Collect agents via get_agents_for_skill()
   e. Return ContextResult(tag="skills", content=XML, agents=filtered)
4. Coordinator extracts agents from pipeline results
```

## Key Decisions

### Path Resolution via `Path(__file__)` over `importlib.resources`

**Choice**: Resolve the built-in skills directory using `Path(__file__).parent / "builtin"` in the bootstrap hook.
**Why**: The project is a CLI application, not zip-importable. `Path(__file__)` works in both development (source tree) and installed packages (site-packages). `importlib.resources` adds complexity (context managers, `Traversable` API) for a capability the project doesn't need.
**Alternatives Considered**:
- `importlib.resources.files()`: More formally correct for zip-importable packages, but the project will never be zip-imported (CLI tool installed via `uv tool install`). Adds unnecessary indirection.
- Configuration-based path: Over-engineered for a path that's always relative to the code.

**Consequences**:
- Pro: Simple, no new dependencies, works in all deployment scenarios
- Pro: Familiar Python pattern — easy to understand
- Con: Won't work if the package is ever distributed as a zipapp (not a realistic concern for this project)

### Generic `list[Path]` Interface over Named Parameters

**Choice**: `SkillRegistry.__init__(skill_sources: list[Path])` instead of `SkillRegistry.__init__(workspace_path, builtin_path=None)`.
**Why**: The registry doesn't need to know about source types — it just scans directories. A generic list is simpler, more testable, and naturally extends to additional sources (e.g., plugin skills from DLT-048).
**Alternatives Considered**:
- Named parameters (`workspace_path`, `builtin_path`): Explicit but couples the registry to specific source types. Would need modification when DLT-048 adds plugin sources.

**Consequences**:
- Pro: Registry is source-agnostic — callers decide what paths to include
- Pro: Naturally supports DLT-048 (plugins can add paths to the list)
- Pro: Easy to test — pass any list of temp directories
- Con: Caller must assemble paths in correct order (precedence via ordering)

### Registry Creation in Bootstrap Hook over Provider-Owned

**Choice**: The skills bootstrap hook creates the `SkillRegistry` and exposes it via `ctx.extras["skill_registry"]`. The provider receives it via constructor injection.
**Why**: Path resolution (built-in + workspace) is an infrastructure concern that belongs in the bootstrap phase. The provider is a consumer that shouldn't need to know about source paths. This matches the established extras pattern used by database, session_registry, and task_repository.
**Alternatives Considered**:
- Provider creates registry internally (current design): Would require the provider to know about built-in paths, mixing infrastructure and consumption concerns.
- Module-level helper in registry.py: Keeps resolution near the registry but doesn't match the project's bootstrap extras pattern.

**Consequences**:
- Pro: Consistent with existing bootstrap → extras → consumer pattern
- Pro: Provider becomes simpler — just uses the registry
- Pro: Registry is available to other consumers if needed (e.g., DLT-038 hot-reload)
- Con: skills_hook gains more responsibility (directory creation + registry creation)

### Last-Wins Precedence via Scan Order

**Choice**: Built-in sources are scanned first, workspace sources second. When a workspace skill has the same name as a built-in, the workspace version entirely replaces the built-in (all metadata, body, and agents).
**Why**: Simple and predictable — no merge logic, no partial overrides. Users who want to customize a built-in skill copy it to their workspace and modify it. The replacement is automatic via Python dict assignment.
**Alternatives Considered**:
- Explicit merge (combine agents from both sources): Complex, surprising behavior, hard to reason about
- Registry-level source tracking (store source alongside skill): Over-engineered for the current use case

**Consequences**:
- Pro: Dead simple — dict assignment handles everything
- Pro: Predictable — workspace always wins, completely
- Pro: Users can fully customize or disable a built-in by shadowing it
- Con: No partial override (can't add workspace agents to a built-in skill without copying the whole thing)

## System Behavior

### Scenario: First launch with built-in skills

**Given**: The built-in `skill-authoring-guide/` directory exists in the package and no workspace skills exist
**When**: The skills hook runs during bootstrap
**Then**: The hook creates the workspace `skills/` directory, resolves the built-in path, creates a registry scanning `[builtin, workspace]`. The registry discovers the authoring guide from built-in. The provider can classify and inject it when a user asks about creating skills.
**Rationale**: Built-in skills are available out of the box without user action.

### Scenario: Workspace skill shadows a built-in

**Given**: A workspace skill named `skill-authoring-guide/` exists alongside the built-in version
**When**: The registry scans both sources
**Then**: The built-in version is loaded first (skill metadata + agents stored). When the workspace version is loaded, `_load_skill` detects the name collision, removes the built-in's agents from `_agents`, then loads the workspace version's metadata and agents. Only the workspace version's metadata, body, and agents are retained — no orphaned agents from the built-in remain.
**Rationale**: Last-wins precedence with complete replacement — workspace always takes priority, enabling full customization.

### Scenario: Built-in path doesn't exist (corrupted installation)

**Given**: The `builtin/` directory is missing from the installed package
**When**: The skills hook resolves `Path(__file__).parent / "builtin"`
**Then**: The path doesn't exist. The hook logs a warning and omits it from `skill_sources`. The registry is created with only the workspace path. System operates normally with workspace skills only.
**Rationale**: Graceful degradation — a missing built-in directory shouldn't crash the system.

### Scenario: User asks to create a new skill

**Given**: The skill authoring guide is loaded in the registry and a user sends "I want to create a new skill for code review"
**When**: The skills context provider classifies relevance
**Then**: The guide's description (crafted for broad authoring intents per R6) triggers a match. The guide's body is injected as a `<skills>` XML block. The agent reads the comprehensive guide and produces a well-structured skill, consulting `references/agents.md` via file-system tools if agent definitions are needed.
**Rationale**: Core use case — the guide activates on authoring-adjacent intents and provides everything the agent needs.

### Scenario: User asks an unrelated question

**Given**: The skill authoring guide is loaded and a user sends "what's on my calendar today?"
**When**: The skills context provider classifies relevance
**Then**: The guide's description doesn't match. The guide is not injected. No wasted context.
**Rationale**: Precision — the guide's description targets authoring intents, not general queries.

### Scenario: DLT-038 hot-reload interaction

**Given**: The registry is created during bootstrap and exposed via extras
**When**: DLT-038 implements hot-reload
**Then**: Hot-reload can access the same registry from extras and replace/refresh it. Both built-in and workspace sources would be re-scanned.
**Rationale**: Exposing the registry via extras makes it accessible to future subsystems.

## Notes

- The skill authoring guide is content-only — no agents directory, no delegated sub-agents. The main agent reads the guide body and references directly.
- The `references/agents.md` file is not automatically injected. The SKILL.md body directs the agent to read it via file-system tools when agent definitions are needed. This follows the established pattern where references are regular files loaded on demand.
- The `builtin/` directory is a data directory, not a Python package — no `__init__.py` needed.
- When DLT-024 packages the project, the build backend must be configured to include `*.md` files from `src/tachikoma/skills/builtin/` in the distribution. This is a DLT-024 concern.
