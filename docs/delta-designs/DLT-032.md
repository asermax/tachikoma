## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | Skill authoring guide SKILL.md — informational body covering: directory conventions, SKILL.md format, available capabilities overview (agents, references, subdirectories), detection tuning tips, writing best practices, and concise inline examples. Includes pointers to the agent reference file. The frontmatter description is a critical design element for R6 (broad detection). | |
| **S2** | Agent definition reference file — `references/agents.md` with detailed agent definition format (frontmatter fields, model options, tool scoping), prompt-writing tips, and a concise example. | |
| **S3** | Built-in skill source in codebase — a package directory (e.g., `src/tachikoma/skills/builtin/`) scanned by the registry alongside workspace skills. Built-in scanned first, workspace second (workspace overwrites). Graceful fallback if path unresolvable. | ⚠️ |
| **S4** | System prompt trimming — reduce the static Skills section in `context/loading.py` to awareness-level content (skills exist, auto-detected, can create/manage) removing structural details now in the guide. Must retain enough for the agent to recognize skill-authoring intents. | |

### Flagged Unknowns

#### S3 — Built-in skill source path resolution
How does the registry resolve the built-in skill package path at runtime? The path must work both in development (source tree) and when installed as a package (via uv). Key questions:
- Should the path be derived from `__file__` of the skills module?
- Should it use `importlib.resources`?
- How does multi-path scanning interact with the existing workspace-only discovery?
