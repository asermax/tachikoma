### Flagged Unknowns

#### S6 — Session task message injection
The coordinator creates a fresh `ClaudeSDKClient` per message and is driven by the channel (Telegram/REPL) calling `send_message()`. To inject a session task message, we need a mechanism that:
- Triggers `send_message()` on the coordinator as if a user message arrived
- Routes the response back to the active channel for delivery (Telegram message, REPL output)
- Handles the race condition of a real user message arriving simultaneously (the Telegram channel has `_is_processing` flag — how does this interact with scheduler-initiated messages?)
- Doesn't require the channel to poll for scheduler events

Options to investigate: coordinator exposes an internal message queue that channels and the scheduler both feed into, or the scheduler calls the channel's message handling interface directly, or a callback/event pattern where the scheduler notifies the channel to initiate a send.

#### S7 — Background task adapted pipeline
The current pipeline is assembled once in `__main__.py` with a fixed set of processors. Background tasks need a different processor selection (skip FactsProcessor, PreferencesProcessor, CoreContextProcessor). The mechanism needs to:
- Create a fresh session with its own pipeline configuration
- Use a different base system prompt (explaining background task context)
- Reuse existing processor implementations selectively
- Enforce a concurrency limit across concurrent background task sessions

Options to investigate: parameterized pipeline factory, a separate background-task coordinator that assembles its own pipeline, or bypass the Coordinator entirely and use standalone SDK queries (similar to `fork_and_consume` pattern in post-processing).
