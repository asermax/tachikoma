## Shape

| Part | Mechanism | Flag |
|------|-----------|:----:|
| **S1** | Add `session_idle_timeout` field to `AgentSettings` (default 900s, 0 disables) and update default config generation | |
| **S2** | Periodic idle-close check loop — async task checking inactivity against timeout at a 60-second interval | |
| **S3** | Coordinator idle-close method — close session in registry, trigger async post-processing, clear SDK state, store previous summary | |
| **S4** | Snooze mechanism — when idle timeout is reached but coordinator is busy (message in flight or session task dispatched and in-flight), defer for 5 minutes (capped to timeout) and retry | ⚠️ |

### Flagged Unknowns

#### S4 — Task dispatch tracking signal
The snooze mechanism needs to detect when a session task has been dispatched (`SessionTaskReady`) but not yet completed delivery. The coordinator alone cannot observe this — it only knows when `send_message()` is active. The design phase needs to determine how this signal is communicated: options include a shared flag/counter on the event bus, a callback from the scheduler, or a coordinator-level "task delivery in progress" tracker set by the channel before calling `send_message()`.
