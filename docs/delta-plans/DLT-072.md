#### Step 2: Add `task_type` field to `UpdateTaskArgs` and handler (S2)

- Modify: `src/tachikoma/tasks/tools.py`
- What to do:
  1. Add import: `from typing import Literal`
  2. Add `task_type: Literal["session", "background"] | None = None` field to `UpdateTaskArgs`
  3. In the `update_task` handler, add `task_type` to the updates dict following the same pattern as other optional fields:
     ```python
     # After existing field checks in update_task, before the `if not updates:` guard:
     if parsed.task_type is not None:
         updates["task_type"] = parsed.task_type
     ```
- Test: Add tests in `tests/tasks/test_tools.py`:
  - **Model-level**: `UpdateTaskArgs` validates `task_type="session"` and `task_type="background"` successfully; rejects invalid values with `ValidationError`; defaults to `None` when not provided
  - **Handler-level**: Create a "background" definition via the repo, call the `update_task` handler directly with `task_type="session"`, then read the definition back from the repo and assert `task_type` was changed to "session". This covers the AC "the definition's `task_type` is updated".

#### Step 3: Normalize error handling across all four tools (S3)

- Modify: `src/tachikoma/tasks/tools.py`
- What to do:
  1. Add import: `from tachikoma.tasks.errors import TaskRepositoryError`
  2. For `list_tasks`: refactor the single `except Exception` block into two tiers тАФ extract `model_validate` into its own `try/except ValidationError` (returning field-level error), then wrap the repository call in `try/except TaskRepositoryError` (surfacing `__cause__`) + `except Exception` (unexpected fallback)
  3. For `create_task`, `update_task`, `delete_task`: replace the generic `except Exception` blocks around repository calls with `TaskRepositoryError`-specific catch + generic fallback:
     ```python
     except TaskRepositoryError as exc:
         cause = f" Cause: {exc.__cause__}" if exc.__cause__ else ""
         return {"is_error": True, "content": [{"type": "text", "text": f"{exc}{cause}"}]}
     except Exception as exc:
         return {"is_error": True, "content": [{"type": "text", "text": f"Unexpected error: {exc}"}]}
     ```
  4. Verify that existing `ValidationError` catches in `create_task`, `update_task`, and `delete_task` (which format as `"Invalid arguments: {exc}"`) already produce field-level messages via Pydantic's default error formatting (e.g., field name + accepted values for `Literal` types). No formatting change needed тАФ Pydantic's `str(exc)` naturally includes this detail.
- Test:
  - When repository raises `TaskRepositoryError` with `__cause__`, the tool response includes both wrapper message and cause string
  - When repository raises `TaskRepositoryError` without `__cause__`, only the wrapper message is returned
  - `list_tasks` `ValidationError` catch: pass un-coercible value like `{"archived": "notabool"}` and verify error response includes the field name
