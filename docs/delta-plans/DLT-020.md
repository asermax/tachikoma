### Steps

#### Step 1: Add phase constants and restructure pipeline internals

- Modify: `src/tachikoma/post_processing.py`
- What to do:
  - Add phase constants at module level:
    ```python
    # In src/tachikoma/post_processing.py, after imports:

    # Fixed phase identifiers — validated at registration
    MAIN_PHASE = "main"
    FINALIZE_PHASE = "finalize"
    _VALID_PHASES = frozenset({MAIN_PHASE, FINALIZE_PHASE})
    ```
  - Replace `_processors: list[PostProcessor]` with `_phases: dict[str, list[PostProcessor]]` in `__init__`, pre-populated with empty lists for each valid phase:
    ```python
    # In __init__:
    # Pre-populate so register() can append without KeyError
    self._phases: dict[str, list[PostProcessor]] = {p: [] for p in _VALID_PHASES}
    ```
  - Add `_phase_order` as a class-level attribute: `_phase_order = [MAIN_PHASE, FINALIZE_PHASE]`
- Test: Existing tests should still import and construct `PostProcessingPipeline()` without errors

#### Step 2: Update register() with phase parameter and validation

- Modify: `src/tachikoma/post_processing.py`
- What to do:
  - Change `register()` signature to `register(self, processor: PostProcessor, phase: str = MAIN_PHASE) -> None`
  - Add validation: if `phase not in _VALID_PHASES`, raise `ValueError` listing valid phases
  - Append processor to `self._phases[phase]`
- Test: `pipeline.register(proc)` still works (backward-compatible). `pipeline.register(proc, phase="invalid")` raises `ValueError`.

#### Step 3: Update run() for sequential phase execution

- Modify: `src/tachikoma/post_processing.py`
- What to do:
  - Iterate `_phase_order` sequentially
  - For each phase, skip if no processors registered (empty list)
  - Gather processors in parallel within the phase via `asyncio.gather(return_exceptions=True)`
  - Update the existing error logging loop to include phase context: `"Processor failed: processor={name} phase={phase} err={err}"`
  - Continue to next phase even if current phase has failures (error isolation across phases)
- Test: Run the full existing test suite — all tests should pass since existing processors default to "main" phase.

#### Step 4: Add tests for phased pipeline behavior

- Modify: `tests/test_post_processing.py`
- What to do: Add tests covering the new phased execution acceptance criteria:
  - Unknown phase at registration raises `ValueError` with clear message
  - Finalize-phase processors run after main-phase processors complete
  - Finalize-phase processors run even when main-phase processors fail
  - Default phase is "main" (backward compatibility)
  - Empty phase is skipped (no error)
  - Multiple finalize processors run in parallel
  - Phase constants are exported and usable
- Test: `pytest tests/test_post_processing.py` — all tests pass

### Steps

#### Step 5: Create git package structure

- What to do:
  - Create the `git/` package directory
  - Add `__init__.py` with re-exports placeholder (updated after steps 6–7)
- Test: Package is importable

#### Step 6: Create git bootstrap hook

- Create: `src/tachikoma/git/hooks.py`
- What to do:
  - Define `git_hook(ctx: BootstrapContext) -> None` async function
  - Read workspace path from `ctx.settings_manager.settings.workspace.path`
  - Check if `workspace_path / ".git"` exists — if yes, return early (idempotent)
  - Run git commands via `asyncio.create_subprocess_exec` (cwd=workspace_path):
    1. `git init`
    2. `git config user.name "Tachikoma"`
    3. `git config user.email "tachikoma@local"`
    4. `git commit --allow-empty -m "Initial commit"`
  - For each subprocess: check return code, raise with stderr if non-zero
  - Add logging per DES-002 (`component="git"`, log init completion)
  - No `.gitignore` creation
- Test: Unit tests in Step 9

#### Step 7: Create git post-processor and prompt

- Create: `src/tachikoma/git/processor.py`
- What to do:
  - Define `GIT_COMMIT_PROMPT` module-level constant with instructions for the agent:
    - Inspect `git status` and `git diff` to understand changes
    - Group changes by workspace subdirectory/purpose
    - Create one `git add` + `git commit` per group with descriptive message
    - Include all non-ignored changes (tracked and untracked)
    - Only use safe git commands (no push, branch, reset, checkout, etc.)
    - Never ask for confirmation
  - Define `query_and_consume(prompt: str, cwd: Path) -> None` helper:
    - Create `ClaudeAgentOptions(model="haiku", cwd=cwd, permission_mode="bypassPermissions")` — no `max_turns` or `max_budget_usd` (per design decision: task is naturally bounded, Haiku cost is minimal)
    - Call `query(prompt=prompt, options=options)`
    - Fully consume the async iterator
  - Define `GitProcessor(PostProcessor)`:
    - `__init__(self, cwd: Path)`: store workspace path
    - `async def process(self, session: Session)`:
      1. Run `git status --porcelain` via `asyncio.create_subprocess_exec` in `self._cwd`
      2. If stdout is empty → log debug, return (no-op)
      3. Call `await query_and_consume(GIT_COMMIT_PROMPT, self._cwd)`
      4. Run `git status --porcelain` again (post-agent verification)
      5. If stdout non-empty → log warning about remaining uncommitted changes
  - Add logging per DES-002 (`component="git"`)
- Test: Unit tests in Step 10

#### Step 8: Wire up hook and processor in \_\_main\_\_.py

- Modify: `src/tachikoma/__main__.py`
- What to do:
  - Add import: `from tachikoma.git import git_hook, GitProcessor`
  - Add import: `from tachikoma.post_processing import FINALIZE_PHASE`
  - Register git hook after all other hooks (workspace must exist before git init):
    ```python
    bootstrap.register("git", git_hook)  # After workspace — directory must exist before git init
    ```
  - Register git processor in finalize phase:
    ```python
    pipeline.register(GitProcessor(cwd=settings.workspace.path), phase=FINALIZE_PHASE)
    ```
  - Update `src/tachikoma/git/__init__.py` re-exports: `git_hook`, `GitProcessor`
- Test: Application starts without errors; bootstrap includes git init on first run

#### Step 9: Add tests for git bootstrap hook

- Create: `tests/git/__init__.py` and `tests/git/test_hooks.py`
- What to do: Tests covering repo initialization acceptance criteria:
  - Hook initializes git repo when no `.git` exists (git init, config, initial commit)
  - Hook is idempotent when `.git` already exists (skips, no error)
  - Hook re-initializes when `.git` was deleted
  - Repo-local identity is configured (user.name, user.email)
  - No `.gitignore` is created
  - Hook fails with clear error when git init fails
  - Hook works without global git config
  - Follow pattern from `tests/memory/test_hooks.py` (SettingsManager fixture, BootstrapContext setup)
  - Use `tmp_path` for workspace isolation — these tests call actual git commands
- Test: `pytest tests/git/test_hooks.py`

#### Step 10: Add tests for git processor and helper

- Create: `tests/git/test_processor.py`
- What to do: Tests covering processor and helper acceptance criteria:
  - Processor calls `query_and_consume` when workspace is dirty (mock subprocess + query)
  - Processor returns no-op when workspace is clean (no agent spawned)
  - Processor runs post-agent `git status` check and logs warning if changes remain
  - `query_and_consume` calls `query()` with correct options (model="haiku", cwd, permission_mode, no resource limits)
  - `query_and_consume` fully consumes async iterator
  - `query_and_consume` propagates query() errors
  - Prompt references safe git commands only
  - Prompt instructs grouping by subdirectory
  - Prompt instructs not to use destructive commands
  - Follow pattern from `tests/memory/test_episodic.py` (mock query helper, test prompt content)
- Test: `pytest tests/git/test_processor.py`
