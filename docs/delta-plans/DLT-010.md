  - Uses configured timezone

# In src/tachikoma/tasks/scheduler.py

async def instance_generator(
    repository: TaskRepository,
    settings: TaskSettings,
) -> None:
    """Async loop that evaluates task definitions and creates instances.

    Runs every ~60 seconds. For each enabled definition:

    - Check for existing pending/running instance (duplicate prevention)
    - Create pending instance if schedule fires and no duplicate exists
    - Auto-disable one-shot definitions after firing
    """

#### Step 2: Create background task executor with evaluator

- Add to: `src/tachikoma/tasks/executor.py`
- What to do: Implement `BackgroundTaskExecutor` class:
  - `__init__`: accepts repository, settings, bus, cwd, cli_path
  - `execute(instance: TaskInstance)` method:
    1. Mark instance as running in repository
    2. Run pre-processing: assemble `PreProcessingPipeline` with `MemoryContextProvider`, run against instance.prompt to get enriched prompt with memory context
    3. Build adapted system prompt explaining background task context
    4. Build `ClaudeAgentOptions` with system prompt (SystemPromptPreset with append), permission_mode="bypassPermissions", cwd
    5. Create `ClaudeSDKClient`, call `client.query(enriched_prompt)`
    6. Evaluator loop (max_iterations):
       a. Consume response via `client.receive_response()`, collect response text and session ID from ResultMessage
       b. Call evaluator: separate standalone `query()` (DES-005: fully consume) with lightweight prompt asking for JSON `{"status": "complete"|"continue"|"stuck", "feedback": "..."}`
       c. Parse evaluator response
       d. If complete: break
       e. If continue: call `client.query(evaluator_feedback)` (resume for multi-turn)
       f. If stuck or max iterations reached: mark failed, break
    7. Run adapted post-processing pipeline using extracted session ID:
       - Create `PostProcessingPipeline`, register `EpisodicProcessor` (main phase) and `GitProcessor` (finalize phase) only
       - Build a `Session` dataclass with the background task's sdk_session_id for pipeline compatibility
    8. Mark instance completed/failed in repository
    9. Dispatch notification:
       - If complete and definition has non-null notify: `bus.dispatch(TaskNotification(message=..., severity="info"))`
       - If failed: `bus.dispatch(TaskNotification(message=..., severity="error"))`
  - Error handling: wrap entire execute in try/except, mark failed on unexpected errors

## Notes

- Schema management uses Alembic migrations following the DLT-028 pattern (`sessions/migrations/`). The tasks subsystem has its own migration environment (`tasks/migrations/`) with a separate `TaskBase` metadata, since it uses a separate SQLite database. The repository's `initialize()` runs migrations via `conn.run_sync(run_migrations)` from the migration env.
- The evaluator uses a standalone `query()` call (not in-session) per the design decision. DES-005 applies: the generator must be fully consumed.
- The background task executor builds a `Session` dataclass for the post-processing pipeline using the SDK session ID from `ResultMessage` received during execution.
- Background task pre-processing: the executor assembles its own `PreProcessingPipeline` with `MemoryContextProvider` to inject relevant memory context before the first query, per spec R5 and AC "pre-processing pipeline runs."
- `TaskNotification` delivery follows the design's transient instance pattern: when a notification arrives and the user is active, a transient session task instance is created in the DB for idle-gated delivery by the session scheduler. When idle, delivery is immediate. The channel never imports `TaskRepository` — the notification handler communicates with the task subsystem via the event bus.
- The `on_complete` callback on `SessionTaskReady` is a closure that marks the instance as completed in the repository. Channels never import `TaskRepository` — decoupling maintained via callback.
- R17 (instance cleanup by retention) is out of scope per the spec's "Nice-to-have" status.
- The instance generator (Batch 3, Step 2) is the most logic-dense step in the plan, covering schedule parsing, duplicate prevention, one-shot auto-disable, catch-up, and timezone handling. The implementing agent should give this step extra attention and testing.
